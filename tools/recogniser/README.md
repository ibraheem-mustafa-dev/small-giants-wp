# Recogniser v1

HTML-mockup → SGS WordPress block-markup pipeline. Spec: `.claude/plans/recogniser-v1.md`.

## Pipeline

```
HTML mockup
  → section_detector.py     (mechanical DOM walk)
  → fingerprint_indexer.py  (SGS DB → core WP → WC catalogue)
  → recogniser.py           (AI match per section, 4-tier output)
  → style_extractor.py      (computed CSS → SGS palette tokens)
  → serialiser.py           (block-comment markup)
  → output_router.py        (post / template-part / pattern)
```

## Quick start

```bash
python tools/recogniser/recogniser.py \
  --html sites/mamas-munches/mockups/homepage/index.html \
  --variation mamas-munches \
  --output reports/recogniser-run-$(date +%F).md
```

## Architecture invariant

Recogniser-first. Match HTML against existing blocks (SGS → core → WC). Transformer is a fallback that v1 does not build — gaps are surfaced for human decision, not auto-generated.

## Modules

| File | Role |
|------|------|
| `section_detector.py` | Mechanical DOM split into semantic sections |
| `fingerprint_indexer.py` | Build catalogue of all known blocks + their HTML signatures |
| `recogniser.py` | Main entry — runs full pipeline; AI match per section via Claude CLI |
| `style_extractor.py` | Map computed CSS to SGS palette/spacing/typography tokens |
| `serialiser.py` | Emit valid WP block-comment markup |
| `output_router.py` | Route output: post_content vs template-part vs pattern |
| `prompts/recogniser-prompt.md` | Cold-prompt template for AI match step |
| `data/fingerprints.json` | Generated by indexer; consumed by recogniser |

## AI integration

Uses Claude CLI subscription — NOT direct Anthropic SDK (keys exhausted). Shells out per section:
