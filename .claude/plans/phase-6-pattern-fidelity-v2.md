# Phase 6 - Pattern Fidelity (v2)

**Hard pass criterion: ≤1% pixel diff at 375 / 768 / 1440 viewports against the source mockup.**

## Why v2

v1 (commit `<original>`) framed Phase 6 as "patch composer + chrome + hero". Four QC reviewers verified that framing was symptom-driven, not architecture-driven. The real picture surfaced in the 2026-05-13 tooling audit:

- **13 modules are built+tested but unwired** into the live `/sgs-clone` path
- **`extract.py` has the right architecture but only hero uses it** - generic role strategies don't populate CSS consumption
- **Stage 0.7 monolithic CSS dump is wrong-architecture** - retires automatically once extract.py generalises
- **`compose_atomic_pattern` is a fallback, not the main path** - replaced by the autonomy chain (bucket-c-classifier + atomic-block-scaffold already wired; gap-writers still need wiring)
- **`register_patterns.py` bypasses `uimax_write` validator** - Rosetta Stone discipline silently broken on every clone
- **Two inline functions need extracting** - `compose_atomic_pattern` and `stage_0_7_css_lift` violate "deterministic not inline" rule

Plus small wins: theme.json caching, 5 dead-table retirement, `validate-naming.py` ghost-reference resolved (already done 2026-05-13).

## End goal

Pipeline runs end-to-end on a Bean-controlled mockup and produces ≤1% pixel diff at 3 viewports. Stretch: every clone also writes valid Rosetta Stone payloads + surfaces gaps to the operator.

## Three refactor jobs + wiring + cleanup

### J1: extract.py CSS-consumption generalisation (~90 min)

**Problem**: Hero override has CSS-consumption tracking (`_consumed_rules` + `_consumed_decls`). Generic role strategies in `extract_strategies.py` do NOT. Result: only hero sections produce a scoped-CSS remainder; everything else's CSS is silently lost or dumped to the variation file via Stage 0.7.

**Solution**:

1. **Move `collect_section_css` + `emit_scoped_custom_css` out of `overrides/hero.py`** into a sibling module `tools/recogniser-v2/css_consumption.py`. They're not hero-specific - they're the generic pipeline. ~15 min.
2. **Extend each of the 11 role strategies in `extract_strategies.py`** to populate `_consumed_rules` and `_consumed_decls` in the context dict whenever they consume a CSS declaration. ~45 min.
3. **Update extract.py main()** to import from the new module path (5 min mechanical edit).
4. **Update overrides/hero.py** to import from new module path + remove the now-duplicated functions (5 min).
5. **Regression check**: run hero golden test (`tests/golden/hero-extraction-baseline.json`) - must stay PASS. ~10 min.
6. **Coverage check on non-hero section**: run extract.py against Mama's `.sgs-featured-product` section and verify _consumed_rules > 0 (currently 0). ~10 min.

**Files touched**: `tools/recogniser-v2/extract.py`, `tools/recogniser-v2/extract_strategies.py`, `tools/recogniser-v2/overrides/hero.py`, new `tools/recogniser-v2/css_consumption.py`.

**Doc updates** (per docs-registry):
- `tooling-map.md` - extract_strategies status updated; new css_consumption.py added
- `cloning-pipeline-flow.md` - Stage 4 description updated (consumption tracking generalised)
- `decisions.md` - log decision: "CSS-consumption moved from hero override to generic css_consumption.py module"

### J2: Stage 0.7 retirement (~30 min, follows J1)

**Problem**: `stage_0_7_css_lift` at sgs-clone-orchestrator.py:253 dumps ALL mockup CSS into one variation file. Wrong architecture per `feedback_cloning_preserves_intentional_bespoke_detail`.

**Solution**: Once J1 lands, Stage 0.7 has no remaining job - per-section CSS-consumption handles everything. Retire it:

1. **Delete `stage_0_7_css_lift` from sgs-clone-orchestrator.py** + remove the call site (~10 min).
2. **Delete any `theme/sgs-theme/styles/<client>.css` files** generated by prior runs (~5 min).
3. **Verify**: Mama's mockup re-runs end-to-end without the monolithic dump; per-section _pending_custom_css emits attached to the page via wp:html (~15 min).

**Files touched**: `plugins/sgs-blocks/scripts/sgs-clone-orchestrator.py` (delete inline function + call).

**Doc updates**:
- `cloning-pipeline-flow.md` - REMOVE Stage 0.7 block entirely
- `tooling-map.md` - no change (was inline; never had a row)
- `decisions.md` - log: "Stage 0.7 retired; per-section consumption replaces it"

### J3: compose_atomic_pattern → autonomy chain (already mostly wired)

**Problem**: `compose_atomic_pattern` (sgs-clone-orchestrator.py:339) is a fallback that fires when confidence-matrix returns no block match. Inline; emits flat dump.

**Solution**: The autonomy chain is the architectural replacement. Half of it is already wired (`bucket-c-classifier` + `atomic-block-scaffold`). The remaining 3 modules need wiring (handled by Step 4 below). compose_atomic_pattern stays as a degraded-but-better-than-nothing fallback for cases where the autonomy chain itself fails or errors.

**Action in Phase 6**: do NOT delete compose_atomic_pattern. Leave it as the safety net. After Phase 6 the autonomy chain should cover most cases; compose_atomic_pattern becomes truly cold-path code.

**Doc update**:
- `cloning-pipeline-flow.md` - Stage 7 fallback note updated to clarify "fires only when autonomy chain unable to scaffold"

### Step 5: Rosetta Stone discipline fix (~20 min)

**Problem**: `register_patterns.py` writes to uimax via direct sqlite3, bypassing `uimax_write.py` + validator. Licensing + Rosetta Stone checks silently skipped on every clone.

**Solution**: Refactor `register_patterns.py` to import `uimax_write.validate_and_write(...)` instead of `sqlite3.connect()` + `INSERT`. Same payload, single validation chokepoint.

**Files touched**: `plugins/sgs-blocks/scripts/orchestrator/register_patterns.py`.

**Verification**: write a deliberately-non-compliant payload (missing `equivalent_implementations.sgs_block`) and confirm validator rejects (ValidationError raised). Then write a compliant payload and confirm it lands in uimax.

**Doc updates**:
- `tooling-map.md` - register_patterns.py note updated; uimax_write.py + uimax-write-validator.py status flipped from PARTIAL to YES
- `cloning-pipeline-flow.md` - +REGISTER tail block updated (no longer "BYPASSED")
- `decisions.md` - log: "Rosetta Stone discipline restored on /sgs-clone path"
