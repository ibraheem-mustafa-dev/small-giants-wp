# P2.5 Phase 3 — Nav block architecture

## 0. Locked design decisions (Bean, 2026-07-19 — do NOT re-litigate; the council STRESS-TESTS, does not overturn)

| # | Decision | Detail |
|---|---|---|
| DP1 | **Menu data = `wp_navigation` CPT** | The one settled source. Resolved via a shared source library. |
| DP2 | **Mega = saved reusable panels** | Each panel built once, referenced by a menu item by ID at any position (fixes the always-last bug by construction). Astra/Spectra model. |
| DP3 | **Interaction = core-WP-nav model** | Dropdown/mega opens on **hover on non-touch devices** (default), **tap on touch**, keyboard-operable throughout (Enter/Space open, Esc close, focus return). Top-level items stay real links. WCAG 1.4.13 hover contract (dismissible/hoverable/persistent) applies. Consistent familiar model across breakpoints. **Safe-triangle (Amazon trajectory tracking) is IN v1 (Bean 2026-07-19): because hover is the default open behaviour, the diagonal-dismissal defect is real, so its fix is a build requirement, not later polish.** |
| DP4 | **Drawer = full modal** | Full-screen dialog (focus trap, background `inert`, ESC). Header rows can OPTIONALLY render at the drawer TOP (burger→X close), reusing the header-CPT block groups, with a **per-row show/hide checkbox** (one per header row). |
| DP5 | **Header scroll behaviour** | Static (default) + toggles **sticky / shrink / transparent / hide-on-scroll** — all v1. Owned by the header container, not the nav. |
| DP6 | **v1 scope** | v1 = two-row header, cart badge, dropdowns, saved-panel mega, modal drawer, header scroll toggles. **Extra phase:** drill-down deep menus (accordion is the v1 default), command palette (Cmd-K), sticky in-page sub-nav. |
| DP7 | **Evidence-decided (not forks):** overlay-not-push · link-count = soft editor notice not a gate · body-portal for transform-survival · disclosure (bar/dropdown/mega) vs dialog (drawer) are SEPARATE behaviours sharing a small focus/inert utility, never one component. | |

## 1. Design method (how the decomposition was derived)

First-principles decomposition by **distinct responsibility × distinct ARIA contract × reuse surface**, then
mapped back to the inventory's HAVE to see what's reusable. A responsibility becomes its own block when it is
(a) something a client independently places/configures, OR (b) a distinct render target, OR (c) a distinct
interaction contract. Sub-responsibilities that only ever appear inside a parent become that parent's
internals (attrs/InnerBlocks/plumbing), NOT separate blocks — avoiding core-nav's over-fragmentation
(`navigation` + `navigation-link` + `navigation-submenu` + `navigation-overlay-close` = 4 blocks for one menu).

## 2. THE RECOMMENDED BLOCK SET

Five nav blocks + one CPT + reused header/utility blocks + one shared plumbing module. Grouped by layer.

### Layer 1 — Header container (P2; EXISTS, reused as-is — the nav's home)
- **`sgs/site-header`** (section-root, `SGS_Container_Wrapper`) — owns **DP5 scroll behaviour** (static/sticky/
  shrink/transparent/hide-on-scroll as attrs), publishes `--sgs-header-height`. Contains rows.
- **`sgs/site-header-row`** (layout) — a header row. **Two-row header = two rows** (utility row + nav row),
  each independently configurable for sticky/dismiss (DP5 + the two-row NEED). Hosts typed elements.
> These are P2's deliverable, already built. Nav slots INTO a row. Listed for completeness; not re-derived here.

### Layer 2 — Navigation blocks (the P2.5 build)

**B1 — `sgs/nav` (the navigation orchestrator)** — *the block a client drops into a header row.*
- **Responsibility:** the desktop bar chrome + the responsive collapse decision + the burger toggle that
  opens the drawer. Reads a `wp_navigation` menu (attr `ref`; inheritable). Provides the resolved menu +
  breakpoint as **block context** to its children.
- **Renders:** the horizontal top-level bar (via an inner menu renderer, B2); at/below the content-driven
  collapse width, hides the bar and shows the **burger toggle** (owns `aria-expanded`/`aria-controls` → the drawer).
- **Interaction contract (DP3):** top-level items are real `<a>` links; items with children get a
  disclosure toggle (hover-open non-touch / tap-open touch / keyboard) opening a dropdown or a mega panel.
- **KIND:** layout composite → keeps `SGS_Container_Wrapper` (composite-mirror). **Variant_attr:** `layout`
  (bar layouts: inline / split / centred) — NET-NEW `variant_slots` (none exist today).
- **Why its own block:** the collapse/burger/orchestration is a distinct responsibility from rendering a link
  list; it's what the client configures ("this is my main menu, collapses at X, opens this drawer").

**B2 — `sgs/nav-menu` (the menu-list renderer)** — *the reusable one-source→dual-render primitive.*
- **Responsibility:** render a `wp_navigation` tree as an accessible list — top-level links + **disclosure**
  submenus. **Modes:** `inline` (horizontal, used inside B1's bar) · `stacked`/`accordion` (used inside the
  drawer B4 and standalone in a footer column). ONE data source drives both (R5 / FR-S9-4).
- **Menu source:** attr `ref` (null = inherit B1's context; picker overrides — the Spec-34 escape hatch,
  re-derived). Usable with no `sgs/nav` ancestor (footer menu) → context absent → `ref`/page-list fallback.
- **Interaction contract:** disclosure only (never `role=menu`); hover/tap/keyboard per DP3.
- **KIND:** content-block (renders its own scoped `<style>`, no grid/section machinery) → **block-private**
  styling per D294 (like quote), no wrapper call needed.
- **Why separate from B1:** it's the reuse point — the drawer and footer render the SAME menu without the
  bar's collapse/burger chrome. Merging it into B1 would force the drawer to re-implement menu rendering
  (the exact duplication that produced today's 3-store mega bug).

**B3 — `sgs/mega-panel` + the `sgs_mega_panel` CPT (saved reusable mega content) — DP2.**
- **Responsibility:** a reusable, visually-built panel of content (columns / logo-grid / photo-grid /
  info-box / CTA — composed from existing SGS content blocks) that a menu item references by **ID**.
- **Storage:** a dedicated **`sgs_mega_panel` CPT** (portable via `wp_insert_post`; converter emits one post
  per panel with a naming convention `sgs-mega-{client}-{slug}` per the council). A menu item flags "has a
  mega panel" via an **ID reference** (NOT slug-string — the council's must-fix), so the panel renders at the
  item's REAL position (fixes always-last). `sgs/nav` reads the reference and renders the panel into the
  item's disclosure panel.
- **Why a CPT not inline InnerBlocks:** DP2 (Bean) — reuse across menu items/menus + decouples panel content
  from menu structure (the root cause of the position bug). Why a CPT not a `wp_block` reusable block:
  evaluated (§3) — a dedicated CPT gives nav-scoped management UX + a clean converter-emit target + avoids
  polluting the client's general reusable-blocks list. (Council to confirm CPT vs `wp_block`.)
- **KIND:** the panel's INNER content uses existing content/section blocks (each already no-inline). The
  panel wrapper is content/layout as needed.

**B4 — `sgs/nav-drawer` (the off-canvas modal) — DP4.**
- **Responsibility:** the mobile/collapsed menu surface — a **full-screen modal dialog** (focus trap,
  background `inert`, ESC, focus-return-to-burger). **Body-portalled** on open so it survives a transformed
  (hide-on-scroll) header ancestor (R6 / §15.3).
- **Content:** InnerBlocks — default `sgs/nav-menu` (accordion mode) — PLUS the **DP4 header-rows-at-top**
  option: the header-CPT rows can render at the drawer top (burger→X), each gated by a **per-row show/hide
  checkbox**. Reuses the header-CPT block groups (same settings).
- **Interaction contract:** DIALOG (distinct from B1/B2's disclosure) — this is why it's a separate block
  with its own view.js behaviour, per DP7/R2. Shares only the focus/inert utility.
- **KIND:** section/layout composite (genuine overlay geometry) → keeps `SGS_Container_Wrapper` (or a
  drawer-specific scoped wrapper); geometry explicit + header-relative (Spec-34 lessons re-derived clean).
- **Wiring:** `sgs/nav`'s burger `aria-controls` → this drawer's uid (not DOM adjacency, since portalled).

**B5 — `sgs/nav-cart` (cart with accessible live badge) — the A3#13 NEED.**
- **Responsibility:** the header cart affordance with a **live count** announced via `role="status"
  aria-live="polite" aria-atomic="true"` (the WCAG 4.1.3 gap most WooCommerce themes leave open).
- **Reuse:** `sgs/cart` EXISTS — re-derive/extend it with the live-count wrapper rather than a new block IF
  its current shape fits; otherwise a clean cart-slot block. (Council/Phase-4 to confirm reuse vs rebuild —
  flagged, not assumed, per the zero-influence guardrail.)

### Layer 3 — Reused utility blocks (EXIST; populate header rows + drawer)
`sgs/product-search` (header search slot — inline/icon; `sgs/filter-search` is NOT usable here, it is
`ancestor`-locked to `woocommerce/product-filter-attribute`, corrected 2026-07-19) · `sgs/button` / `sgs/multi-button` (CTAs) ·
`sgs/responsive-logo` (logo) · `sgs/social-icons` · `sgs/business-info` · `sgs/notice-banner` (announcement
row content). No new blocks — they slot into header rows and/or the drawer.

### Layer 4 — Shared plumbing (NOT blocks)
- **`sgs-nav-interactions` (a `viewScriptModule`)** — SGS-namespaced (`data-wp-interactive="sgs/nav"`)
  open/close/focus/`inert` primitives. Exposes TWO distinct behaviours — **disclosure** (dropdown/mega: no
  trap, hover/tap, WCAG 1.4.13, **safe-triangle trajectory tracking on hover-open** per DP3) and **dialog**
  (drawer: trap + inert + ESC) — sharing one small focus/inert utility. NEVER one merged component (DP7/R2). Ported pattern under SGS's own store (net-new, not a free
  reuse of core-nav internals — §15.2/§15.7).
- **`SGS_Nav_Menu_Source`** (EXISTS) — `wp_navigation` → tree resolution; reused by call.
- **Mega-panel resolver** — menu-item ID → `sgs_mega_panel` post → rendered panel at position.
- **Shared breakpoint source** (`--sgs-header-height` publisher EXISTS) + content-driven collapse measurement.

## 3. Alternatives considered (and why rejected)

| Alternative | Rejected because |
|---|---|
| **Fold B2 into B1** (one `sgs/nav` block, drawer as its InnerBlocks) | Fewer blocks, but the drawer + footer would re-implement menu rendering → the exact duplication that produced today's 3-store mega bug. Reuse of ONE menu renderer is the whole point of R5. |
| **Inline mega (InnerBlocks on the menu item)** instead of a CPT | Bean chose saved reusable panels (DP2); inline couples content to menu markup (position bug) + no reuse + harder converter emit. |
| **`wp_block` (native reusable blocks) for mega storage** instead of a dedicated CPT | Viable + very WP-native (Spec 35 Part G "prefer native"), BUT pollutes the client's general reusable-blocks list and lacks nav-scoped management UX + a clean naming/emit convention. Kept as the council's explicit alternative to confirm. |
| **One unified `sgs/nav-panel` with an `isModal` flag** (bar dropdown + drawer in one) | The two ARIA contracts are structurally incompatible (disclosure vs dialog — R2, the codebase's own view.js note, and 3 external sources). A boolean flag can't reconcile focus-trap-present/absent + ESC-mandatory/optional + inert-present/absent. |
| **Core `core/navigation` as the engine** | Ruled OUT (P2 §15.1): no mega, full-screen-only overlay, inline styles (Spec 32), no device tiers, unsettled submenu a11y. |
| **More granular blocks** (separate top-level-item / dropdown / submenu blocks, like core nav's 4) | Over-fragmentation; a non-coder client shouldn't assemble a menu from 4 block types. The menu tree comes from `wp_navigation` data, rendered by ONE B2 renderer. |

## 4. Composition + data flow

```
sgs/site-header (scroll behaviour, --sgs-header-height)
├─ sgs/site-header-row  [utility row]     → notice-banner / social / business-info / language
└─ sgs/site-header-row  [nav row]         → responsive-logo · sgs/nav · sgs/nav-cart · search · CTA
                                              │
   sgs/nav  (reads wp_navigation ref; provides context) ──────────────┐
   ├─ renders bar via sgs/nav-menu [inline mode]                       │ menu items flagged mega
   ├─ owns burger toggle ── aria-controls ─▶ sgs/nav-drawer            │   └─▶ ID ref ─▶ sgs_mega_panel CPT
   │                                                                    │        rendered at item position
   └─ dropdown/mega disclosure (hover non-touch / tap touch / kbd)  ◀──┘        (sgs/mega-panel)

sgs/nav-drawer  (full modal, body-portalled, inert bg)
├─ [optional] header rows at top (per-row checkbox; burger→X)   ← reuses header-CPT groups (DP4)
└─ sgs/nav-menu [accordion mode]  ← SAME wp_navigation source (one-source dual-render)
```
**Data:** one `wp_navigation` post → `SGS_Nav_Menu_Source` → both the bar (inline) and the drawer (accordion).
Mega items carry an ID ref → resolver → `sgs_mega_panel` post rendered at the item's real position.
**Converter (Spec 31/33 Part 2):** emits (a) the `wp_navigation` post (create-then-reference ordering), (b)
one `sgs_mega_panel` post per rich panel (naming convention, idempotent), (c) the header/nav block markup with
the ref wired. Body-portal cloner-emit question (Phase-1 D4) → **emit nested + runtime re-parent, tie via
`aria-controls`** (keep authoring semantic) — flagged for the converter-emittability design-gate.
