# P2.5 Phase 3 — Navigation block architecture

**Plain English.** This decides the *shape* of the new nav system: how many blocks, what each one does,
how they fit together, and — the part you flagged — how the mega-menu is built. It's derived **only** from
the Phase-2 inventory (everything we need × everything we have), with no influence from the old blocks.
Nothing here builds anything; it's the design you sign off, then it gets stress-tested by a council in
Phase 5 before a single line is written.

## 1. What the evidence FORCES (not up for choice — these bind every option below)

These fall straight out of the inventory + the four research passes. They are constraints, not preferences:

1. **The a11y contract splits the surfaces into two families.** A dropdown and a mega-menu are the same
   kind of thing — a **disclosure** (a button that shows/hides links; no focus-trap; Escape closes). The
   mobile drawer is a **dialog** (traps focus, Escape mandatory, background `inert`). These contracts are
   *incompatible* (primary-source-confirmed: W3C APG, MDN, Roselli), so **the drawer cannot be the same
   block as the bar/mega**, and **the bar + mega CAN share machinery** (mega = a bigger disclosure).
2. **Shared behaviour lives in ONE JS utility, not one mega-component.** An open/close/focus/`inert`
   plumbing utility (an SGS `viewScriptModule` + our own Interactivity-style store) is consumed by the
   distinct disclosure and dialog surfaces. Proof-test: write the call-sites; if they need per-mode `if`
   branches, it isn't one primitive. (Council finding #4.)
3. **The drawer renders in the browser top layer** (`<dialog showModal>` / Popover API) so it survives a
   `transform`-ed hide-on-scroll header ancestor by construction — no fragile JS `<body>` portal.
4. **Menu DATA = `wp_navigation` (locked).** The block set is the *presentation + behaviour* layer that
   reads a referenced `wp_navigation` tree and emits its OWN scoped SGS markup (the established
   data-source mechanism — `wp_navigation` stores the link tree; our render.php walks it). Menu structure
   is NEVER a bespoke store and NEVER core-nav's rendered output.
5. **The mega association is by ID/position, and renders at the item's REAL position.** The old block's
   fatal bug (mega could only render LAST → couldn't place Indus "Brands" at position 4) must be
   *structurally impossible* in the new model.
6. **Everything is converter-emittable by construction** (a scraped nav maps onto it; Spec 33 Part 2's
   emit target repoints here) and **styles via scoped `<style>`** (Spec 32) with the **full Spec 35
   control set per block**.

**Two boundaries these settle immediately:**
- **The two-row header is a header-CPT concern, not a nav block.** The utility strip is another
  `sgs/site-header-row` (P2's header builder); the nav is one row inside the header. Clean separation —
  the nav block set does not absorb the header's rows.
- **Bottom-tab-bar and priority+/"More" overflow are MODES of the bar (attributes), not separate blocks.**
  They're behaviours over the same menu data + the same disclosure contract, not distinct editing surfaces.

## 2. The real open decision — three ranked architectures

They differ on **block count** and **the mega-menu mechanism** (your flagged sub-decision). Every option
includes the shared JS plumbing utility + `wp_navigation` data source from §1.

### Option A — **3 blocks + shared plumbing; mega = its own block (InnerBlocks), reuse via native synced patterns** ⭐ RECOMMENDED

| Block | KIND | Does |
|---|---|---|
| `sgs/nav-bar` | layout (sits in `sgs/site-header-row`) | The desktop bar. Reads a `wp_navigation` ref, emits scoped SGS markup, drives the **disclosure** behaviour (dropdown + mega triggers, intent-timing, safe-triangle). Holds the item→panel map + trigger/style/mode attrs (priority+, bottom-tab). |
| `sgs/mega-panel` | layout composite (InnerBlocks) | The rich panel. **Template variants** via `variant_attr` (photo-grid / split-aside-CTA / logo-grid / info-box / link-columns). Placed as an InnerBlock of `sgs/nav-bar`, associated to a bar item by ID/position (`forItem`). Optionally saved as a **native WP synced pattern** for reuse. |
| `sgs/nav-drawer` | section (top-layer dialog) | The mobile off-canvas **dialog**. Renders the same `wp_navigation` collapsed (accordion default / drill-down option) + a CTA/featured slot. Separate by the a11y contract (§1.1). |

- **Mega mechanism:** InnerBlocks in `sgs/mega-panel`. Authored **in place** in the nav context (best
  WYSIWYG for a non-coder), the panel IS an SGS block tree (**best converter target** — the converter
  already emits block trees), reuse comes from WP's **native synced patterns** (Spec 35 Part G "prefer
  native, don't hand-roll" — no bespoke CPT, no bespoke admin screen).
- **Pros:** cleanest 1:1 map to the forced a11y split; minimum block count the split allows; mega ordering
  bug impossible by construction (walk-order render + ID association); every piece is a native block →
  Spec 32 + converter-emit fall out for free.
- **Cons:** cross-menu/cross-site reuse is via synced patterns rather than a single editable record (fine
  for clone-per-site; rare within a site); `sgs/nav-bar` carries the item→panel map (a modest attr).

### Option B — **2 nav blocks + a dedicated mega-panel CPT** (your Astra/Spectra idea)

| Block/store | Does |
|---|---|
| `sgs/nav-bar` | As Option A, but mega items reference an **external CPT post by ID** (a picker in the bar). |
| `sgs/nav-drawer` | As Option A. |
| `sgs_mega_panel` **CPT** (+ render) | Panels authored in a **dedicated admin screen**, reused by ID across menus/sites. |

- **Pros:** panels are reusable + live in a findable admin screen (the Astra/Spectra feel); one edit
  updates every use.
- **Cons:** a **bespoke CPT + admin UX to build and maintain**; the panel is edited **away from the nav
  context** (context-switch for a non-coder); **more converter ceremony** (create CPT post → reference by
  ID → naming convention to avoid client collisions); the reuse benefit is **marginal for single-site
  clones** (each clone is fresh; two items rarely share a panel). Native synced patterns (Option A) give
  most of the reuse with none of the bespoke cost.

### Option C — **Leanest: 2 blocks (one unified nav with variants + the drawer)**

| Block | Does |
|---|---|
| `sgs/nav` | ONE block with **variants** (`variant_attr`: bar / bar+mega); mega panels as InnerBlocks inside it. |
| `sgs/nav-drawer` | Separate (the a11y split forces it). |

- **Pros:** fewest blocks; uses the SGS variant machinery.
- **Cons:** the single block gets **heavy** (bar + every panel + the item map in one); **"variant" is the
  wrong fit** — the SGS variant mechanism models *different structure* (hero `split` vs `video`), whereas
  bar-with-mega is the *same* structure with optional child panels, not a structural variant; the
  mega-panel loses its own-block clarity (worse for reuse + a slightly muddier converter target).

## 3. Recommendation — **Option A**, with reasoning

Option A is the smallest architecture that honours **every** forced constraint, and it wins on the two
things this project weights highest:

1. **Client experience (CLAUDE.md "primary").** The mega panel is authored **in place** with native block
   editing + full Spec 35 controls — a non-coder builds the panel where they see it, not in a separate
   admin screen.
2. **Converter-emittable by construction (DP6).** Every piece is a native SGS block; the converter already
   emits block trees, so a scraped mega panel is just a block tree it writes as InnerBlocks — no bespoke
   CPT-creation step, no naming-collision scheme.

It also gets **reuse from native synced patterns** instead of a bespoke CPT — the exact "prefer native,
don't hand-roll" rule (Spec 35 Part G) — so your CPT instinct is *honoured in outcome* (reusable saved
panels) without the maintenance cost. And it makes the **mega ordering bug impossible**: the panel is a
child rendered in walk-order at its item's real position.

**Your CPT idea isn't wrong — it's the right *feature* (reusable saved panels) reached by a lighter
*mechanism*.** If cross-site panel reuse turns out to be a real, frequent need, Option B is the fallback;
but the evidence says native synced patterns cover it.

## 5. Proof it reproduces the two client menus (Option A)

- **Mama's (flat 5 + featured + cart badge):** `sgs/nav-bar` renders the 5-item `wp_navigation`; the
  "featured" item is an item-level style flag; the cart badge is a bar utility affordance; mobile →
  `sgs/nav-drawer` (accordion) with the CTA slot. Bottom-tab-bar mode is available (5 items fit).
- **Indus (7 items, 3 dropdowns, 1 mega at position 4, 5 panels, 2-row header, accordion drawer):**
  `sgs/nav-bar` renders the 7-item `wp_navigation`; 3 items are simple dropdowns (rendered from the
  submenu tree), item 4 ("Brands") has a `sgs/mega-panel` child with `forItem=4` → renders a mega **at
  position 4** (bug fixed); the 5 rich panels are 5 `sgs/mega-panel` instances using the template
  variants; the utility strip is a second `sgs/site-header-row` (header CPT, not nav); mobile →
  `sgs/nav-drawer` accordion/drill-down. Priority+ mode keeps 7 items discoverable at mid-widths.

## 6. Converter-emit mapping (DP6 — by construction)

`Spec 33 Part 2` clones a draft header/nav → writes a `wp_navigation` post (link tree) + a `sgs/nav-bar`
(with item→panel map + mode/style attrs) + `sgs/mega-panel` InnerBlocks (one per rich panel, template
variant chosen by the panel's layout) + a `sgs/nav-drawer`. All native blocks → no inline styles, no core
blocks, no bespoke CPT step. The emit target **repoints from `sgs/adaptive-nav`** here.
