# P2.5 — Navigation Requirements + Available-Tooling Inventory

**One-liner (plain English).** Before we decide how many nav blocks to build and what each does, this
doc lays out **everything we need the new navigation to do** (NEED) next to **everything we already
have to build it with** (HAVE). The block architecture (Phase 3) is decided from THIS inventory only —
with zero influence from the current or previously-planned nav architecture (Bean-locked).

## 0. Scope, ground rules, and what's locked

- **All THREE current nav blocks are REFERENCE-ONLY** — `sgs/adaptive-nav`, `sgs/nav-menu`, and
  `sgs/mega-menu` are gone; none is carried forward (Bean, 2026-07-18: full rework, no salvage — the
  `nav-menu` addition confirmed mid-session on 2026-07-18). Their behaviour informs the NEED list;
  their code does not constrain the architecture.
- **LOCKED (Bean):** (1) full rework of the whole set; (2) menu data = `wp_navigation` CPT; (3) feature
  bar = WP-core-baseline + top competitors + general web/UX + Spec 35 + more (meet-and-exceed);
  (4) mega-menu is a rebuild — its *implementation* is NOT locked (a CPT is a candidate to weigh in
  Phase 3, not a decision).
- **DELIBERATELY OPEN (decide in Phase 3 from this inventory):** number of blocks, what each does, how
  they compose, and the mega-menu implementation.
- **Phase-1 evidence base:** four sourced research reports (dated 2026-07-18) — mega/dropdown UX,
  mobile/drawer/emerging, accessibility-beyond-WCAG, and a live visual teardown of Stripe/Apple/Nike/
  Shopify. Every NEED row below that came from research cites the report + section it came from.
- **Verification discipline (§15.7 lesson):** platform-capability claims are HYPOTHESES until checked
  against source. Rows that rest on a browser/WP capability carry a **⚠VERIFY** flag naming what must
  be confirmed against MDN/caniuse/live-WP before Phase 3 locks or before build.

## 1. PHASE 1 SYNTHESIS — the convergent findings (what a WP-competitor-only feature set misses)

The four passes converge hard on a small number of high-leverage points. These are the "meet-and-exceed"
differentiators — the things you only get by studying the general web, not just Kadence/Spectra/etc.

| # | Convergent finding | Sources (Phase-1 reports) | Why it's load-bearing |
|---|---|---|---|
| S1 | **Open/close interaction mechanics beat the visual panel.** The make-or-break of rich desktop nav is *intent-based timing* (open-delay ~300–500ms, close-grace ~0.5s) + a **safe-triangle / prediction-cone** hover path — not the panel's looks. WP builders under-build exactly this. | mega-dropdown A1–A5, G2; visual (all 4 exemplars hover-open with restraint) | This is the single biggest quality gap vs competitors, and it's vanilla-JS achievable. |
| S2 | **The ARIA architecture is a hard split: disclosure vs dialog.** Desktop dropdown + mega = **disclosure** (`<button aria-expanded>`, NO focus trap, Escape, no `aria-haspopup`); mobile drawer = **dialog** (`role=dialog`, `aria-modal`, focus-trap, mandatory Escape, `inert`). Contracts are *incompatible* → cannot be one component. Mega is a *bigger disclosure* → dropdown+mega CAN share. | accessibility HEADLINE + §H decision table; council finding #4 | Directly governs how many components the a11y layer forces. Primary-source-confirmed (W3C APG, MDN, Roselli). |
| S3 | **The drawer's correct home is the browser top layer** (`<dialog showModal>` / Popover API) — escapes `transform`/`overflow`/`z-index` ancestors *by construction*, giving `::backdrop` + focus handling largely free, no JS `<body>` portal. This is the clean fix for the council's transform-ancestor / hide-on-scroll landmine. | mobile-emerging 5.1/5.4, top-gap #1; council finding #7 | Turns a known fragility into a solved default. ⚠VERIFY Popover/`<dialog>` Baseline + WP-context usability at build. |
| S4 | **One collapsed breakpoint, not a tablet-specific layout.** All 4 exemplars use the *same* drawer at 768 as at 375; only Apple's bar survives past 768 at all. | visual "MOBILE DRAWER" §; matches Spec 35 D2 (768/1024 device tiers) | Simplifies the responsive model — design bar → (optional priority+) → one drawer. |
| S5 | **Content-driven mega-panel templates, not one rigid grid.** Best navs ship several panel layouts (photo-grid / split-with-aside-CTA / logo-grid / info-box / plain link-columns) chosen per menu. Indus alone needs five. | mega-dropdown B13, top-gap #5; visual (Stripe featured-card / Shopify photo-cards+sidebar / Apple pure-type / Nike dense-columns) | Required to "reproduce the best navs" and to fix the current mega (which the council proved is broken). |
| S6 | **The drawer is a landing surface, not just links.** Every exemplar pins conversion CTAs to the drawer bottom; several add a featured/promo slot. | visual "MOBILE DRAWER" §; mega-dropdown B6/B7 | Nav must carry CTAs/featured/cart-badge as first-class, not bolted-on. Maps to both client menus. |
| S7 | **Progressive enhancement / no-JS baseline is achievable now.** Semantic `<nav>`+`<ul>`+`<a>` that works without JS; `<details name>` exclusive-accordion as the no-JS drawer-accordion; Popover for JS-free dropdowns. Aligns with SGS vanilla-JS/no-jQuery + converter reproducibility. | mega-dropdown F2; mobile-emerging 6.2/6.3; a11y A4 | Robustness + crawlability + clone-fidelity all improve. |
| S8 | **Visible-nav-first, hamburger-last.** Hidden nav measurably hurts engagement; show links up to ~4–5, use priority+/"More" overflow before collapsing, label the hamburger, don't box it. Bottom tab-bar is a real mobile mode WP competitors rarely offer. | mobile-emerging §1, §4, §6.1 | Evidence-backed defaults that quietly beat the generic competitor hamburger. |

**Cross-cutting emerging tech (watch, don't hard-depend — all ⚠VERIFY at build):** Popover API (Baseline —
usable now), CSS anchor-positioning (NOT baseline — progressive enhancement), View Transitions (SPA baseline
~Oct-2025 / MPA Chromium-mostly), scroll-driven animations (Chromium full, others partial), `<details name>`
exclusive accordion (broad; animation newer). Ship vanilla-JS baselines; layer these behind capability
detection. (mobile-emerging §6, §8.)

## 2. NEED — everything the new navigation must be able to do

### 2A. The two client menus — the concrete reproduction targets (the acceptance gate)

These are the pass/fail baseline (council-added hard requirement; R-31-9 universality is proven *by*
reproducing both + generalising).

| Menu | Structure | Nav features it forces |
|---|---|---|
| **Mama's Munches** (restaurant) | Flat 5-item bar + **1 "featured"/highlighted** item + **cart badge**. Collapses to a drawer on mobile. | featured/promoted bar item; cart-count badge affordance; small flat bar (bottom-tab-bar candidate, S8/§4); simple drawer. |
| **Indus Foods** (B2B trade) | **7-item bar**, 4 expandable = **3 simple dropdowns + 1 rich MEGA "Brands" at bar position 4**; **5 distinct rich mega-panel layouts** (photo-grid / split-with-aside-CTA / logo-grid / info-box); **two-row header** (utility strip + nav row); accordion/drill-down collapse on mobile. | mega at an ARBITRARY bar position (the current block's fatal bug — always-last); multiple panel templates (S5); simple-dropdown-vs-mega decision (B10); two-row header (D6); dropdown + mega coexisting in one bar; accordion+drill-down drawer. |

### 2B. Core-nav feature BASELINE (Stream A — the floor we must meet or exceed)

WP `core/navigation` is ruled OUT as the UI engine, but its *feature set* is the base to match: menu bar,
submenus, an overlay/mobile menu, a menu-data model (`wp_navigation`), block-level styling, responsive
collapse. We take the data model (`wp_navigation`) and the feature intent; we do NOT take its rendering
(inline styles, full-screen-only overlay, no mega, no per-device tiers, unsettled submenu a11y). *(P2 §15.0
Stream A.)*

### 2C. Competitor features (Stream B — the "match the field" layer)

The strong WP builders (Kadence/Spectra/GenerateBlocks/Bricks/GreenShift) extend WP's nav plumbing and
converge on: sticky/transparent/shrink headers, off-canvas drawer, mega-menu (as a saved template/pattern),
per-device visibility, one reusable overlay/panel primitive, hover/click submenu triggers, menu-item
badges/icons/descriptions. *(P2 §15.0 Stream B.)* These are table-stakes — the NEED list below (2D) is the
layer that *exceeds* them.

### 2D. General web / UI-UX best practice (Phase 1 research) — the "exceed the field" layer

**Desktop rich nav — trigger, timing, IA, wayfinding** *(mega-dropdown report)*

| Need | Tier | Ref |
|---|---|---|
| Hover-intent open delay (~300–500ms) + close-grace (~0.5s) as **tuned block attributes** | MUST | A1–A3 |
| **Safe-triangle / prediction-cone** hover path (solves the diagonal "menu runs away") | MUST if panels offset | A4–A5, G2 |
| Hybrid trigger: hover-opens + click-also-opens + top item still navigates (keyboard/touch parity) | MUST | A7–A8 |
| Simple-dropdown-vs-mega **decision rule** baked into the model (don't build a mega for 4 links) | MUST (rule) | B10, E8 |
| Column grouping, group headings, one-item-per-group, vertical scan order, "view all", descriptions | MUST/SHOULD | B1–B4, B8, B11 |
| Meaningful imagery/thumbnails (not decorative filler); **featured slot + in-panel CTA/aside** | SHOULD | B5–B7 |
| Multiple content-driven **panel templates** (photo-grid / split-aside-CTA / logo-grid / info-box / columns) | SHOULD (MUST for Indus) | B13 |
| Caret/chevron affordance ONLY on expandable items; active-trail "you are here" (bar + panel) | MUST | C1, C3 |
| Distinct hover AND focus states; soft in-panel hover highlight; cart/utility badges | MUST/SHOULD/NICE | C4–C6 |
| Panel is content-sized overlay, not full-screen takeover; don't cover the whole screen | MUST | B12 |

### 2E. Accessibility contract (the architecture-shaping requirement) *(accessibility report)*

The **disclosure/menubar/dialog decision table** (accessibility §H) is the governing a11y contract. Headlines
that bind the architecture:

- Dropdown + mega = **disclosure** (`<nav aria-label>` + `<button aria-expanded>`, `aria-controls` SHOULD,
  **OMIT `aria-haspopup`**, Tab-through NO trap, Escape closes, focus returns). **NOT `role=menu/menubar`.**
- Drawer = **dialog** (`role=dialog` `aria-modal=true` labelled, focus INTO on open, Tab CONTAINED,
  **Escape MANDATORY**, focus return, background `inert` + visible scrim).
- Hover surfaces MUST satisfy **SC 1.4.13** (dismissible/hoverable/persistent).
- Plus the "basic-pass misses" set: `aria-current="page"`; unique labels on multiple `<nav>`s; icon-button
  accessible names; `:focus-visible` (≥3:1 non-text contrast, SC 1.4.11); **SC 2.4.11 Focus Not Obscured**
  (sticky bar must not fully hide focus); **SC 2.5.8 Target Size 24px** (SGS keeps 44px, AAA); skip-link
  visible-on-focus; `prefers-reduced-motion`; `forced-colors`/HCM (shadow-only borders vanish);
  no colour/motion-only state signalling.

### 2F. Spec 35 control-completeness (the per-block editor DoD)

Every new nav block must pass **Spec 35 Part L** (per-block inspector definition-of-done). Nav-relevant
control needs: `LinkControl` for every menu item / CTA (internal search + new-tab + rel); `StateToggleControl`
for hover/normal states; `ResponsiveControl` for the 768/1024 device tiers; `ToolsPanel` progressive
disclosure on dense panels; `enableAlpha`+clearable colour; box-object attrs (no inline); `hideExtensions`
for irrelevant universal panels; icon controls / `MediaGalleryPicker` (logo-grid mega); reduced-motion gate.
*(Spec 35 Parts B/C/G/I/L.)*

### 2G. Converter-emittability (the clone-target requirement — DP6)

The cloning pipeline (Spec 31 §13 + Spec 33 Part 2) must be able to write a scraped draft nav INTO the new
blocks — **by construction**, not deferred. Concretely (council-verified needs):
- A converter can write a `wp_navigation` post of N items / M submenus (create-then-`ref` ordering).
- A mega item marks "has a panel" by **ID/position reference, NOT slug-string**, and renders at its **real
  bar position** (the current block's always-last bug must be impossible in the new model).
- A converter-owned template/panel **naming convention** so clients never collide (e.g. per-client scoping).
- Panel content emittable as SGS blocks (no core blocks → Spec 32 no-inline holds).
- Spec 33 Part 2's nav emit target **repoints** from `sgs/adaptive-nav` to the new block(s) — a required
  consequence of this rework.

### 2H. Constraints that bind every candidate architecture

Spec 32 no-inline (scoped `<style>` via `wp_style_engine_get_styles`, box-object attrs, responsive tiers +
`:hover` in stylesheet rules, custom breakpoints → `sgsCustomCss`); Spec 35 control-completeness; Spec 31/33
converter-emittable; Spec 17 §S9 (one-menu-source, drawer a11y) + Spec 34 (disclosure drawer geometry);
WCAG 2.1 AA baseline (+2.2 cheap wins); `viewScriptModule` ES-module vanilla JS, **no jQuery**; UK English;
works perfectly inside the header CPT (`sgs_header`); **transform-ancestor drawer survival**; performance
budget (<100KB CSS / <50KB JS per page).

## 4. Constraints + signals the evidence imposes on ANY architecture (inputs to Phase 3, NOT the decision)

These fall directly out of §1–§3. They constrain the Phase-3 architecture without prescribing block count.

1. **A11y forces a component split (S2/2E):** disclosure (dropdown+mega — CAN share) vs dialog (drawer —
   MUST be separate). A single unified nav component is ruled out by primary-source ARIA contracts.
2. **Shared plumbing, not one component (council #4):** an open/close/focus/`inert` UTILITY consumed by the
   distinct disclosure and dialog surfaces — prove it by writing the call-sites; per-mode `if` branches mean
   it isn't one primitive.
3. **Top-layer drawer by construction (S3):** the drawer renders in the top layer (`<dialog>`/popover), not
   as a `position:fixed` descendant of a possibly-transformed header.
4. **One collapsed breakpoint (S4):** bar → (optional priority+) → one drawer; no separate tablet layout.
5. **Mega ≠ one grid (S5/2G):** content-driven panel templates + ID/position reference + real-position
   render are mandatory; the always-last bug must be structurally impossible.
6. **Menu-DATA is `wp_navigation`; the block set is the presentation/behaviour layer over it (locked).**
7. **Converter-emittable by construction (2G/DP6):** whatever the block count, a scraped nav must map onto it,
   and Spec 33 Part 2's emit target repoints here.
8. **Everything styles via scoped `<style>` (Spec 32) + full Spec 35 control set per block.**

**Open questions Phase 3 must answer (from this inventory):**
- How many blocks, and the split between: a nav-bar/header-nav block, a menu/item model, the mega-panel
  mechanism (3D a/b/c), the drawer, the shared plumbing utility, priority+/bottom-tab modes.
- Is the "bar" one block with variants (`registerBlockVariation`) or several blocks?
- Where do mega panels live (CPT vs InnerBlocks vs template-part) — decided with the converter-emit + client-
  edit UX in mind.
- Bottom-tab-bar and priority+: separate blocks, modes of the bar block, or deferred?
- Two-row header: nav concern or header-CPT (P2) concern? (boundary with the header builder).
