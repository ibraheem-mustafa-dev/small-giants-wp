## PART A — NEED (everything the nav must do)

### A1. WP core-nav feature BASELINE (Stream A — the floor we must meet)
The features WP's own `core/navigation` exposes, which we must at least match (it's ruled OUT as the engine,
but its feature set is the base of "meet-and-exceed"):
- Menu data from a reusable source (WP provides `wp_navigation` CPT — we keep this, FR-S9-4 locked).
- Submenus (one level), open-on-click or open-on-hover option.
- A mobile "overlay" menu with an open/close toggle.
- Justification/orientation/spacing layout controls; menu colours (text/background, submenu, overlay).
- `aria-label`, basic keyboard operability, an overlay close button.
- **Baseline gaps in core nav we must EXCEED (Stream A):** no mega-menu (every WP mega is a plugin); overlay
  is full-screen-only (no header-visible drawer); emits inline styles (breaks Spec 32); no per-device tiers;
  submenu a11y unsettled (4 open core issues). → we must beat every one of these.

### A2. Competitor features to ADD (Stream B — Kadence/Spectra/GenerateBlocks/Bricks/GreenShift)
(Full matrix in P2 §15.0 Stream B — summarised here as NEED lines.)
- **Mega-menu** with full-width and element-anchored panels; rich panel content (columns, headings, images,
  CTAs, promos).
- **Multiple header/nav layouts** (logo-left/centre, split nav, stacked); pre-configured nav "flavours".
- **Off-canvas drawer** with configurable side, width, background, close-icon, content alignment, inner spacing.
- **Per-device responsive** everything (the tier system core nav lacks).
- **Sticky / transparent / hide-on-scroll** header behaviours.
- **Search, cart, account, CTA** slots in the header; announcement/utility bar.
- **Collapse breakpoint** configuration; overflow "More".
- The lean convergence (GenerateBlocks + GreenShift): preserve `wp_navigation` compat; ONE overlay/panel
  primitive family reused across surfaces (SGS reads this as "shared plumbing", not one component — R2 below).

### A3. General web / UI-UX best practice to ADD (Phase-1 research — the meet-and-EXCEED layer)
The ranked MISSING list from the Phase-1 findings doc. Reproduced compactly; full sourcing in that doc.

**MUST (definition-of-done):**
1. Disclosure pattern for bar/dropdown/mega (`button[aria-expanded][aria-controls]` + real `<a>`; no `role=menu`).
2. Dialog contract for the drawer (focus containment via `inert`, ESC-close, focus-return) — separate from #1.
3. Click-primary interaction + full keyboard (Enter/Space/Esc); hover only as a degrade-able enhancement.
4. `aria-current="page"` auto-wired (bar, breadcrumb, scrollspy).
5. Mandatory group headings inside every mega panel.
6. One menu source → dual render (mega columns → drawer accordion sections; no content dropped).
7. Body-portal so drawer/overlay survives a transformed (hide-on-scroll) header ancestor.
8. Content-driven collapse breakpoint (measured against actual rendered nav width), not a hard device px.
9. CLS-safe sticky (`position:sticky` + reserved height + IntersectionObserver sentinel).
10. WCAG 2.2 **2.4.11 Focus Not Obscured** (sticky header never hides a focused item).
11. WCAG 2.2 **2.5.8 Target Size** — keep our 44px bar on every link/toggle/close.
12. `aria-expanded`/`aria-controls`/`aria-hidden` wiring + accessible names on icon-only toggles.
13. Cart-badge accessible live count (`role="status" aria-live="polite" aria-atomic="true"`).

**SHOULD (strong meet-and-exceed):**
14. Top-level item = real link with a real landing page, independent of any toggle it also drives.
15. Distinct ≥44px chevron/caret separating "expands" from "navigates".
16. Tunable hover-delay constants IF a hover layer ships (open ~300–500ms, close ~500ms grace) — distinct from "animation speed".
17. Two-row header (utility/announcement + nav) as first-class slots with INDEPENDENT sticky/dismiss state.
18. Breadcrumb-in-drawer / persistent parent-label for drill-down.
19. Priority+/overflow option so low-item-count menus stay visible instead of defaulting to hamburger.
20. Command palette (Cmd/Ctrl-K, ARIA combobox, Esc-close, never the only route).
21. Sticky sub-nav / in-page jump-nav auto-derived from H2 blocks (+`scroll-margin-top`).
22. `:focus-visible` (≥3:1) not bare `:focus`.
23. `aria-haspopup` only when its value matches the popup role, else omitted.
24. Full-viewport backdrop scrim, tap-to-close, body-level stacking.
25. Link-count / column ceiling as an editor INFORMATIONAL notice (NOT a gate — locked policy, Phase-3 D5).
26. Skip-to-content link as the first focusable element, not wrapped in `<nav>`.

**NICE (later waves / data-dependent):** hide-on-scroll header · transparent→solid / shrink-on-scroll ·
Amazon safe-triangle (PROMOTED to v1 — hover is the default open behaviour, Bean 2026-07-19) · scroll-progress · back-to-top · mega imagery/icon slots ·
one capped promo slot · personalised mega content (needs a data pipeline — flag as dependency).

### A4. Spec 35 control-completeness (Part L — the per-block inspector definition-of-done)
Every nav block that ships must pass Part L. Nav-relevant lines: Settings/Styles/Advanced `group` split ·
element-first panels · ToolsPanel on dense panels · colour controls with `enableAlpha`+clearable · lengths
via UnitControl or spacing tokens · 4-value props via `box_family` · shadow/border via real builders · links
via `LinkControl` (new-tab+rel) · responsive props expose the 768/1024 device switcher · states via
`StateToggleControl` · irrelevant universal panels hidden via `hideExtensions` · native supports used over
hand-rolled (sticky/duotone/aspectRatio) · every animation `prefers-reduced-motion`-gated · ARIA-label +
decorative-image where needed · no native-supports panel duplicated · no Part-F anti-patterns.

### A5. Concrete client menus to REPRODUCE (Stream C — the acceptance gate)
The build is not done until BOTH reproduce live + Bean's eye (P2 §15.6 added requirement):
- **Mama's Munches:** flat 5-item bar · 1 featured item · **cart badge with live count** · collapses to drawer.
- **Indus Foods:** 7-item bar · 4 expandable (3 dropdowns + 1 **mega "Brands" at position 4 of 7**) · **5 fully-
  built rich mega panels** (photo-grid / split-with-aside-CTA / logo-grid / info-box shapes) · **two-row header**
  (utility bar + nav) · accordion/drill-down collapse.
- **Universality (R-31-9):** must generalise beyond these two (restaurant, law firm, charity) — no client carve-outs.

### A6. Feature-parity with replaced core blocks (Spec 35 standard, Bean-locked 2026-07-18)
Every SGS nav block must expose AT LEAST the full functionality of any core block it replaces (`core/navigation`,
`core/navigation-submenu`, `core/search`), unless a named exception. A core capability the SGS block lacks is
a parity GAP to close, not silently dropped. [[sgs-block-feature-parity-with-replaced-core]]

### A7. Intrinsic shrink-to-fit (Spec 35 block standard, Bean-locked 2026-07-18)
Every nav block must be intrinsically responsive (min-content ≤ container at every breakpoint) — shrink to fit
on its own, not forced by a container clamp. The framework `min-width:0` grid/flex safeguard is a backstop,
not a substitute. [[blocks-must-shrink-to-fit-container]]

### B2. GENERAL reusable WP mechanisms (§15.2 — verified callable by a bespoke block)
- **`wp_navigation` CPT + `ref`** — the menu-data source (LOCKED). `SGS_Nav_Menu_Source` already resolves it.
- **`registerBlockVariation`** — ship pre-configured nav flavours without forking blocks.
- **Block Hooks API** — optional auto-insert of an SGS CTA/utility block (template-context only; LOW for SGS which clones to Pages).
- **`render_block` filter** — generic post-process escape hatch (sparingly).
- **Interactivity API as a PATTERN to PORT** — SGS writes its OWN `data-wp-interactive="sgs/..."` namespace +
  its own store, shipped as a `viewScriptModule`. A net-NEW build, NOT a free reuse; only for surfaces being
  built fresh.
- **`wp_template_part` (DB `wp_insert_post`, portable) OR theme file** — storage option for a saved mega panel (Bean's CPT candidate territory — decided Phase 3).
- **Block Bindings API** (WP 6.5+) — for any dynamic content (Site Info in header/footer, cart count source).
- **Native block supports** — `position.sticky`, `dimensions.aspectRatio`/`minHeight`, `shadow`, `layout`,
  `filter.duotone`, `background` (free inspector UI + CSS; Spec 35 Part G says prefer these over hand-roll).
- **`templateLock:"contentOnly"`** — client-safe editing of nav/header patterns (free).

### B3. CORE-NAV-PRIVATE — CANNOT be called (§15.2 — do NOT budget as free; verified vs source)
- `get_nav_element_directives()` — `private static`; hardcodes `core/navigation` namespace.
- `block_core_navigation_render_inner_blocks` / `_listable_blocks` filters — fire only inside core nav's render.
- `get_overlay_blocks_from_template_part()` — `private static`; full-screen-only (geometry mismatch is terminal).
- Arbitrary-SGS-blocks-inside-a-`wp_navigation`-post — needs a LIVE check (CPT block-list may reject non-nav blocks); do NOT assume before build.
- **§15.7 discipline binds Phase 3+:** re-verify every platform capability against source before relying on it.

**Headline:** the menu-DATA layer, the container/wrapper layer, the responsive-emission layer, and most
inspector components ALREADY EXIST. The genuinely NEW build is (1) the two distinct interaction behaviours
(disclosure + dialog) under an SGS Interactivity namespace with a shared focus/inert utility, (2) the
rebuilt mega-panel with a real data-shape + position-correct placement, (3) the header scroll/sticky state
machines, and (4) a handful of shared inspector-control gaps (alpha, LinkControl, gallery, shadow).
