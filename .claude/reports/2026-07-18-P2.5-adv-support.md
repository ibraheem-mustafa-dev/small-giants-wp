# Adversarial Council ‚Äî Support Realist ‚Äî Spec 36 (SGS Navigation System) v1.1

**Persona:** the person who takes the 11pm "my menu is broken" call from a non-coder client.
**Reading:** full text of `.claude/specs/36-SGS-NAVIGATION-SYSTEM.md` v1.1, 2026-07-18.

---

## 1. STRENGTHS

- **Stable-ID keying for item‚Üímega (FR-36-5)** kills the two worst tickets before they exist: reordering the
  bar no longer silently detaches a mega panel, and "mega can only render last" is structurally fixed. This
  is the single best support-load decision in the doc.
- **Drawer-as-full-container with a default template (FR-36-6)** ‚Äî `[container, nav-menu, container]` ‚Äî
  means a fresh drawer is not a blank white box out of the gate. A client who never touches it still gets a
  logo slot + menu + second slot, not nothing.
- **Modality-follows-geometry (¬ß11e)** removes a whole class of "the menu button does nothing on my phone"
  tickets that a botched non-modal/modal split would cause ‚Äî one rule, not a per-client toggle to get wrong.
- **Menu-source independence (FR-36-1/¬ß11b) is at least named explicitly** ("neither inherits the other") ‚Äî
  better than silently diverging behaviour a client discovers by accident.
- **Cutover gate exists at all (FR-36-18)** ‚Äî canary-first, live before/after check, zero-stored-instance
  gate before deletion. Most internal specs skip this; this one names it.

## 2. MUST-FIX (ticket-generators / live-breakers)

**M1 ‚Äî No deep link from the bar item to the mega panel it controls (FR-36-5 / FR-36-14).**
The mega is edited on a *different admin screen* than the menu. The spec's only nod to discoverability is
"a mega-panel picker" listed as an *example* of optional custom UI ("welcome where it improves UX," not
mandated). For a tech-illiterate operator, "the menu lives in Appearance ‚Üí Menus (or the nav block), the
rich content lives in a completely separate CPT screen, and the thing that connects them is an internal
`wp_navigation` item ID the operator never sees" is invisible plumbing. Concretely: a client wants to change
the photo in the "Brands" mega. They will not know a `sgs_mega_menu` CPT exists, will not find it in the
default admin menu unless it's surfaced, and will file a ticket or edit the wrong thing (e.g. try to add
images inside the plain menu item, which does nothing). **Fix:** make the "edit this mega panel" deep link
from the nav-menu block's per-item settings a REQUIRED control (not an example of optional UI) ‚Äî clicking a
mega'd item in the inspector opens/creates the linked `sgs_mega_menu` post directly. Cite: FR-36-5 (item‚Üímega
map), FR-36-14 (lists it only as an example of *welcome* custom UI, not required).

**M2 ‚Äî Deleting a menu item with a mega attached orphans the CPT post, with no detection.**
FR-36-5 stores the map on `sgs/nav-menu` keyed by item ID. Nothing in FR-36-5, FR-36-6, or FR-36-18 says
what happens when the operator deletes that `wp_navigation` item (in the native menu editor, which they WILL
use ‚Äî it's the standard WP UI). The map entry becomes a dangling ID‚Üípost reference; the `sgs_mega_menu` post
itself sits orphaned in the CPT list with no owner. Worse: if the operator later adds a NEW item with the
same label, it gets a NEW stable ID (WP navigation items are never ID-recycled) and has NO mega ‚Äî the client
sees "I renamed 'Our Brands' back but the panel's gone" and can't self-diagnose why, because from their side
it's the same menu label. **Fix:** render.php on a stale/missing item ID must fail closed (skip silently,
never crash the bar) ‚Äî but the CPT admin list needs an "orphaned ‚Äî not attached to any menu" indicator so
Bean's QC (or the client) can find and clean these up, and the item‚Üímega map needs a prune-on-save hook when
`wp_navigation` items are removed. Cite: FR-36-5, no FR currently owns this.

**M3 ‚Äî Dangling `drawerRef` when the target drawer block is deleted.**
FR-36-8: burger carries `drawerRef`; "if unset it targets the single drawer on the page; multiple drawers ‚Üí
explicit pick." No FR covers what happens when a client deletes the `sgs/nav-drawer` block the ref points to
(e.g. redesigning the footer/header layout, or an accidental Backspace in the editor). Live result on a
production Mama's/Indus site: the burger renders, `aria-expanded` toggles, and **nothing opens** ‚Äî a
completely silent failure a non-coder cannot diagnose ("the menu button stopped working" with zero console
guidance for them to relay). **Fix:** render.php must detect a dangling `drawerRef` at render time and either
(a) fall back to "the single drawer on the page" resolution used for the unset case, or (b) render a
`data-sgs-nav-error` editor-only Notice (per FR-36-12's informational-only precedent) so this is caught at
edit time, not discovered live. Add to FR-36-8 or FR-36-9.

**M4 ‚Äî Bar and drawer using different menus is a divergence trap with no operator-facing signal.**
¬ß11b makes this a first-class feature: bar and drawer "may reference different menus... neither inherits the
other." That's architecturally fine, but it is also THE classic support call: "I updated my menu but mobile
still shows the old one." A client editing the (default, shared) menu has no reason to know a SECOND,
independent menu binding exists on the drawer instance until it silently diverges ‚Äî e.g. an operator resets
just the drawer's `sgs/nav-menu` block while restyling mobile and doesn't realise it now points to a
different/empty menu. **Fix:** each `sgs/nav-menu` instance's inspector must show which menu it's bound to
in plain language ("This menu shows: Primary Menu" / "‚öÝ different from the desktop bar"), not just a `ref`
attribute nobody sees. Cite: FR-36-1, ¬ß11b ‚Äî currently silent on operator-facing surfacing.

**M5 ‚Äî Two independent item‚Üímega maps (bar instance vs drawer instance) double the data-entry burden with
no cross-population, and nothing in the spec says otherwise.**
The map is "stored on `sgs/nav-menu`" (FR-36-5) ‚Äî but there are (at minimum) two `sgs/nav-menu` instances on
a header+drawer page (bar + drawer-internal). If each instance keeps its OWN map, a client who sets up 4 mega
panels on the desktop bar will find the SAME items render as plain dropdowns inside the mobile drawer,
because the drawer's `sgs/nav-menu` instance has an empty map. FR-36-5's "Mobile: the same panel content
renders inside the drawer" language *implies* a shared map keyed off the same menu, but the FR never states
the map is de-duplicated/shared across instances referencing the same `ref` menu, and given FR-36-1's
explicit "menus may differ per instance" allowance, the two could easily also carry two separate,
un-synced maps. **Fix:** state explicitly whether the item‚Üímega map is (a) keyed per-instance (client must
set it up twice, badly, and WILL diverge) or (b) keyed per-menu `ref` and shared by every instance pointing
at that menu (correct ‚Äî one setup, both surfaces stay in sync). Recommend (b); currently unstated. This is a
day-one setup bug, not a day-two edge case ‚Äî Indus WILL hit this at build (FR-36-16's own acceptance case has
1 mega + bar + drawer).

**M6 ‚Äî No JS fallback stated for the desktop bar/mega disclosure (only the drawer has one).**
FR-36-6 explicitly gives the drawer accordion a `<details name>` no-JS fallback. FR-36-7 asserts "the nav
functions without JS" generally, citing "the semantic base + the `<details>` accordion" ‚Äî but the bar/mega
disclosure trigger is a `<button aria-expanded>` (FR-36-4/FR-36-10), and buttons do nothing without their
JS handler. If `viewScriptModule` fails to load (ad-blocker interference, a CDN hiccup, a plugin conflict ‚Äî
all real production events, not edge cases), every dropdown AND mega submenu on desktop becomes permanently
collapsed and unreachable by mouse/keyboard ‚Äî even though FR-36-17 correctly guarantees the LINKS are present
in server HTML for crawlers, real visitors on that broken page cannot reach them. This is a "the whole
website's navigation is broken" ticket, not cosmetic. **Fix:** either give top-level disclosure buttons a
CSS-only fallback (`:focus-within`/`<details>`-as-button pattern) or explicitly document the residual risk
and a monitoring/alerting story. Currently FR-36-7's "functions without JS" claim is FALSE for the bar as
specced ‚Äî only the drawer has the actual no-JS path.

## 3. SHOULD-FIX

**S1 ‚Äî No menu selected / no primary menu exists yet (fresh site, before onboarding).** FR-36-1 defaults to
"the site's primary menu" ‚Äî on a brand-new SGS site before any menu is created, that's nothing. Undefined
whether the bar renders empty, shows an editor-only placeholder, or errors. Needs an explicit empty-state:
editor shows "Select a menu" placeholder (never a blank live bar with a burger that opens an empty drawer).

**S2 ‚Äî Mega CPT post trashed/deleted while still referenced.** WP-native "Move to Trash" on a `sgs_mega_menu`
post is one click away for anyone with edit access to that screen. FR-36-5's "real-position render" logic
needs an explicit graceful-degrade rule: trashed/missing referenced post ‚Üí the item falls back to rendering
as a plain link (or its native submenu if any), never a broken/empty panel or a PHP notice.

**S3 ‚Äî Cache staleness after an operator edits the menu/mega, not just the dev QC gate.** FR-36-16 mandates
cache-clear *before measurement* (dev workflow) but says nothing about what happens when a real client edits
a menu item or mega panel on a LiteSpeed-cached live site ‚Äî "I added a link but it's not on the live site" is
a guaranteed ticket without an auto-purge-on-save hook for `wp_navigation` / `sgs_mega_menu` saves.

**S4 ‚Äî Priority+ "off by default" vs the Indus acceptance test assuming it's on.** FR-36-8 ships priority+ as
opt-in/off-by-default; FR-36-16's Indus acceptance line says "priority+ keeps 7 items discoverable
mid-width" as if it's simply true. Unless someone explicitly turns it on for Indus at build, the 7-item bar
will overflow/collapse early with no overflow affordance. Needs an explicit build-step: "enable priority+ on
the Indus header" (or fold it into FR-36-18's cutover checklist), not an assumption.

**S5 ‚Äî Below-header non-modal drawer's `inert` scope ("everything except {header row + drawer + scrim +
`#wpadminbar`}") doesn't account for third-party floating UI** (cookie-consent banners, live-chat widgets,
booking-plugin floating buttons ‚Äî all common on SGS client sites, e.g. `sgs-booking`). These would go inert
while the drawer is open, silently hiding a chat bubble a customer expects to reach. Needs an explicit
allowlist mechanism (a filterable selector list) rather than a fixed 4-item scope.

**S6 ‚Äî Starter-pack / operator-created mega title collisions outside the converter path.** FR-36-15 solves
per-client naming collisions for the CLONING emit only. Nothing stops two operator-created mega posts sharing
the same title (e.g. both named "Photo Grid" after using the same starter twice) ‚Äî low severity (WP handles
duplicate titles fine, slugs auto-suffix) but worth a one-line note so it's not mistaken for a gap later.

**S7 ‚Äî Sticky/shrinking header + below-header drawer geometry.** FR-36-9 binds the drawer's top/max-height to
`--sgs-header-height`, and the header can shrink on scroll (Spec 17 FR-S9-9). Unstated whether the drawer
resizes live if the header shrinks WHILE the drawer is open (most likely: user scrolls slightly with the
drawer open) ‚Äî a stale gap or overlap is a visible, screenshot-able bug even if not a support call.

## 4. MISSING (undefined error/empty states, no go-live check, no orphan handling)

1. **No consolidated go-live / launch checklist for an operator's own site build** (as distinct from FR-36-16's
   dev-facing Playwright/axe QC gate). A client finishing a self-serve build has no equivalent "before you
   publish, check: every burger has a live drawer, every mega'd item still exists, the drawer's menu matches
   the bar's" self-check. FR-36-16/18 are internal QC, not an operator go-live gate.
2. **No orphan-detection surfacing anywhere** ‚Äî orphaned `sgs_mega_menu` posts (M2), dangling `drawerRef`
   (M3), stale item‚Üímega map entries after item deletion ‚Äî none of these are listed anywhere as detectable
   or reportable states. They will accumulate silently on every long-lived client site.
3. **No defined behaviour for zero menus / zero drawers on the page** (S1) ‚Äî both the "no menu selected" and
   "burger exists, no drawer block anywhere on the page" cases are unaddressed for the single-drawer
   auto-target rule (FR-36-8) when that single drawer doesn't exist at all.
4. **No error surfacing contract in general** ‚Äî FR-36-12 establishes that a11y feedback is informational-only
   (correct, matches locked policy), but the spec never establishes an equivalent pattern for *structural*
   errors (dangling refs, missing menus, trashed megas). Those are a different class from "your contrast is
   low" ‚Äî they're "your nav is currently broken for visitors" ‚Äî and deserve a harder editor-time warning, not
   silence.
5. **Ownership of the FR-36-18 cutover + FR-36-16 "verify Indus mega count at build" step is unassigned.**
   Both read as "someone must do this" with no named owner, no checklist artefact, no gate that blocks
   `/sgs-wp-deploy` if skipped. Given these are the two production sites, a skipped cutover step is a live
   customer-facing nav going down (mirrors the D336 2.5-hour incident pattern already in this project's own
   decisions.md) ‚Äî this deserves a structural gate, not prose instruction, per this project's own
   `prove-the-cause-before-fix` / structural-enforcement discipline.
6. **No re-clone-vs-live-edit reconciliation rule.** If a client hand-edits a cloned mega panel (adds a
   photo, changes copy) and the site is later re-cloned/re-scraped (the plugin-wide "re-clone, not
   deprecation-migrate" policy, ¬ß1), does the re-clone overwrite the client's hand edits to that
   `sgs_mega_menu` post? Not addressed ‚Äî high risk of "my changes disappeared" given the project's own
   no-deprecations/re-clone convention.

## 5. GRADE

**C+ ‚Äî Operability/Supportability.**

The architecture-level ticket-generators from the OLD nav (position-locked mega, no findable mega home) are
genuinely fixed by the ID-keyed map and full-container drawer. But day-two reality ‚Äî delete an item, delete
a drawer, trash a mega post, diverge bar/drawer menus, lose JS ‚Äî is almost entirely unaddressed: every one of
those produces either a silent broken state or a support call with nothing in the spec telling Bean's QC (or
a future support agent) what SHOULD happen. None of these are exotic; they are the first five things a real
non-coder client will do in month one. Until M1‚ÄìM6 have an explicit fallback/detection rule each, this ships
with a support treadmill baked in, just relocated from "mega renders in the wrong place" to "mega quietly
stops working and nobody can tell why."

---

## 6. Verdict

Fix the plumbing bugs, not just the position bug ‚Äî right now this spec finds the mega, but doesn't know what to do when the client loses it.
