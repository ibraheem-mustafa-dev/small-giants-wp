# P2.5 Adversarial Review — THE WP-PLATFORM REALIST

**Artefact:** Spec 36 — SGS Navigation System (v1.1)
**Reviewer lens:** WordPress core + Gutenberg internals + browser-platform truth. Every platform claim verified against LIVE SOURCE in this repo, or falsified.
**Date:** 2026-07-18
**Grade:** B+ (platform-sound; three claims about "existing surfaces" are wrong or overstated, none fatal)

Legend: **CONFIRMED** = checked against a cited source. **PLAUSIBLE** = consistent with platform knowledge, not live-verified here.

---

## 1. STRENGTHS (verified true against source)

- **FR-36-1 (walk `wp_navigation` → emit bespoke markup) — CONFIRMED, already shipping.** The current `sgs/adaptive-nav` ALREADY does exactly this: `render.php:81-83` reads its own `ref` attribute and calls `SGS_Nav_Menu_Source::get_menu_blocks($ref)`, which does `get_post($ref)` → asserts `post_type === 'wp_navigation'` → `parse_blocks($post->post_content)` (`class-sgs-nav-menu-source.php:131-137`). It walks the parsed tree itself (`find_block_recursive`) and emits its own scoped SGS markup — never core-nav's rendered output. So FR-36-1's core mechanism is not a hope; it is a proven, live code path. The rebuild inherits a working foundation.
- **`ref`-by-reference menu model — CONFIRMED.** `adaptive-nav/block.json:47` declares `ref`, `edit.js:94` selects `wp_navigation` posts via core-data, and the drawer resolves the same source (`class-sgs-nav-menu-source.php` steps 1-5). FR-36-1's "bar and drawer may reference the same OR different menus" is a small delta on an existing resolver, not new infrastructure.
- **`--sgs-header-height` publisher — CONFIRMED EXISTS.** `header-behaviours/view.js:64-95` publishes `--sgs-header-height` (rounded px) to BOTH `documentElement` and `body` via a real `ResizeObserver`, with a `getBoundingClientRect()` degradation path. FR-36-9's below-header drawer binding to this var is sound and rests on a real surface.
- **`<dialog>` top-layer choice for the full-screen modal — CONFIRMED SOUND.** FR-36-6's default (`<dialog showModal>` → top-layer → escapes a transformed header ancestor by construction) is correct platform behaviour: top-layer elements take the viewport as containing block and are not clipped/transformed by ancestor `transform`/`filter`/`overflow`. This is actually SUPERIOR to the current production drawer, which hand-rolls the escape (see MUST-FIX 2 for the caveat).
- **Interactivity-API "port the pattern" (FR-36-7) does NOT smuggle core-nav internals — CONFIRMED, §15.7 trap avoided.** `data-wp-interactive` is an established SGS pattern across `product-card`, `gallery`, `form`, `notice-banner`, `buybox` (all use the PUBLIC `@wordpress/interactivity` `store()` with their OWN namespaces). Declaring `data-wp-interactive="sgs/nav"` is legitimate public-API usage; it is not a call into `core/navigation`'s private store. There is a real in-repo pattern to port from.
- **`sgs_mega_menu` reuses the `sgs_header` CPT machinery — CONFIRMED PLAUSIBLE.** `class-sgs-block-cpts.php:67-161` registers `sgs_header`/`sgs_footer` as standard block-editor CPTs with a `'template' => [['sgs/site-header']]` seed on `init`. Registering `sgs_mega_menu` the same way (block editor + a starter `template` + pattern seeding) is ordinary WP; no bespoke admin UX required.

## 2. MUST-FIX

### MF1 — `data-sgs-header-state` root attribute + state names `visible|sticky|shrunk` DO NOT EXIST. (VERDICT: **FALSE** vs source)
**The claim (FR-36-9):** "The header publishes `--sgs-header-height` ... + a `data-sgs-header-state` root attribute / body classes (`visible|sticky|shrunk` — exists). The nav blocks READ only these."
**Source checked:** `plugins/sgs-blocks/src/header-behaviours/view.js` (whole file) + repo-wide grep for `data-sgs-header-state`.
**Finding:** There is NO `data-sgs-header-state` attribute anywhere in live source — the string appears ONLY in Spec 36 and the P2.5 QC reports. What the header actually publishes is **body classes with different names**: `is-header-scrolled` (transparent→opaque, scrollY>50), `is-header-shrunk` (scrollY>50), and a dormant `is-header-scrolling-down` (`view.js:120-146`). These are toggled on `document.body`, NOT as a root `data-*` attribute, and are gated by input flag classes `sgs-header-behaviour-{transparent,shrink,hide-on-scroll-down}`. The values `visible`/`sticky` do not exist at all.
**Why it matters:** FR-36-9 is the entire header→nav decoupling contract. If a build agent writes the below-header drawer to read `data-sgs-header-state="sticky"`, it reads `null` forever and the adaptation silently no-ops (exactly the class of bug the §15.7 lesson warns about — an assumed surface that isn't there).
**Fix:** Rewrite FR-36-9 to the REAL surface: the nav reads `--sgs-header-height` (real) and the `document.body` state classes `is-header-scrolled` / `is-header-shrunk` (real). If a stable `data-sgs-header-state` attribute is genuinely wanted as the published contract, that is a NEW Spec 17 deliverable to BUILD in `header-behaviours/view.js` — not an existing surface to cite. State it as build-work, and delete the "(exists)" claim.

### MF2 — "`<dialog showModal>` survives a transformed ancestor by construction" is TRUE at the platform level, but CONTRADICTS the deliberate live decision to avoid `showModal()`. (VERDICT: claim **CONFIRMED true**; the spec **omits** the reason the codebase rejected it)
**The claim (FR-36-6):** the full-screen drawer uses `<dialog showModal>`; "top-layer → survives a transformed header ancestor by construction ... do NOT also add `role=dialog`/`aria-modal`."
**Source checked:** `adaptive-nav/view.js:84-131` + `render.php:19-29`.
**Finding:** The CURRENT production drawer deliberately uses **non-modal `dialog.show()`** (`view.js:125`), a **hand-rolled selective `inert`/`aria-hidden` freeze** (`freezeBackground`, `view.js:129`), a **real scrim element** (because `::backdrop` "ceases to exist once we drop showModal()", `render.php:163`), and a **re-parent-to-`<body>` on first open** because "a transformed/filtered ancestor would convert `fixed` into ancestor-relative positioning" (`view.js:102-117`). So the team already hit the transform trap and solved it WITHOUT `showModal()`. Spec 36 is right that `showModal()` would solve the transform trap more cleanly for the cover-header case — but it presents this as settled with no acknowledgement that the live code chose the opposite, and why (they wanted the header row to stay live — which is precisely Spec 36's SEPARATE below-header non-modal variant).
**Why it matters:** Whoever builds this will find production code that says the opposite of the spec. Without the reconciliation written down, they either distrust the spec or re-introduce the hand-rolled path under the modal variant.
**Fix:** Add one paragraph to FR-36-6: "The live drawer used non-modal `.show()` + body-reparent + hand-rolled freeze to keep the header interactive; the full-screen MODAL variant SUPERSEDES that with `showModal()` (top-layer removes the reparent + the manual freeze), while the below-header variant keeps the non-modal approach. Verify top-layer transform-escape on the target Chromium/WebKit/Firefox at build (there was a pre-2022 Chromium bug where top-layer dialogs WERE affected by ancestor transforms — long fixed, but the header's own `transform`-based shrink makes it worth a 5-minute live check, per the render.php:26-29 warning)."

### MF3 — `ResponsiveTriStateControl` + `labelCollapse` (FR-S9-8) are cited as reusable surfaces; NEITHER exists in live source. (VERDICT: **FALSE** as "existing" / TRUE as "P2 design")
**The claim (FR-36-8 / FR-36-14):** per-device visibility "uses `ResponsiveTriStateControl` (P2 §4.1) + Spec 17 FR-S9-8's `labelCollapse` ... reuse, do NOT invent a parallel mechanism (R-31-9)."
**Source checked:** Glob `ResponsiveTriStateControl.js` → **no file**. Grep `labelCollapse` across `plugins/sgs-blocks/src` → **not present** (adaptive-nav does not appear in the match set). P2 §4.1 itself labels `ResponsiveTriStateControl` as **NEW** (`plugins/sgs-blocks/src/components/ResponsiveTriStateControl.js (new)`), and P2 frontmatter records FR-34-5's controls as "absent — additive".
**Finding:** These are unbuilt P2 DESIGN artefacts, not machinery to "reuse". The "reuse, do NOT invent" framing is correct in spirit (don't build a parallel control) but MIS-STATES their status as if they exist.
**Why it matters:** Spec 36's build now carries a hidden hard dependency: the nav's per-device visibility cannot be built until P2 ships `ResponsiveTriStateControl` AND Spec 17 ships `labelCollapse`. That sequencing is invisible in Spec 36.
**Fix:** Change the wording from "uses/reuse" to "depends on P2's `ResponsiveTriStateControl` (unbuilt) + a `labelCollapse` attr on `sgs/nav-menu` (unbuilt)"; add both to a dependency list in §9/§1a with the note that the nav phase cannot start its visibility work until they land.

## 3. SHOULD-FIX

- **S1 — Popover API + `<details name>` Baseline honesty (FR-36-6).** VERDICT: honestly flagged, mild overstatement. Popover API is Baseline "newly available" (Chrome 114 / Safari 17 / Firefox 125, 2023-24) — safe by 2026 WITH the stated `.show()`+body-portal fallback (good). `<details name>` exclusive accordion (Chrome 120 / Safari 17.2 / Firefox 130, late-2024) is NOT yet Baseline "widely available" (≈16-18 months in); Spec 36 correctly marks it "⚠VERIFY" and treats it as no-JS enhancement only. Keep the flags; add the concrete degradation sentence: "browsers without `<details name>` fall back to independently-openable `<details>` — still functional, just non-exclusive." That closes the honesty gap.
- **S2 — The "CPT admin home" the mega reuses is itself unshipped.** `sgs_header` is registered with `show_in_menu => false` (`class-sgs-block-cpts.php:100`); P2 §2 says exposing it is "a small delta (flip `show_in_menu`)". So FR-36-3's "edited in the SAME CPT admin home as headers/footers" reuses a home that P2 has DESIGNED but not yet surfaced. State the dependency.
- **S3 — The "converter pack factory (P2 §9)" (FR-36-3) is unbuilt.** The header/footer clone WALKER (Spec 33 Part 2) is "NOT started" per the specs index. Emit-by-construction (FR-36-15/DP6) is correctly scoped as a schema-conformance target, but FR-36-3's phrasing "reuses the converter pack factory" implies an existing factory. Downgrade to "will share the header/footer emit path (Spec 33 Part 2, unbuilt)".
- **S4 — `showModal()` + WP admin bar.** When a logged-in operator previews, `#wpadminbar` is a fixed element the modal `<dialog>` top-layer will cover (top-layer sits above everything). FR-36-6's below-header variant already exempts `#wpadminbar` from the freeze; note that the MODAL variant will overlay it (acceptable, but call it so it isn't logged as a bug during QC).

## 4. MISSING (platform realities the spec ignores)

- **M1 — Programmatic `wp_navigation` WRITE with submenus is asserted (FR-36-15) but the WRITE side is unproven in-repo.** The repo proves READ/walk of `wp_navigation` (confirmed above), but nothing currently WRITES a `wp_navigation` post. The converter claim "write a `wp_navigation` post (link tree, create-then-`ref` ordering)" is PLAUSIBLE (it is just `wp_insert_post(['post_type'=>'wp_navigation','post_content'=>serialize_blocks($tree)])` with `core/navigation-submenu` nesting), but Spec 36 should name the concrete shape: submenus are `core/navigation-submenu` wrapping `core/navigation-link` children — `SGS_Nav_Menu_Source` already parses that nesting, so the write must mirror it. Add the block-name contract so the emit isn't guessed.
- **M2 — `core/navigation` back-compat coupling to WooCommerce.** `class-sgs-nav-menu-source.php:38-44` documents that `core/navigation` is where "WooCommerce hooks its mini-cart/customer-account". The from-scratch `sgs/nav-menu` replacing `core/navigation` inherits that consideration for WC clients — not in scope for the two current clients, but MISSING from the cutover risk (FR-36-18) for any future WC site.
- **M3 — `parse_blocks` on an operator-authored `wp_navigation` can contain `core/navigation-link` with `type`/`kind`/`id` that resolve to deleted posts.** The walker must tolerate dangling links (a link whose target post was trashed). The current source just re-emits parsed blocks; the new bespoke emitter needs an explicit "link target missing → skip or render as text" rule. Not addressed in FR-36-4/17.
- **M4 — Editor preview of a dynamic (render.php) nav.** FR-36-2 makes `sgs/nav-menu` and `sgs/nav-drawer` dynamic (render.php). The MEMORY lesson `ssr-fixes-hand-built-preview-drift` + P2 §4.4 (`<ServerSideRender>` precedent, brand-strip) both apply here and are NOT referenced in Spec 36's editor-controls section (FR-36-14). A hand-built edit.js preview of the bar/drawer WILL drift from render.php. Name `<ServerSideRender>` as the preview approach (with the known tradeoff that view.js disclosure/drawer animation won't run in the static preview).
- **M5 — `<dialog>` inside `sgs_header` CPT editor canvas.** The drawer is a `<dialog>` in the block canvas; in the editor iframe it must be reachable for editing (P2 §2.3 "Edit menu drawer" affordance). Spec 36 FR-36-6 gives the drawer its own InnerBlocks but doesn't restate the editor-reachability mechanism — carry P2 §2.3's affordance into FR-36-14 so the drawer's InnerBlocks aren't unreachable-because-closed.

---

## 5. GRADE

**B+ for platform-soundness.** The load-bearing mechanisms are real and mostly proven in live code (wp_navigation walk, `--sgs-header-height`, top-layer `<dialog>`, public Interactivity API, standard CPT registration). No fatal platform fiction. It loses the A on three "(exists)"-style overstatements that a build agent would trust and get burned by: the non-existent `data-sgs-header-state`/`visible|sticky|shrunk` surface (MF1), the un-reconciled contradiction with the live drawer's deliberate non-`showModal()` design (MF2), and citing two unbuilt P2/Spec-17 controls as reusable machinery (MF3).

---

## 6. ONE-LINE VERDICT

Platform-sound and largely proven in live code — but delete the phantom `data-sgs-header-state` surface, reconcile the `showModal()` choice with the drawer the team already shipped, and mark `ResponsiveTriStateControl`/`labelCollapse` as unbuilt dependencies before any build starts.
