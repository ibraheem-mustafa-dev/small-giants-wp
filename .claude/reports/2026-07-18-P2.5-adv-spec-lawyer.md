# Adversarial Review — THE SPEC-LAWYER — Spec 36 (SGS Navigation System) v1.1

**Reviewer stance:** every MUST clause is a courtroom exhibit. If two competent implementers, reading only this document, would write different code, the clause has failed — no matter how well-intentioned. QC council (completeness/clarity/consistency) already swept the surface; this pass hunts what survived that sweep: residual ambiguity, undefined load-bearing terms, and contradictions introduced BY the v1.1 fixes themselves.

---

## 1. STRENGTHS

- FR-36-4's hover-intent numbers are genuinely falsifiable: `default 300ms; attribute, tunable 100–500ms` gives an implementer a concrete range to build a control for, and a tester a concrete value to assert against.
- FR-36-5's "real-position render" fix states the failure mode it forecloses ("couldn't place Indus Brands at position 4") and the mechanism (derive from walk order, never stored position) — a rare case of a requirement that names both the bug and the structural guarantee that makes the bug impossible.
- §8's acceptance gate is unusually concrete for a spec at this stage: named tool calls (`hosting_clearWebsiteCacheV1`, `wp litespeed-purge all`), named viewport widths, a named regression guard (mega at position N). Most of it is independently executable by a QA agent without further interpretation.
- FR-36-10's disclosure-vs-dialog split resolves what is normally the single biggest ARIA nav bug class (menu/menubar misuse) by naming the exact primitives (`aria-expanded`, no focus trap, no `aria-haspopup`) and citing primary sources.
- FR-36-18's cutover clause requires a measured precondition ("zero live stored instances") before deletion, not a assumption — this is a rare spec that treats "nothing left on the old shape" as a checkable fact rather than a belief.

---

## 2. MUST-FIX

### M1 — "Stable ID" for `wp_navigation` items is asserted, never defined (FR-36-5, FR-36-15, §11d)
FR-36-5 requires the item→mega map to be "keyed by the `wp_navigation` item's **stable ID** (NEVER position)". `wp_navigation` items are `core/navigation-link` blocks parsed from `post_content` — WordPress core does **not** give them a persistent identifier by default. A `core/navigation-link`'s `id` attribute (when present) is the *target* post ID, not a per-item key, and is absent for custom-URL items; block `clientId` is editor-session-only and is not guaranteed stable across saves/re-parses. The spec never states the mechanism that MINTS and PERSISTS this "stable ID" (a generated `metadata.name`? a custom block attribute added at write-time? a hash of label+url?). Two implementers will build genuinely different — and differently fragile — systems. **Fix:** add an FR that names the exact attribute/mechanism used to mint + persist the stable ID (e.g. "`sgs/nav-menu` write path assigns a `metadata.name`/custom `navId` attribute to every top-level `core/navigation-link` on first save if absent; the map keys on that value"), and state what happens when an item lacks one (map-orphaned mega, or auto-repair).

### M2 — "the site's primary menu" has no defined resolution mechanism (FR-36-1)
`wp_navigation` has no native "primary menu" concept analogous to classic-menu `register_nav_menu()` locations. FR-36-1 says the block "default[s] to the site's primary menu" as if this is a given WP feature. Is "primary" (a) the most-recently-published `wp_navigation` post, (b) a slug/meta convention (`sgs-primary-menu`), (c) a value stored in `wp_options`, or (d) something the header CPT designates? Undefined. This is load-bearing for FR-36-6's "same as the bar by default" language too (see M3) — if "primary" isn't a fixed, queryable thing, "default" is unfalsifiable. **Fix:** name the storage location (option key or meta convention) that defines "primary", per the project's DB-first rule (R-31-1 in the parent CLAUDE.md forbids exactly this kind of unwritten convention).

### M3 — Self-contradiction: does the drawer's menu default *mirror* the bar's, or independently default to the same site default? (FR-36-6, §11b)
FR-36-1 states bar and drawer "may reference the **same** menu ... OR **different** menus ... **neither inherits the other**." FR-36-6 then says: "the drawer's `sgs/nav-menu` references a menu independently ... **same as the bar by default**, or a different mobile-specific menu. It does **NOT** inherit the bar's menu by context." "Same as the bar by default" and "does not inherit the bar's menu" are two different claims read side by side: reading A — both blocks independently default to the site's primary menu, so they merely *happen* to match; reading B — the drawer's default value is *sourced from* whatever the bar instance is currently set to (live or snapshot mirror), which IS a form of inheritance and directly contradicts the very next sentence. The QC council apparently accepted this phrasing at face value; it is still ambiguous on a re-read. **Fix:** replace "same as the bar by default" with "defaults independently to the site's primary menu (§FR-36-1) — which is numerically the same menu the bar also defaults to, not a reference to the bar's live selection." One sentence removes the contradiction.

### M4 — Geometry × modality cross-product is incompletely specified (FR-36-6)
The modality rule is binary and keyed only on `coverHeader`: "Full-screen / cover-header (default)" → modal; "Below-header option (`coverHeader:false`)" → non-modal. But `edge` is a 4-value enum (`full-screen | left | right | top`) independent of `coverHeader`. What is the modality and behaviour for `edge:left, coverHeader:true` (a partial-width side drawer that still sits above the header)? Is it modal per the coverHeader branch, even though it visually covers only e.g. 30% of the viewport width — leaving the header's other 70% *visible but inert* per FR-36-9/FR-36-6's selective-freeze machinery, which is only defined for the non-modal branch? The spec's own justification for modal-on-cover ("The drawer holds its own logo + close, so covering the header is correct") assumes the drawer covers the *whole* header, which a left/right edge drawer at partial width does not. This is a real fork: one implementer ships `edge:left + coverHeader:true` as a full top-layer modal with the header rendered-but-invisible-behind-scrim; another ships it as a visible partial header with only the covered strip inert. **Fix:** either restrict `coverHeader:true` to `edge:full-screen` only (simplest — declare the other 3 combinations undefined/disallowed and have the editor UI hide the combination), or add an explicit modality row per `edge` value.

### M5 — "top" position-binding is stated for the below-header case but never generalised to `edge:left/right` (FR-36-6, FR-36-9)
FR-36-9 says a below-header drawer "binds its `top`/`max-height` to `--sgs-header-height`" — but `top` positioning is meaningless for `edge:left`/`edge:right` full-height side drawers (which would bind `left`/`right`/`height`, not `top`). The spec never states the equivalent binding for those two edges. Same root cause as M4 — the geometry enum was extended in v1.1 (FR-36-8→FR-36-6 discoverability rework) but the binding/modality rules were only ever written for the `top`/`full-screen` cases.

### M6 — Default drawer template violates the drawer's own accessibility MUST (FR-36-6)
FR-36-6 states the default InnerBlocks template is `[ container, nav-menu, container ]` (two *empty* containers). The same FR's "Dialog a11y" bullet states: "**a visible × close is authored in the drawer**" as a MUST, not a should. An operator who inserts the drawer block and does nothing else (the default, out-of-the-box state) gets a drawer with no close control anywhere in its InnerBlocks tree — violating the spec's own a11y MUST at the moment of insertion. Either the default template must ship a close-button block pre-populated in one of the two containers, or FR-36-6 must state that the *block itself* renders a structural close affordance outside the InnerBlocks zone (in which case "the operator authors its own top row (logo at any size + close ×)" in the same paragraph is misleading — it implies the operator is responsible for that markup, not the block). **Fix:** state explicitly which of the two is true, and if operator-authored, seed the starter template with a close-button block by default (matches P2's own starter-pack precedent, FR-36-3).

### M7 — Priority+ overflow mode is marked "SHIP" but has zero implementation FR, control, or acceptance criteria (FR-36-8, §8 Indus)
FR-36-8 records priority+/"More" overflow as "recorded — not dropped," then in the same sentence instructs it to "**SHIP** as an opt-in mode (high value for the 7-item Indus bar)" — imperative, not deferred like bottom-tab-bar/⌘K in the same bullet. §8's Indus acceptance explicitly leans on it: "priority+ keeps 7 items discoverable mid-width" is part of the live-QC gate's expected behaviour. Yet: no FR defines the trigger geometry (how is "mid-width" measured — remaining bar width vs. summed item widths?), no FR defines the "More ▾" disclosure's own a11y contract (is it itself a disclosure per FR-36-10, presumably yes, but never stated), no attribute is named (`priority` per-item attribute is mentioned once, no default/int semantics), and FR-36-14's control-completeness checklist never lists a priority-order control. A requirement that is load-bearing for the acceptance gate cannot ride on a single parenthetical. **Fix:** either (a) write a real FR-36-8a with trigger mechanism + attribute + a11y contract + editor control, or (b) demote it out of the §8 Indus acceptance gate and mark it explicitly deferred like the other two optional modes, resolving the "SHIP" vs "recorded, not dropped" tension honestly.

### M8 — Two independent breakpoint systems can visually desync in their overlap band (FR-36-8 vs FR-36-14)
FR-36-8 introduces a **collapse-point attribute `N`** (default 768, operator-tunable, explicitly "distinct from the locked 768/1024 device-tier system"). FR-36-14 requires `ResponsiveControl for 768/1024 style tiers` on the same block. If an operator sets `N = 900` (a legitimate, spec-sanctioned value per FR-36-8), the bar still visually renders as a horizontal bar (uncollapsed, per FR-36-8's own default) from 768px up to 900px, while the block's *device-tier* styling attributes have already switched to "tablet" styling at 768px per FR-36-14/Spec 35 D2. Nothing in the spec states which set of styles govern the 768–900 band, or that the two systems are even allowed to be visually independent variables (bar-mode based on N, box/spacing/typography based on the fixed tier). This is exactly the kind of two-breakpoint interaction the parent CLAUDE.md's own "device-tier vs visual breakpoint" rule warns is easy to conflate — but here it's not conflated, it's left silently uncomposed. **Fix:** state explicitly that the collapse-point governs *layout mode only* (bar vs burger) and is visually independent of the tablet/mobile *style* tier, and that the acceptance gate (§8) must include a sweep at a non-default `N` to prove the two systems don't visually collide (currently §8 only sweeps 375/768/1440, i.e. only ever tests `N` at its own default).

## 3. SHOULD-FIX

- **"content-sized overlay" (FR-36-4) has no defined bound.** "Never a full-screen takeover" is falsifiable in the negative, but there's no positive definition (max-height as % viewport? intrinsic-content sizing with an overflow scroll fallback for a very tall mega panel?). A photo-grid mega with many rows could legitimately grow taller than the viewport — what happens then is unstated (internal scroll? clip? auto-shrink?). Add a bound.
- **"~500ms" close-grace (FR-36-4)** uses `~` where the sibling trigger-delay clause used a precise tunable range (100–500ms). Either give it the same tunable-attribute treatment or state explicitly it is a fixed non-configurable constant — the tilde currently makes it untestable to the millisecond in a way the rest of the FR isn't.
- **"header row" is ambiguous under Indus's two-row header (FR-36-6 selective-freeze).** The below-header non-modal freeze exempts "{header row + drawer + scrim + `#wpadminbar`}" — Indus's header (§8) has two rows; does "header row" mean the row containing the nav only, or all header rows? A two-row header with only one row exempted from `inert` would leave the other row simultaneously visible-but-frozen, which is a real, distinct UX from "both rows interactive." Name it precisely.
- **"schema-friendly markup" and "must not emit markup that blocks `seo-schema`" (FR-36-17) are both unfalsifiable as written.** No shape, attribute, or structural requirement is given for what "friendly" or "blocking" mean concretely (e.g., must every top-level `<nav>` carry a stable `id`/`data-*` hook the `seo-schema` skill can select against? Must item text avoid inline elements that would break `itemprop` nesting?). Since JSON-LD emission is explicitly deferred to another skill, this FR should at minimum name the DOM contract (selectors/attributes) that skill depends on, or point to where that contract is defined — right now nothing defines it anywhere in the referenced companions either.
- **"SGS Interactivity-style store" (FR-36-7) is ambiguous between "uses WP core's `@wordpress/interactivity` API" and "a bespoke hand-rolled store that merely resembles the pattern."** Given this utility is explicitly designed to be framework-reusable (cart flyout, search overlay), the choice has real blast radius and should be a named decision, not inferable from adjective choice ("style").
- **The default collapse breakpoint of 768 has no cited basis** ("every top exemplar collapses the same at 768 as at 375" — no source, no comparison table), unlike the hover-intent numbers which cite "research S1." Minor, but the spec elsewhere holds itself to sourced numbers; this one slipped through.
- **FR-36-15's "per-client naming/scoping convention"** (explicitly flagged as a MISSING term by the review brief) is asserted as existing ("carry a per-client naming/scoping convention") without stating what it is — e.g. `{client-slug}--{starter-slug}--{n}` for `sgs_mega_menu` post_name. Should point to Spec 33/31's existing per-client naming rule if one exists, or define it here.

## 4. MISSING (stated requirement, no acceptance criterion)

- **Desktop a11y is never gated in §8.** The acceptance gate runs `axe = 0 on the OPEN drawer` only. FR-36-10/FR-36-11's disclosure contract (dropdown AND mega — `aria-expanded`, no focus trap, Escape returns focus, `aria-current`) governs the *desktop* bar and mega, which are never axe-swept or focus-tested in §8 at all. The mega/dropdown open-state is arguably the higher-risk surface (custom disclosure + hover-intent + safe-triangle JS) and has zero listed test.
- **`elementFromPoint` sweep "≥ baseline"** — "baseline" is never defined anywhere in the document (not a prior measurement, not a formula, not a fixed count). As written, no implementer or QA agent can determine pass/fail for this clause; it reads as a placeholder for a number that was never filled in.
- **No acceptance test exists for `edge:left/right/top` geometry**, despite FR-36-6 defining all four as first-class attribute values. §8's geometry check only covers `below-header` (`top` = header-bottom ±1px) and `cover mode` ("full-viewport", itself under-specified — no left/width/height assertion given). Three of the four `edge` values are entirely untested at the acceptance gate.
- **No acceptance test for the Mama's "featured item" visual treatment** (FR-36-4's per-item highlight flag is required for the Mama's reproduction target in §8, but §8 never states a computed-style or DOM check that would fail if the flag rendered as a no-op).
- **No acceptance test for FR-36-17's "no layout shift"/no-CLS claim** — no CLS score, no threshold, no named measurement tool (Lighthouse? Chrome DevTools trace?), despite the FR making a specific, checkable-in-principle performance claim.
- **No defined arbitration process for the Indus "1-vs-5 mega panels" deferral.** FR-36-16 explicitly punts the exact count/layout choice to "VERIFIED against the Indus mockup at build" — reasonable to defer the *number*, but the spec never says who resolves it if the mockup itself is ambiguous (an implementer's read vs Bean's read), nor does it commit to writing the resolved number back into the spec before the acceptance gate can be run. As it stands, §8's Indus clause cannot be executed as a pass/fail gate today — it can only be executed after a separate, undocumented resolution step that has no owner or record-back requirement.
- **`sgs_mega_menu`'s own container KIND is never classified** against the D294 wrapper-vs-block-private selector cited project-wide (content-KIND → block-private; section/layout-KIND → keep `SGS_Container_Wrapper`). FR-36-13 states the *nav bar/drawer* keep the wrapper and inner content goes block-private "where it uses only box+width," but never assigns a KIND to the mega panel's own root wrapper — and a mega panel ("rich full mega panel," multi-column, background, grid layouts) reads far more like a section/layout composite than a content composite. Left to implementer judgement, this is exactly the ambiguity D294 was written to eliminate elsewhere in the framework.

---

## 5. GRADE

**C+ for testability/precision.**

Reasoning: the FRs that got explicit QC-council attention (position-based mega rendering, hover-intent numerics, disclosure-vs-dialog split) are genuinely falsifiable and well-sourced — better than most specs at this stage. But the v1.1 additions that were rushed through to answer Bean's late corrections (geometry/modality, discoverability, cutover) introduced at least one internal contradiction (M3), one incompletely-generalised cross-product (M4/M5), one requirement that contradicts its own a11y MUST at default state (M6), and one load-bearing mechanism that is simply never defined despite being cited twice as settled (M1, the stable-ID key). The acceptance gate (§8) — the part of the document whose entire job is to be checkable — contains an undefined term ("baseline") and covers only 2 of 4 geometry values and none of the desktop a11y surface. A spec at "pending Bean sign-off before any build" gate should not still contain a foundational data-model question (how is a nav item identified?) with no answer.

## 6. Verdict

Nice bar. Shame about the drawer that ships with no way out, the menu that both does and doesn't inherit itself, and the acceptance gate whose "baseline" is a number nobody wrote down.
