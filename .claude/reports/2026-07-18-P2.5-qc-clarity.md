## HIGH — would cause a wrong build

### H1 — FR-36-16 (Indus acceptance): the mega counts are numerically impossible
**Type:** INCONSISTENT / AMBIGUOUS
**Exact text:** "**Indus (B2B):** 7-item bar; 3 simple dropdowns; **1 mega "Brands" at position 4** (rendered at position 4, not last); **5 rich mega panels** (the 5 layouts as `sgs_mega_menu` posts)".
**Problem:** On a 7-item bar, "3 simple dropdowns" + "5 rich mega panels" = 8 disclosure items on 7 items — impossible. It also self-contradicts: "**1 mega**" vs "**5 rich mega panels**". A builder cannot tell whether the Indus bar has one mega (Brands) or five, nor how the five layout templates relate to the bar. This is the pass/fail acceptance gate (§8) — it must be buildable to.
**Fix:** State the two things separately and unambiguously, e.g. "Bar = 7 items: 3 plain links, 3 simple dropdowns, **1 mega panel (Brands) at position 4**. Separately, all **5 mega layouts are authored as `sgs_mega_menu` starter-template posts** to prove the layout set — only Brands is attached to this bar." If more than one mega is genuinely on the bar, give the exact item→layout mapping so 3 dropdowns + N megas ≤ 7.

### H2 — FR-36-2 contradicts FR-36-5 + §10 sub-decision d on where the mega reference lives
**Type:** INCONSISTENT
**Exact text (FR-36-2):** "…`sgs_mega_menu` … **Referenced from a menu item by ID**; rendered at the item's real bar position…"
**Contradicting text (FR-36-5):** "the item→panel reference is stored **on `sgs/nav-menu`** (a map keyed by `wp_navigation` item ID/position → `sgs_mega_menu` post ID), **NOT on the `wp_navigation` item**." (§10 d repeats: "not on the menu item".)
**Problem:** FR-36-2 reads as if the reference lives on the menu item ("Referenced from a menu item by ID"); FR-36-5 and sub-decision d say the reference is an item→mega map stored on the `sgs/nav-menu` block and explicitly NOT on the menu item. This is architecture-shaping (the data-purity rationale in FR-36-5 depends on it) — two readers build two different data models.
**Fix:** Reword FR-36-2 to: "Referenced by `sgs/nav-menu`'s item→mega map (keyed by the menu item), never stored on the `wp_navigation` item (FR-36-5); rendered at the item's real bar position."

### H3 — item→mega map "keyed by ID/position" is undefined; load-bearing for the fatal-bug fix
**Type:** AMBIGUOUS
**Exact text (FR-36-5, repeated §10 d and FR-36-15):** "a map keyed by `wp_navigation` item **ID/position** → `sgs_mega_menu` post ID".
**Problem:** "ID/position" presents two different keys as interchangeable. Item **ID** is stable across reordering; item **position** changes when items are reordered. The "real-position render (the fatal-bug fix)" in FR-36-5 says the panel renders "at the referenced item's **actual bar position** (by walk order)" — i.e. position is *derived* from walk order, which means the association key must be the stable ID, not position. As written, a builder keying by position will silently mis-associate megas the moment the operator reorders the menu (re-introducing a placement bug close to the one this FR exists to kill).
**Fix:** Choose one: "keyed by the `wp_navigation` item's **stable ID**; the render position is derived from walk order (never stored)." Drop "/position" from the key description everywhere (FR-36-5, §10 d, FR-36-15).

### H4 — FR-36-2 vs FR-36-8: does a collapsed `sgs/nav-menu` render the list, or a burger?
**Type:** INCONSISTENT
**Exact text (FR-36-2):** "Renders a `wp_navigation` ref as a horizontal **bar** … (desktop) or a vertical **accordion/drill-down list** (collapsed/in-drawer)."
**Contradicting text (FR-36-8):** "below its collapse breakpoint the menu renders a labelled, unboxed **burger** … that opens the associated `sgs/nav-drawer`."
**Problem:** FR-36-2 lumps "collapsed" and "in-drawer" together as both rendering the vertical list. But FR-36-8 says a nav-menu on a header row, when collapsed, renders a **burger** (not the list); the list is rendered by the *separate* nav-menu instance inside the drawer (FR-36-6 default template `[container, nav-menu, container]`). A builder reading the roster table alone (H4's most-cited quick-ref) builds an inline accordion on collapse instead of a burger-opens-drawer flow — a materially different architecture.
**Fix:** Split the two states in FR-36-2: "…or, **when placed inside the drawer**, a vertical accordion/drill-down list. On a header row below its collapse breakpoint it renders a **burger** that opens the associated `sgs/nav-drawer` (FR-36-8) — it does not render the list inline."

## MED — would slow or mislead the build

### M5 — §1 non-goals: broken cross-reference + a mode recorded nowhere
**Type:** INCONSISTENT
**Exact text (§1):** "bottom-tab-bar and command-palette navigation (**FR-36-14** records them as optional/deferred modes)".
**Problem:** Two errors. (1) The optional/deferred modes are recorded in **FR-36-8** (and §10 c), not FR-36-14 (which is control-completeness / Spec 35 Part L). The reference points a reader to the wrong FR. (2) **Command-palette** navigation is named here as recorded, but it appears in NO FR (FR-36-8 records only bottom-tab + priority+; §10 c likewise). It is the "third pattern" FR-36-8 warns against inventing — yet it is left dangling as an unrecorded non-goal.
**Fix:** Change the ref to FR-36-8. Either add a one-line command-palette entry to FR-36-8's optional-modes list (recorded-not-dropped, like bottom-tab) or remove the "records them" claim for command-palette.

### M6 — FR-36-5: wrong FR cross-reference + mis-attributed no-core-blocks rule
**Type:** INCONSISTENT
**Exact text:** "Panel content = SGS blocks only (**no core blocks → Spec 32 holds**); styled via scoped `<style>` (**FR-36-12**)."
**Problem:** (1) "(FR-36-12)" is wrong — FR-36-12 is *operator a11y feedback is informational only*. The scoped-`<style>`/no-inline requirement is **FR-36-13**. (2) The no-core-blocks ban is not what Spec 32 governs (Spec 32 = component styling / no-inline). FR-36-15 correctly lists "no inline styles, **no banned core blocks**" as two separate constraints; here they are conflated under Spec 32.
**Fix:** "…styled via scoped `<style>` (**FR-36-13**)"; and "no banned core blocks (per the core-block ban) + no inline styling (Spec 32)."

### M7 — FR-36-8: is the collapse breakpoint N free, or locked to 768/1024?
**Type:** AMBIGUOUS
**Exact text:** "**One collapsed breakpoint** … Device tiers use the locked 768/1024 standard (Spec 35 D2). **Collapse-below-N is an attribute.**"
**Problem:** The FR invokes Spec 35 D2's "locked 768/1024 … never a bespoke third value" in the same breath as "Collapse-below-N is an attribute" (implying a free operator-chosen N). Spec 35 D2 governs responsive *attribute tiers*; a nav collapse point can legitimately be a design/visual breakpoint. But as written a builder cannot tell whether N is constrained to {768, 1024} or free.
**Fix:** State it explicitly: "The collapse point N is a *visual* breakpoint (operator attribute, default 768), distinct from the locked 768/1024 device-tier system used for responsive style attributes (Spec 35 D2)." — or, if N must be one of the two tiers, say "N ∈ {768, 1024}."

### M8 — FR-36-4: the simple-dropdown-vs-mega selection is guidance, not a mechanism
**Type:** AMBIGUOUS
**Exact text:** "Two disclosure kinds, chosen per item by a **documented rule**: **Simple dropdown** — … Use for short flat lists … **Mega panel** — … Use when items/imagery/sub-levels justify it."
**Problem:** "chosen per item by a documented rule" then gives only editorial advice ("use for short lists"), not a decidable mechanism. Is the kind an operator per-item attribute, or auto-derived (e.g. an item renders a mega iff it has a `sgs_mega_menu` mapping in the item→mega map, else a simple dropdown)? A builder can't implement "a documented rule" from prose.
**Fix:** Name the mechanism: "An item renders a **mega** iff it has an entry in `sgs/nav-menu`'s item→mega map (FR-36-5); otherwise its `wp_navigation` submenu renders as a **simple dropdown**. The list-length guidance is editorial advice to the operator, not an automatic switch."

### M9 — FR-36-5: mobile mega rendering "either … or" has no selection rule
**Type:** AMBIGUOUS
**Exact text:** "**Mobile:** the same panel content renders **inside the drawer** — **either** expanded inline under its item (like a dropdown) **or** as a fullscreen sub-panel the tapped item swaps to."
**Problem:** Two rendering modes offered with no rule for which one applies. FR-36-6 defines a drawer attribute (accordion default / drill-down option) — the mobile mega mode almost certainly should follow it, but that link is never stated.
**Fix:** Tie it to the drawer's submenu attribute: "The mobile mega follows the drawer's submenu model (FR-36-6): **inline-expanded** when the drawer is in accordion mode, **fullscreen sub-panel** when in drill-down mode."

### M10 — FR-36-6: subcategory-volume guidance uses undefined terms that don't map to the drawer's two models
**Type:** UNDEFINED-TERM / AMBIGUOUS
**Exact text:** "Subcategory-volume guidance (**<6 accordion / 6–15 section-menu / >15 landing page**) informs the default."
**Problem:** The drawer offers exactly two in-drawer submenu models (accordion default / drill-down option). The guidance introduces a three-bucket taxonomy — "section-menu" and "landing page" — that are undefined here and do not map onto accordion/drill-down. A builder cannot act on "6–15 → section-menu" (what is a section-menu in this system?), and it conflicts with "accordion … is the **default**".
**Fix:** Either drop the three-bucket guidance or restate it in the drawer's own vocabulary: "<6 items → accordion (default); ≥6 → consider drill-down." If "section-menu"/"landing page" are real deliverables, define them or cross-reference where they are defined.

### M11 — FR-36-9: "READ the header's state and ADAPT" specifies no mechanism
**Type:** AMBIGUOUS
**Exact text:** "`sgs/nav-menu` and `sgs/nav-drawer` **READ the header's state** (visible/sticky/transparent/shrink — Spec 17 FR-S9-9) and **ADAPT** to it … the open drawer's top edge **attaches to the bottom of the header**".
**Problem:** The adaptation is called "a first-class, tested requirement" but the channel by which the nav reads header state is unspecified (a root/body class? a CSS custom property the header exposes? a shared JS store?). The drawer's top-edge-attach geometry can't be built without knowing how the drawer learns the header's current height/visibility. "READ … and ADAPT" is a vague verb pair with no defined HOW where the HOW is load-bearing.
**Fix:** Name the contract, e.g. "The header exposes its state + measured height as CSS custom properties / a root class (`data-header-state`, `--sgs-header-height`) that the nav blocks read; the drawer's `top`/`max-height` bind to these. The header never reads the nav."

### M12 — FR-36-8: the burger→drawer "association" targeting is under-specified
**Type:** AMBIGUOUS
**Exact text:** "the menu renders a … burger … that opens the **associated** `sgs/nav-drawer`. (The drawer stays a separate authorable container; **the menu targets it**.)"
**Problem:** Since the drawer is a separate block that may appear anywhere (even multiple drawers), "the associated drawer" / "the menu targets it" needs a concrete mechanism (a `drawerRef`/`aria-controls` attribute on nav-menu holding the drawer's anchor/ID; and the default resolution when unset). Parallel to the item→mega map, this is a reference the builder must implement but isn't defined.
**Fix:** "`sgs/nav-menu` carries a `drawerRef` attribute (the target `sgs/nav-drawer`'s block anchor/ID), wired to `aria-controls`; if unset it targets the nearest drawer in the same header. Exactly one drawer per burger."

### M13 — §8 Mama's requires capabilities no §3 behaviour FR defines
**Type:** INCONSISTENT (acceptance vs FRs)
**Exact text:** "**Mama's (restaurant):** flat 5-item bar + a **featured** item + a **cart badge**; mobile → drawer (accordion) + CTA."
**Problem:** No behaviour FR in §3 defines a "featured item" treatment or a "cart badge" on `sgs/nav-menu`. A builder implementing §3 faithfully would not produce either, then fail the §8 gate. Acceptance criteria must be traceable to a requirement.
**Fix:** Add the two capabilities to a behaviour FR (e.g. FR-36-4: a per-item "featured" style flag; a cart-badge affordance — WooCommerce count — as an optional nav-menu element), or explicitly note in §8 that these are generic block content the operator places, not nav-menu features.

### M14 — the new `sgs/nav-menu` reuses the retired `sgs/nav-menu (old)` slug; migration/collision unaddressed
**Type:** AMBIGUOUS
**Exact text (§1):** "the previous nav trio — `sgs/adaptive-nav`, **`sgs/nav-menu` (old)**, `sgs/mega-menu` — … are REFERENCE-ONLY and replaced". **Exact text (FR-36-2):** roster block "**`sgs/nav-menu`**".
**Problem:** The new roster registers a block under the same slug (`sgs/nav-menu`) as an explicitly-retired block. The spec never says whether this is a deliberate same-slug rebuild and how existing `sgs/nav-menu` (old) posts are handled (re-cloned vs migrated), nor whether the old registration must be removed first. For a cold implementer registering `block.json`, same-slug reuse is a real collision/migration concern.
**Fix:** Add one line: "The new `sgs/nav-menu` is a from-scratch rebuild registered under the same slug; old-shape posts are re-cloned, not deprecation-migrated (plugin-wide no-deprecations policy). The retired registration is removed."

### M15 — FR-36-6: the drawer's whole survival mechanism is a load-bearing ⚠VERIFY with no fallback
**Type:** AMBIGUOUS (unresolved load-bearing verify)
**Exact text:** "renders in the browser top layer (`<dialog showModal>` / Popover API) so it survives a `transform`-ed … header ancestor **by construction — no JS `<body>` portal**. (⚠ build-time VERIFY: Popover/`<dialog>` Baseline + WP-render-context usability.)"
**Problem:** The transform-ancestor-survival guarantee — a named first-class requirement (FR-36-9) — rests entirely on top-layer `<dialog>`/Popover, flagged unresolved. The spec simultaneously *forbids* the usual fallback ("no JS `<body>` portal"). If the VERIFY fails at build time, there is no stated plan B, so the requirement becomes unmeetable. This is the spec's single largest technical risk left open.
**Fix:** Keep the VERIFY but add the contingency: "If top-layer `<dialog>`/Popover proves unusable in the WP render context, the fallback is [X]" — and reconcile with the "no `<body>` portal" prohibition (state whether the portal ban is absolute or lifts if the VERIFY fails).

## LOW — clarity polish

### L16 — FR-36-13: "nav-bar container" is an undefined term
**Type:** UNDEFINED-TERM
**Exact text:** "Section/layout-KIND composites (**nav-bar container**, drawer) keep the shared `SGS_Container_Wrapper`".
**Problem:** "nav-bar container" is not a roster name (FR-36-2 names `sgs/nav-menu`). A reader infers it = nav-menu in bar mode, but it should be explicit.
**Fix:** "…composites (`sgs/nav-menu`, `sgs/nav-drawer`) keep the shared wrapper."

### L17 — FR-36-4: "default ~300–500 ms" is a range, not a default
**Type:** AMBIGUOUS
**Exact text:** "a tuned hover-intent delay (**default ~300–500 ms**)".
**Problem:** A "default" should be a single value; a "~"-range leaves the initial attribute value undefined.
**Fix:** Give one default and note it's tunable: "default 300 ms (attribute; tune 100–500 ms)."

### L18 — FR-36-6: native `<dialog showModal>` plus `role="dialog"`+`aria-modal="true"` is redundant/discouraged
**Type:** (a11y precision) AMBIGUOUS
**Exact text:** "**Top-layer dialog** … (`<dialog showModal>` / Popover) … `role="dialog"` + `aria-modal="true"` + label".
**Problem:** A native `<dialog>` opened with `showModal()` already exposes dialog semantics + implicit modality; adding `role="dialog"`/`aria-modal="true"` is redundant and (for `aria-modal` on a native dialog) generally discouraged. Harmless if a non-`<dialog>` element is chosen, but ambiguous about which element is authoritative.
**Fix:** "If the native `<dialog showModal>` element is used, rely on its built-in dialog semantics/modality (do not add `role`/`aria-modal`); apply `role="dialog"`+`aria-modal="true"` only to a non-`<dialog>`/Popover implementation."

### L19 — FR-36-2: roster table titled "Block + CPT roster" includes a non-block/non-CPT row
**Type:** UNDEFINED-TERM (minor)
**Exact text:** table heading "Block + CPT roster" with a fourth row "Shared nav plumbing … (NOT a block)".
**Problem:** Minor — the plumbing utility isn't a block or CPT, so the table title slightly over-claims. Not build-affecting.
**Fix:** Retitle "Block + CPT + plumbing roster" (or note the fourth row is supporting infrastructure).
