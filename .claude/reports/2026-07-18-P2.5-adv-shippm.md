# P2.5 Adversarial Council — THE SHIP-PM verdict on Spec 36 (SGS Navigation System v1.1)

**Reviewer lens:** ruthless delivery lead. Attack surface = SCOPE + SEQUENCING + STALL-RISK only (clarity already covered by the QC council). **Date:** 2026-07-18.

**One-sentence framing:** This is an excellent *requirements* document and a non-existent *plan*. It describes, in full and well-cited detail, the finished state of a ~multi-week system, and it never once says what to build first, how to prove the architecture cheaply, or where the line between "shipped" and "gold-plate" sits. For a solo AI-driven builder who is easily stalled by scope, that omission is the single biggest risk in the document.

---

## 1. STRENGTHS

- **Decisions are actually resolved, not deferred.** §11 closes the five load-bearing sub-decisions (burger home, drawer content, priority+/tab/⌘K, item→mega storage, drawer modality) with a rationale + FR pointer each. That kills the usual "we'll decide during build" stall.
- **The fatal historical bug is named and structurally targeted.** FR-36-5's "mega renders at real bar position N, derived from walk order, never a stored position" directly kills the old "mega can only render last" defect that blocked Indus "Brands" at position 4. Naming the exact past failure is the right instinct.
- **The hardest architectural constraint (a11y) is threaded through, not bolted on.** FR-36-10's disclosure-vs-dialog split genuinely *shapes* the block roster (FR-36-2 explicitly forces the split), rather than being a late audit. Good — that's the one thing you can't retrofit.
- **Non-goals are named, not silently dropped** (§1: bottom-tab, ⌘K, footer menus → core WP, header cart/search → Spec 17). This is real scope hygiene and prevents a later phase re-inventing them.
- **A concrete live-QC gate exists** (FR-36-16 / §8): cache-clear-first, axe-0-on-open-drawer, elementFromPoint sweep, real-scrollbar, crawl assertion, Bean's eye. When it eventually runs it is a strong gate. The problem is *when* it runs (see §4), not *what* it checks.

---

## 2. MUST-FIX (scope / stall / sequencing — the killers)

### MF-1 — No vertical slice, no MVP boundary, no build order. The document cannot be executed without one. *(whole spec; §8 FR-36-16; §11)*
**Problem:** The spec defines the complete end-state and stops. There is no "build this first," no phase structure, no demoable checkpoint. §11 resolves *decisions* but imposes zero *sequence*. A solo builder handed this will try to hold four new parts (block + CPT + drawer + shared store) + safe-triangle + priority+ + dual-modality + converter-emit + a 15-point terminal QC gate in their head at once — the exact overwhelm profile that stalls this owner.
**Concrete fix:** Add a `§ Phasing (MVP → prove → extend)` section that pins the MVP as the smallest slice that reproduces **one whole client live** and proves the architecture:
> **MVP (Phase 1) = Mama's, end-to-end.** `sgs/nav-menu` (flat bar + burger, render.php tree-walk of `wp_navigation`, no-inline FR-36-13, crawlable `<a href>` FR-36-17) + `sgs/nav-drawer` (full-screen **modal** `<dialog showModal>` only, accordion only) + the shared plumbing utility (FR-36-7, open/close/focus/inert/Escape) + converter-emit of *those two blocks only* (FR-36-15). Gate = FR-36-16 Mama's row + the transform-survival + a11y + crawl checks. **Explicitly NOT in Phase 1:** the CPT, any mega, safe-triangle, priority+, dropdown/mega disclosure mechanics, below-header non-modal, drill-down.
This ships something demoable in the first slice and reproduces a real client. Everything else is Phase 2+.

### MF-2 — The motivating feature (Indus rich mega at position 4) is gated behind the entire tedious middle of the spec. Classic stall trap. *(FR-36-3, FR-36-5, §8)*
**Problem:** The rich Indus mega — the "meet-and-exceed competitors" showcase, the thing worth building — requires ALL of: the `sgs_mega_menu` CPT, reuse of P2's CPT admin home + starter-picker + pack-factory (FR-36-3), five git-versioned starter templates, the stable-ID item→mega map, the real-position walk render, AND the mobile-in-drawer mega render (FR-36-5). That is the most plumbing-heavy, least-visible, most-copyable work in the document, and the payoff sits on the far side of all of it. Meanwhile FR-36-16 bundles **both** client menus into **one** acceptance gate — so you cannot close Mama's without having built the entire Indus mega stack.
**Concrete fix:** (a) Split FR-36-16 into **gate-1 (Mama's / MVP)** and **gate-2 (Indus / mega)**; a build can pass and ship gate-1 with zero mega code. (b) Move the whole mega CPT stack to **Phase 2**, gated behind a *passing gate-1*. (c) Note the cross-spec dependency explicitly: FR-36-3 *reuses* P2's CPT home + pack-factory, which per the companions is a **LOCKED design, not necessarily built** — Phase 1 (nav-menu + drawer) has no P2 dependency; the mega phase is blocked on P2 being real. Sequencing Phase 1 first also de-risks that dependency.

### MF-3 — Pre-register a "prove-it-before-you-build-the-rest" gate. Right now §8 is one terminal 15-point gate, not a staged one. *(§8 FR-36-16; the task's cited P2/§10 pattern)*
**Problem:** The document's own lineage (P2 §10) uses a pre-registered prove-first gate. Spec 36 has the opposite: a single monolithic acceptance gate at the very end that only fires once *everything* is built. That is precisely the structure that lets scope balloon undetected until the final QC pass.
**Concrete fix:** Promote gate-1 (MF-2) into a **pre-registered Phase-1 exit gate** with its predicted-outcome/baseline/command bindings written *before* Phase 1 starts: render.php emits a crawlable flat bar + burger; `<dialog showModal>` drawer survives a `transform`ed header ancestor (top-layer, by construction); axe-0 on open drawer; ESC + focus-return + Tab-containment; elementFromPoint ≥ baseline; converter round-trips the two blocks. **Phase 2 does not begin until gate-1 is green on the canary + Bean's eye.** This is the single structural change that converts the doc from stall-prone to ship-safe.

### MF-4 — Safe-triangle / prediction-cone hover JS is Phase-2 gold-plate — and Mama's proves it. *(FR-36-4)*
**Problem:** FR-36-4's hover-intent delay, safe-triangle prediction-cone, and close-grace timing are a genuine time-sink and a well-known JS rabbit-hole. Crucially, **Mama's is a flat 5-item bar with no submenus at all** — so the *entire* desktop disclosure mechanism (hover-intent, safe-triangle, the dropdown-vs-mega switch of FR-36-4) is irrelevant to the MVP. It only earns its keep once dropdowns/megas exist (Phase 2).
**Concrete fix:** Move all of FR-36-4's interaction mechanics to Phase 2 alongside the dropdown/mega work. Ship Phase 1 with real-link top-level items only. This makes the MVP dramatically smaller than the spec's flat prose implies — lean into it.

### MF-5 — The below-header **non-modal** drawer mode is the most bug-prone piece in the spec, needs an unbuilt header feature, and neither client uses it. Defer it. *(FR-36-6, FR-36-9)*
**Problem:** The `coverHeader:false` non-modal mode requires: selective `inert` of everything-except-{header row + drawer + scrim + wpadminbar}, the **Popover API** (spec itself flags ⚠ maturity), `--sgs-header-height` binding, and — per FR-36-9 — a **"header hidden on scroll" state that is NOT YET BUILT** (P2 §15.6 / future Spec 17). A requirement whose forward dependency literally does not exist cannot be completed now; carrying it as in-scope is how a phase silently blocks. Both clients default to **full-screen cover-header modal** (FR-36-6 default; FR-36-16 makes no non-modal demand).
**Concrete fix:** Cut the non-modal below-header mode to **Phase 3 / on-demand**, explicitly conditioned on the hide-on-scroll header feature landing. Phase 1–2 ship `<dialog showModal>` full-screen only. Record it (don't delete it) — same treatment §1 already gives bottom-tab/⌘K.

## 3. SHOULD-FIX

- **SF-1 — "Designed framework-reusable" (FR-36-7) is speculative generality.** Building the shared store to serve a *future* header search-overlay / cart flyout that §1 + FR-36-7 both say are **out of scope** is YAGNI. Build it nav-shaped, prove it against the three real nav call-sites, generalise when the second consumer actually exists. Keep "don't paint into a corner"; drop "designed reusable" as a Phase-1 constraint.
- **SF-2 — Five starter mega templates is gold-plate against an unknown need.** FR-36-16 admits "1-vs-5 Indus panels is unresolved, verified at build." Building 5 git-versioned starter patterns + pack-factory generation (FR-36-3/FR-36-5) when the live requirement may be **one** is effort spent on spec, not ship. Build the 1–2 layouts the client mockup actually uses; add the rest on demand.
- **SF-3 — Drill-down drawer submenu mode is deferrable.** Accordion is the default and the guidance says "<6 items → accordion." Both clients ship on accordion (Indus's 7 is "*consider* drill-down," not required). Ship accordion only; defer drill-down to when a client tree genuinely needs it (FR-36-6).
- **SF-4 — priority+/"More" overflow (FR-36-8) is a whole layout engine; don't let "opt-in" hide that it's still unscoped build work.** Measuring item widths + folding into a More disclosure is real work neither client's MVP needs (Indus's 7 items collapse to the drawer fine). Tag it Phase 2+ explicitly, not "opt-in mode" floating with no phase.
- **SF-5 — Don't make an unverified platform feature a Phase-1 acceptance criterion.** The no-JS `<details name>` exclusive-accordion baseline (FR-36-6) carries the spec's own ⚠VERIFY support flag. Fine as progressive enhancement; must not be a *gate* condition until support is confirmed.

---

## 4. MISSING

- **No vertical slice.** (The headline gap — MF-1.) There is no "thinnest end-to-end path that touches render.php + a11y + drawer + converter + live gate and proves the architecture." The Mama's flat-bar + modal-drawer slice is sitting right there in FR-36-16 and the spec never names it as the slice.
- **No MVP boundary.** Nothing in the document distinguishes "ship" from "eventually." §11 resolves decisions; it does not bound scope.
- **No build order / phase structure.** Zero sequencing. The derived-from plans exist but the spec should hand `/phase-planner` explicit phase cuts, not a flat FR list.
- **No pre-registered prove-first gate.** §8 is terminal and monolithic (MF-3). The P2/§10 staged-gate pattern the lineage uses is absent.
- **No demoable-first checkpoint** — nothing the builder can put in front of Bean after slice 1 to earn a dopamine hit and confirm direction before the tedious middle.
- **No effort/phase estimate + no cross-spec dependency ordering** (the FR-36-3 → P2-built dependency, MF-2c, is unsequenced).

---

## Honest build-order (what I'd hand the builder)

1. **Phase 1 — Mama's, end-to-end (the proof slice).** `sgs/nav-menu` flat bar + burger + `sgs/nav-drawer` full-screen modal accordion + shared utility + converter-emit of both + Phase-1 pre-registered gate. No CPT, no mega, no safe-triangle, no priority+, no non-modal. **Demoable, ships one whole client.**
2. **Gate-1 (pre-registered) must go green on the canary + Bean's eye before Phase 2 opens.**
3. **Phase 2 — Indus mega (the motivating feature, now unblocked).** `sgs_mega_menu` CPT (reusing *built* P2 machinery) + item→mega stable-ID map + real-position render + desktop disclosure mechanics incl. safe-triangle + the 1–2 mega layouts Indus actually needs + mobile-in-drawer mega. Gate-2 = FR-36-16 Indus row.
4. **Phase 3 / on-demand — the deferred set:** below-header non-modal drawer (waits on hide-on-scroll header), priority+/More, drill-down, extra starter templates, framework-generalising the store. Each pulled forward only when a real need lands.

---

## 5. GRADE — shippability: **D+**

As a *requirements* artefact this is a B+/A- — thorough, cited, a11y-grounded, decisions closed. As a *shippability* artefact it is a **D+**: it is a complete description of a multi-week system with no MVP, no build order, and no prove-first gate, and it buries the one motivating feature behind the entire tedious plumbing middle. It will stall this builder. The gap is **entirely fixable and cheap** — MF-1 through MF-3 (add a phasing section + split the gate + pre-register gate-1) convert it to a B+ shippable plan without changing a single functional requirement. Nothing here is a design flaw; it's a sequencing vacuum.

## 6. Verdict

**CONDITIONAL GO — build nothing until MF-1/2/3 land: name the Mama's slice, split the gate, pre-register gate-1. The spec knows what to build; it just forgot to say "this bit first" — and right now the fun part is padlocked behind the boring part, which is exactly how this one dies.**
