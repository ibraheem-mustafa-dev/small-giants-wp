# P2.5 Adversarial Council — THE CYNIC on Spec 36 (SGS Navigation System v1.1)

**Reviewer lens:** grizzled framework maintainer. Assumes every clean-sheet rewrite re-accretes the mess it replaced. Every complaint is actionable with a concrete fix + cited FR.
**Date:** 2026-07-18
**Verdict up front:** the a11y bones and the decoupling are genuinely good. But the *coordination substrate* — 4 reference edges across 4 stores with **no integrity owner** and a **"stable item ID" the spec never proves exists** — is the exact disease that killed the predecessor, re-created with cleaner names. This is a NO-GO until MUST-FIX 1–4 are closed.

---

## 1. STRENGTHS (honest, brief)

- **Position-keying → stable-ID-keying is the right diagnosis** of the old "mega renders last / can't place Brands at position 4" bug (FR-36-5). Real-position render derived from **walk order** (not a stored index) fixes it structurally, not by patch. Correct.
- **Native `<dialog showModal>` for the default drawer drops Spec 34's re-parent-to-`<body>` hack.** A top-layer modal dialog is positioned relative to the viewport regardless of ancestor transforms — this genuinely retires Spec 34 FR-34-1's whole re-parent/inert-ancestor saga *for the default path*. Real simplification.
- **Disclosure-vs-dialog a11y split (FR-36-10) is primary-source-grounded** (APG, Roselli, MDN): buttons+`aria-expanded`, no `role="menu"`, Tab-through on disclosure, contained on dialog. This is the part most competitors get wrong. Keep it exactly as-is.
- **One-directional nav→header coupling via a published state surface (FR-36-9)** is the correct decoupling primitive — reads `--sgs-header-height` + `data-sgs-header-state`, header never reads nav.
- **Crawlable/no-AJAX (FR-36-17)** is a real, defensible differentiator vs Kadence/Spectra mega-menus that lazy-load panels.
- **Non-goals named, not dropped** (FR-36-8 bottom-tab/⌘K); cutover exists with a "measure zero stored instances before deletion" gate (FR-36-18). Good discipline.

## 2. MUST-FIX (build-breaking / will-rot)

### MF-1 — The "stable `wp_navigation` item ID" is asserted, never proven to exist. The entire fatal-bug fix rests on a platform primitive that isn't native.
**Problem.** FR-36-5 and FR-36-15 key the item→mega map on the `wp_navigation` item's **"stable ID"** and repeat "NEVER position." But `wp_navigation` stores its tree as serialised **`core/navigation-link` / `core/navigation-submenu`** blocks. Those blocks carry `label / type / id / kind / url / opensInNewTab / rel / description / className` — and **no persistent, unique, per-item opaque key**. The `id` attribute is the *referenced object* ID (the linked page), so it is **not unique** (link the same page twice → two items, same `id`) and **absent** on custom-URL items. There is no native GUID that survives the operator editing the item. Worse: **the Navigation editor is core's, not SGS's** — MEMORY (`verify-platform-claims-and-use-code-grounded-council`) already burned this project once, when 4/5 "WP nav levers" turned out to be uncallable core-nav internals. Minting a stable per-item key means extending core's navigation-link block insertion, which SGS does not own.
**Effect.** If the "stable ID" is really "the linked-object id" or a positional fallback in disguise, the new map has the **identical disease** that killed the old mega-menu — with a new name. The whole "structurally impossible to mis-place a mega" claim (FR-36-5) collapses.
**Fix.** Add an FR that *defines the stable-ID mechanism concretely*: (a) register a block-attribute filter that mints a UUID onto every `core/navigation-link`/`-submenu` at insert (and prove it survives re-serialisation in core's Navigation editor — a live test, not an assertion, per prove-the-cause-before-fix); (b) have the converter (FR-36-15) mint the same UUID when it writes the `wp_navigation` post; (c) specify the fallback when an item has **no** UUID (legacy/hand-added items) — deterministic, not positional. Until (a) is proven live in core's editor, treat this as an **unverified platform claim** and gate it with a code-grounded critic.

### MF-2 — Four reference edges, four stores, ZERO named integrity owner. Nobody garbage-collects orphans. This is the predecessor's exact fragility, doubled.
**Problem.** Enumerate the edges the system must keep coherent:
1. `sgs/nav-menu.ref` → `wp_navigation` post
2. `sgs/nav-menu.itemMegaMap[itemUUID]` → `sgs_mega_menu` post ID (FR-36-5)
3. `sgs/nav-menu.drawerRef` → `sgs/nav-drawer` anchor/ID (FR-36-8)
4. `sgs/nav-drawer` → its **own** `sgs/nav-menu.ref` → a possibly-**different** `wp_navigation` (FR-36-1/FR-36-6)

The spec specifies **storage** for these but **not one word on referential integrity**. What happens when:
- an operator **deletes a menu item** in the Navigation editor → map entry (edge 2) is orphaned AND its `sgs_mega_menu` post is now an un-GC'd zombie in the CPT home;
- an operator **trashes a mega panel** in the CPT admin → edge 2 dangles → render.php must survive a null lookup (unspecified);
- the operator **deletes/re-inserts the nav-menu block** (new uid) → edge 3 `drawerRef` targets go stale;
- a **drawer is deleted** → the burger opens nothing.

The predecessor died because *nobody owned coherence across `wp_navigation` + the item→mega map + the mega store*. Spec 36 has **more** stores and **still** no owner.
**Effect.** Orphan accumulation, dead map entries, burgers that open nothing, zombie mega posts cluttering the CPT home (see MF-4). Guaranteed rot by the 3rd client.
**Fix.** Add an FR-36-19 "Referential integrity contract": (a) render.php **null-safe** on every edge (missing mega → simple dropdown; missing drawer → burger disabled with a Notice, not a JS error); (b) a **cleanup hook** on `wp_navigation` save + `sgs_mega_menu` trash that prunes/flags orphaned map entries; (c) an operator-facing **informational Notice** (per FR-36-12) listing dangling references; (d) the acceptance gate (FR-36-16) must include a **delete-an-item + trash-a-mega + reorder** integrity sweep, not just the happy-path reproduction.

### MF-3 — When the drawer references a DIFFERENT menu than the bar, "the mega renders inside the drawer too" is incoherent — the map lives only on the bar.
**Problem.** FR-36-1/FR-36-6 permit the drawer's `sgs/nav-menu` to reference a **different** `wp_navigation` than the bar (deliberate mobile-vs-desktop choice). FR-36-5 says the mega panel "renders on mobile **inside the drawer** too." But the item→mega map is keyed by the **bar menu's** item UUIDs and lives on the **bar's** nav-menu instance. The drawer's nav-menu is a **separate block instance** (own uid/attributes) pointing at a **different** item tree whose UUIDs don't exist in the bar's map. Even in the common case (same menu), you now have **two nav-menu instances**, so either the drawer's instance carries its **own** map (→ two maps drift out of sync) or it must read the bar's (→ cross-instance coupling the spec never defines).
**Effect.** Mega-in-drawer either silently shows nothing on mobile (different-menu case) or the two maps drift (same-menu case). Both are invisible until a client on mobile reports "my Brands panel is gone."
**Fix.** Decide and specify **one map owner** in FR-36-5: either (a) the map lives on the `wp_navigation` menu's associated meta (single source, both instances read it — but this re-opens MF-1's "where does it live" question), or (b) the map lives on the **drawer/bar pair via a shared reference** so there is exactly one authoritative map per menu. Then define explicitly what mega-in-drawer does when bar≠drawer menu (recommendation: **no mega in drawer for a different-menu drawer** — state it, don't leave it implied).

### MF-4 — The item→mega **map-editing UX is the entire product for a non-coder, and it is unspecified.** Three screens, no bridge.
**Problem.** To associate item X with rich panel Y, the operator must touch **three separate surfaces**: the **Navigation editor** (create the item — core's screen), the **CPT admin home** (build the mega post), and the **nav-menu block inspector** (wire item→post in the map). FR-36-14 name-drops "the mega-panel picker" as permitted custom UI, but **no FR describes how it works**: how does the nav-menu inspector *list the `wp_navigation` items* (a different entity the block only holds by `ref`) so the operator can pick one? How are nested items shown? Create-new-mega-from-here vs pick-existing? What updates the picker when items are renamed/reordered/**deleted** in core's Navigation editor? Kadence and Spectra win precisely here — they attach mega assignment **on the menu item itself**, one screen. SGS splits it across three with an undefined bridge.
**Effect.** The headline "meet AND exceed Kadence" fails on the one workflow that matters most. Cross-entity editing in Gutenberg (reading `wp_navigation` children into another block's inspector, staying live as core mutates them) is notoriously painful and is the hardest thing in the build — specced as a parenthetical.
**Fix.** Promote this to a first-class FR-36-20 "Mega association UX": specify the picker's data source (`useEntityRecords('postType','wp_navigation')` + a resolver that flattens the block tree to `{itemUUID, label, depth}`), the create-vs-pick flow, the nested-item handling, and the **live-sync behaviour when core mutates the menu**. Prove it against the Indus 7-item + 1-mega case in the build gate (FR-36-16), not just the render.

### MF-5 — Two drawer modalities (modal + non-modal selective-freeze) in the from-scratch pass re-imports every Spec 34 bug on day one.
**Problem.** FR-36-6 ships **both** a full-screen **modal** (`<dialog showModal>`, clean) **and** a below-header **non-modal** path requiring: Popover API top-layer, a `.show()`+`<body>`-portal fallback, **selective `inert`** of everything except {header row + drawer + scrim + `#wpadminbar`}, `role="dialog"` without `aria-modal`, top-edge bound to `--sgs-header-height`, plus a **forward dependency on a "header hidden on scroll" feature that doesn't exist yet** (FR-36-9). This non-modal selective-freeze path **is Spec 34** — the exact code that needed Opus + a qc-council + a re-parent hack + a self-inert lesson and still bled. The `.show()` fallback (non-modal, **not** top-layer) does **not** survive a transformed header ancestor — it re-imports Spec 34's original positioning bug verbatim.
**Effect.** The clean-sheet rewrite ships the predecessor's single most fragile subsystem on day one, as an **option most clients won't use**. Maintenance treadmill from commit one.
**Fix.** Ship the **modal full-screen default only** in the from-scratch build (native `showModal`, `::backdrop`, contain-by-construction — covers Mama's + Indus per FR-36-16). **Defer the below-header non-modal modality to its own later spec with its own gate** and its own forward-dependency (the hide-on-scroll header). Cite FR-36-6 + FR-36-9. Do NOT build both to "meet the exemplar list"; the exemplars collapse the same way at every width (the spec's own FR-36-8 argument).

## 3. SHOULD-FIX

- **SF-1 — `drawerRef` staleness/collision is the uid problem Spec 34 already hit.** FR-36-8 wires `drawerRef` to a drawer "block anchor/ID," and Spec 34 FR-34-4 documented that md5(attributes)-derived uids **collide for two default instances on one page**. The "if unset, target the single drawer on the page" auto-target breaks the moment a client has two headers (desktop + a landing variant → the 6th-client case). *Fix:* specify anchor **stability + uniqueness** (operator-visible required anchor field, or a persistent minted id like MF-1's), and define behaviour for 0 / 2+ drawers explicitly, not "explicit pick."
- **SF-2 — No migration story for OPERATOR-authored megas + maps.** The no-deprecations policy ("re-cloned, not migrated") works for **pipeline-emitted** content because it's re-derivable from a draft. The **item→mega map and the hand-built mega posts are operator-authored** — there is no draft to re-clone. The first post-launch change to the map attribute shape breaks every client's hand-wired associations with **deprecations banned** (§FR-36-18 / plugin policy). *Fix:* acknowledge in §9 that operator-authored nav data needs a shape that is **frozen at launch** (like the Spec 17 attr freeze) OR a one-off migration exemption — pick one, state it.
- **SF-3 — The 5 mega layouts live in three unsynced places.** Starter patterns (git, FR-36-3), the converter's layout→template map (FR-36-15), and the acceptance gate's "which layouts Indus uses" (FR-36-16). Classic three-copy drift — add a 6th layout and someone forgets one. *Fix:* make the layout roster **DB/registry-defined** (consistent with the project's DB-first rule R-31-1); starter-picker and converter both **read** it.
- **SF-4 — Discoverability rule fights the performance budget on image-heavy megas.** FR-36-17 mandates "NO lazy-load of nav OR mega content"; FR-36-5 allows photo-grid/logo-grid megas. The nav is global, so **every** mega's markup **and images** ship in the header of **every** page. A 6th client with a rich mega per item blows the <100 KB CSS budget and no-CLS promise. *Fix:* split the rule — **links** are never lazy (crawlability), but below-fold mega **`<img>`** may carry `loading="lazy"` (doesn't affect crawlable `<a href>`). State this distinction in FR-36-17 so it doesn't get "fixed" to flat no-lazy.
- **SF-5 — The "one utility, per-mode `if` branches" is honest but is the named seam where the monster grows.** FR-36-7 already admits it "is not one primitive." It must serve: hover-intent+safe-triangle (bar/mega only), Escape (all), focus-contain (modal drawer), inert-all (modal), inert-selective (non-modal), Popover (non-modal), showModal (modal), `<details>` no-JS (drawer only). *Fix:* if MF-5 is accepted (modal-only default), this shrinks dramatically. Regardless — require the spec to enumerate the branch matrix (surface × behaviour) as a table in FR-36-7 so a future maintainer can see which branch serves which surface; an un-mapped if-pyramid at 600 lines is the two-year rot.
- **SF-6 — No-JS baseline rests on a ⚠VERIFY primitive but is listed as a hard §9 constraint.** FR-36-6's `<details name>` exclusive-accordion (2024, marked ⚠VERIFY) can't be both "verify support" **and** a binding constraint. *Fix:* either guarantee the baseline with plain `<details>` (accept multiple-open, no `name`) or demote the no-JS accordion from §9 constraint to enhancement.

## 4. MISSING (highest-value absent dimensions)

- **M-1 — A referential-integrity / orphan-lifecycle model** (the substance of MF-2). Absent entirely. This is the single highest-value missing dimension: it's *what killed the predecessor* and the spec adds stores without adding an owner.
- **M-2 — The map-editing UX as a specified surface** (MF-4). The product's core workflow is a parenthetical.
- **M-3 — The `sgs_mega_menu` CPT-home scaling/organisation model.** FR-36-3 dumps megas into the header/footer CPT admin home; FR-36-15 admits titles/slugs collide and offers only "a naming convention" — the weakest possible integrity mechanism, and exactly the "discipline over structure" anti-pattern the project's own MEMORY rejects. By the 6th client with multi-item mega bars, "my Headers" is a junk drawer of panels. *Missing:* a taxonomy/scoping/filter model (client scope + a "mega panels" filtered view separate from headers/footers).
- **M-4 — Concurrency / two-editors + the "who wins" story** across the 4 stores. Two operators (or the converter re-cloning while an operator edits) touching `wp_navigation` + the map + a mega post has no defined ordering. Given the converter is a first-class writer (FR-36-15), a re-clone over hand-edited associations needs a stated rule.
- **M-5 — An explicit "what the converter does with a scraped nav it can't cleanly map" fallback.** FR-36-15 assumes a clean draft→system write. Real scrapes have `role="menu"` markup, JS-only dropdowns, hamburgers with no semantic tree. The emit target is defined; the **degradation path** (emit a plain crawlable bar + a Notice) is not.
- **M-6 — RTL / logical-property behaviour for the drawer edge + safe-triangle geometry.** `edge: left|right`, safe-triangle cones, and drill-down slide direction are all direction-sensitive; nothing addresses RTL (a real concern for the client base). Cheap to state now, expensive to retrofit.

---

## 5. GRADE (maintainability): **D+**

The pilot menus (Mama's, Indus) will ship and look good — the a11y contract, the native-modal default, and the decoupling are sound. But **maintainability** is graded on what happens at the 6th client and the first post-launch schema change, and there the spec re-creates the predecessor's core disease: a multi-store coordination substrate with **no integrity owner**, a **map key that isn't a proven platform primitive**, an **unspecified cross-entity editing UX**, and a **doubled disclosure/dialog branch-monster** shipped on day one. These aren't polish gaps — MF-1 through MF-5 are load-bearing. Close them and this moves to a B. As written, it will rot on the same rails as Spec 34, just with cleaner FR numbering.

---

## 6. Verdict (one line)

*You renamed the three-store mega-menu that rotted, added a fourth store and a "stable ID" nobody's proven exists, left no one holding the integrity mop — and shipped Spec 34's selective-freeze ghost as a bonus option. Clean FRs, same treadmill. NO-GO until MF-1–5 close.*
