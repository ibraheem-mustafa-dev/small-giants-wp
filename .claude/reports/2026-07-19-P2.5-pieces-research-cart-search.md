# P2.5 pieces research — Header CART + SEARCH (the ideal, unbiased)

> Purpose: define what a best-in-class header cart + search SHOULD do, before looking at the current
> `sgs/cart` / `sgs/product-search` code — so readiness is measured against the ideal, not against what
> already exists. (The prior reused-blocks audit measured cart against a thin "live-count badge" bar and
> called it ~90% ready; this is the proper bar.)

## CART

### Feature set (ranked)
**MUST:** live item-count badge · AJAX add-to-cart no reload (Store API, NOT legacy cart-fragments — cache-safe) ·
accessible live-region announcement on count change (WCAG 4.1.3, `role="status"`/`aria-live="polite"`/
`aria-atomic="true"` on the WHOLE "N items" string) · mini-cart/drawer preview (thumbnails, name, qty, line
price) · inline qty edit + remove (no page redirect) · clear "View Cart" + "Checkout" CTAs · full-cost
transparency (subtotal + shipping/tax note) · distinct actionable empty-cart state (not a blank panel) ·
no conflict with full-page caching · keyboard-operable + focus-managed + ESC.

**SHOULD:** slide-in drawer as a selectable display mode (vs flyout) · auto-open on add-to-cart (configurable,
dismissible) · subtotal shown on the header trigger itself · fully theme-able icon/badge · hide the mini-cart
on cart/checkout pages · persistent cart across session/device (logged-in) · recommendations/recently-viewed
in the empty state.

**NICE:** free-shipping progress bar (needs drawer real-estate) · cross-sell/upsell slot · sticky mobile
"view cart" bar.

### Display-mode taxonomy (client picks one)
1. **Icon + badge → cart-page link** (simple stores) — plain link, `aria-label` states count.
2. **Mini-cart flyout/dropdown** (default rec) — anchored panel, page stays usable → **DISCLOSURE** pattern.
3. **Slide-in cart drawer (off-canvas)** — larger, dims/scrims the page → **DIALOG (modal)** pattern.
4. **Auto-open-on-add** — a behaviour layered on 2 or 3, must be dismissible/non-blocking.

### A11y contract (the recurring rule)
The ARIA pattern is decided by ONE question: **does the open panel leave the rest of the page usable
(→ DISCLOSURE: `button[aria-expanded][aria-controls]`, ESC, no focus trap) or does it dim/block the page
(→ DIALOG: `role="dialog" aria-modal="true"`, focus trap, `inert` background, ESC)?** Count changes = live
region (WCAG 4.1.3), the same for search result counts.

### Differentiators (EXCEED)
1. **displayMode (link|flyout|drawer) as ONE attribute that auto-swaps the ARIA pattern** (disclosure↔dialog) — competitors implement flyout vs drawer as separate systems with inconsistent a11y.
2. **Cache-safe by default** (Store API, no legacy cart-fragments — WooCommerce's own guidance; competitors mostly still route through fragments).
3. **Empty-cart state as a first-class inspector-configurable slot** (recommended/recently-viewed/continue-shopping) — none of Kadence/Blocksy/Spectra/Bricks expose this.
4. **Correct WCAG 4.1.3 live-region by default** — competitor docs don't mention it.

## SEARCH

### Feature set (ranked)
**MUST:** predictive/live suggestions as you type (Baymard: ~24% more sales, present on 80% of sites — the
highest-leverage search feature) · debounced, capped list (≤10 desktop / 4–8 mobile) · `<form role="search">`
no-JS fallback that still returns real results · full ARIA COMBOBOX pattern (`role="combobox"` +
`aria-expanded`/`aria-controls`/`aria-activedescendant` + `role="listbox"`/`role="option"`, arrow/Enter/Esc) ·
highlight the matched portion (not the typed portion) · result previews (thumbnail + price for products) ·
scoped search with an UNSCOPED default (NN/g rule: default unscoped, scope as an option).

**SHOULD:** icon-that-expands as a selectable display mode · full-screen mobile search overlay · recent-searches
on focus (labelled) · enriched suggestions (top sellers/categories) CLEARLY labelled · dimmed page background
while suggestions open.

**NICE:** popular/trending searches when empty · inline faceted results · voice input.

### Display-mode taxonomy
1. **Always-visible inline bar** (search-primary/content-heavy sites).
2. **Icon that expands** (design-led, space-precious) → the reveal is a DISCLOSURE.
3. **Full-screen overlay** (mobile-first / search-primary) → the shell is a DIALOG, combobox nested inside.
All modes converge on the same combobox contract once focused.

### A11y contract
Form = `<form role="search" aria-label="Site search">` · expand-icon = disclosure · the live-suggestions
widget = editable combobox with listbox popup (focus stays on the input, `aria-activedescendant` tracks the
option) · full-screen overlay = dialog around the combobox · result count / "no results" = live region (4.1.3).

### Differentiators (EXCEED)
1. **Framework-native predictive search wired to the Store API (products) + post search (content)** with the
   ARIA combobox built in — `core/search` has none of this; competitors bolt it on via premium modules /
   third-party (FiboSearch).
2. **One shared combobox implementation reused across inline/icon/overlay** (correct + less code) — no
   competitor documents APG-combobox conformance.
3. **Result preview (price/thumbnail) native**, not a paid add-on.
4. **Scoped search to NN/g's rule** (default unscoped, scope as an explicit toggle).

## Cross-cutting (for the spec)
- Store API for cart; debounced (not per-keystroke) fetch for search — both avoid the cache-conflicting
  legacy admin-ajax path WooCommerce advises against.
- Mobile: dedicated full-viewport treatment (drawer for cart, full-screen overlay for search), not a squeezed
  desktop flyout.
- **The disclosure-vs-dialog question is the ONE a11y decision gate for EVERY interactive header utility**
  (cart, search, dropdown, drawer) — make it the spec's explicit gate.
