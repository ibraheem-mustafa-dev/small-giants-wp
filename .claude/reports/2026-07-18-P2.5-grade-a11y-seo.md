# Grade — Spec 36 v1.2, accessibility + technical SEO/AI-search

Scope: FR-36-10/11/12 (a11y contract), FR-36-17 (discoverability), §8 acceptance gate, plus everything else
in the spec that bears on these two axes. I did not re-report items the spec itself marks as already folded
from the adversarial council. One live-code spot-check performed (see below); no live site/DOM testing done
— this is a spec-quality grade, not a QC pass on a built artefact.

---

## 1. Accessibility — score 4/5

**Evidence.**

- FR-36-10 correctly separates two distinct WAI-ARIA patterns and states the anti-pattern explicitly: dropdowns/mega = **disclosure** (`<button aria-expanded>`, no `role="menu"/"menubar"`, Tab passes through, no trap), drawer = **dialog** via native `<dialog showModal>`. This is the single most common nav a11y bug in the wild (menu/menubar misuse on hover nav — well documented by APG/Adrian Roselli), and the spec names it as a thing to avoid. Correct call, matches current APG guidance (disclosure pattern for nav submenus, not the menu/menubar pattern which is reserved for application-style menus).
- `aria-haspopup` is explicitly **omitted** (FR-36-10) — correct; `aria-haspopup` on a disclosure-pattern submenu is a known false-positive VoiceOver/NVDA verbosity bug, APG doesn't require it for disclosure.
- FR-36-6 dialog contract: focus-into on open, Tab contained (native, since `showModal` provides this), Escape closes, focus returns to burger, body-scroll-lock incl. iOS fix named explicitly, swipe-close never the *only* close path. This is a complete, correct modal-dialog checklist. Explicitly says "do NOT add `role=dialog`/`aria-modal`" — correct, native `<dialog>` doesn't need ARIA duplication and adding it is a known anti-pattern (redundant/conflicting semantics in some AT).
- FR-36-11 WCAG coverage is genuinely broad and specific, not generic: `aria-current="page"`, unique `<nav>` labels (multiple-nav labelling — a real, commonly-missed WCAG 4.1.2/1.3.1 issue), accessible names on icon-only buttons, live `aria-expanded`, `:focus-visible` ≥3:1 (SC 1.4.11 non-text contrast), **SC 2.4.11 Focus Not Obscured** (2.2 AA, correctly the "cheap win" the project baseline calls for), **SC 2.5.8 target size** — and the spec states SGS's own 44px bar exceeds the 2.2 minimum (24px) rather than just meeting it. `forced-colors`/`prefers-contrast`, "no colour/motion-only state" also present. This is a well-above-average WCAG enumeration for a nav spec — most nav specs stop at focus/keyboard and miss 2.4.11/1.4.11/1.4.13.
- FR-36-4 also names **WCAG 1.4.13 (Content on Hover or Focus)** explicitly with all three sub-requirements (Dismissible/Hoverable/Persistent) for the hover-opened mega/dropdown — this is frequently missed entirely in nav specs and is directly relevant to hover-intent mega panels.
- Safe-triangle + hover-intent + close-grace delay (FR-36-4) addresses the "diagonal mouse movement closes the menu" usability bug, which is also an indirect a11y/motor-impairment concern (SC 2.5.x adjacent).
- No-JS honesty (FR-36-7) is unusually rigorous for a spec: it explicitly states the SCOPE of "crawlable without JS" is links-only, not panel-interactivity, and forbids overclaiming ("state this scope honestly"). This is a genuine anti-inflation discipline baked into the spec itself.
- FR-36-9a (referential integrity) treats a broken mega/drawer reference as a **user-facing failure mode**, not just a build concern — dangling refs degrade to a plain link or a no-op-with-warning burger rather than a silent broken UI. This is good defensive a11y/UX engineering, distinct from and complementary to the WCAG checklist.
- FR-36-12 makes the (Bean-locked, already-decided) call that operator-facing a11y feedback is informational-only, never a gate — this is a locked policy decision (see MEMORY: `a11y-validation-feedback-informational-not-gate`), correctly not re-litigated here, and the spec is right to note it doesn't touch the WCAG default-render baseline.
- Live-code check: I grepped the plugin tree and confirmed `--sgs-header-height`, `is-header-scrolled`, `is-header-shrunk` are real, live symbols (`header-behaviours/view.js`, `class-sgs-header-behaviours.php`, `header-behaviours.css`) — FR-36-9's claimed "real published state surface" is not a fabricated citation.

**Gaps found (a11y).**

1. **[B] FR-36-11 doesn't name the specific 2.2 SC for "Focus Appearance" (2.4.13) even as AAA-optional context, and doesn't state a concrete `:focus-visible` outline spec (width/offset) — "≥3:1" contrast is stated but not the geometry.** Minor: WCAG 1.4.11 (3:1 non-text contrast) covers the colour requirement adequately for AA; 2.4.13 is AAA so its absence is not a compliance gap, just a missed "beyond-AA" opportunity. Low fix value at AA baseline.
2. **[B] The in-drawer `<details name>` no-JS fallback is explicitly flagged `⚠VERIFY` and "degrades to non-exclusive `<details>`" (FR-36-6) without stating what happens to `aria-expanded` state parity between the JS-enhanced and no-JS states.** Browser support for `<details name>` (exclusive accordion groups) is real but recent (Chrome 120+, Firefox 125+, Safari 17.2+) — the spec is honest about needing a build-time check, which is correct discipline, but doesn't specify the fallback's keyboard/AT behaviour if `name` isn't supported (does it become N separate expand/collapse widgets with correct `aria-expanded` on `<summary>`? — implicit but not stated).
3. **[C] No explicit statement of what happens to `aria-expanded`/focus state for the "More" overflow disclosure (FR-36-8a) when its own trigger is itself keyboard-collapsed** — i.e. the "More ▾" is itself a disclosure "per §5" (correctly stated) but the interaction of priority+ reflow (items moving in/out of the overflow at resize) with a currently-open overflow panel isn't addressed (a resize while a disclosure is open is a known edge case that can leave a detached/orphaned open panel). Low likelihood in practice, not fatal, but untested territory.
4. **[C] FR-36-4's "caret on expandable items only" + "distinct hover+focus states" don't specify whether the caret itself is `aria-hidden` (decorative) — implicit best practice but not stated, so it's inheritable-by-omission rather than an explicit contract line.** Trivial to satisfy but the spec doesn't gate it.
5. **[D] The "safe-triangle" mechanism is a JS geometry heuristic — the spec doesn't name a fallback for touch/hybrid devices (tablet with mouse+touch) where hover-intent timing behaves inconsistently.** FR-36-4's "click/Enter/Space" alternative already covers this in practice (touch triggers activate on tap = click-equivalent), so this is close to a non-issue, flagged only for completeness.

None of these are gate-blocking; all are refinements on an already-strong contract. The a11y contract is close to exemplary in its enumeration of WCAG SCs and correct pattern selection, but stops short of "beyond-AA, proactively hardened for every edge case" — hence 4, not 5.

## 2. SEO / crawlability / AI-search — score 4/5

**Evidence.**

- FR-36-17 states unambiguously: **every bar/dropdown/mega link is a real `<a href>` in the initial server HTML, no lazy-load of nav or mega content, no AJAX (D334)** — this is the single highest-value technical-SEO property a nav can have (JS-hydrated/AJAX-loaded mega menus are a well-known crawl-budget and AI-crawler-visibility failure mode; most page-rendering AI crawlers — GPTBot, ClaudeBot, PerplexityBot — do not execute JS reliably). Correctly scoped and named as the differentiator, not just a checkbox.
- FR-36-7's no-JS honesty statement reinforces this: **links are always crawlable even when the enhancement JS fails or doesn't run** — this is architecturally the right guarantee (progressive enhancement, not JS-required rendering), and it's stated as a hard contract line, not aspiration.
- FR-36-17 also states the *moat* correctly and modestly: "plain crawlable links are table stakes; the differentiator is server-rendered rich mega content (no AJAX) + AI-auto-generation of the whole nav." This is a mature, non-inflated self-assessment — it doesn't oversell "we emit `<a>` tags" as a competitive advantage, which is good anti-inflation discipline for a spec grading itself.
- Semantic markup: `<nav>` landmarks + unique labels, real `<ul>/<li>/<a>` (not div-soup), descriptive anchor text requirement — correctly ties nav to internal-linking SEO value, which is real (anchor text in primary nav carries disproportionate topical-relevance signal).
- Schema: correctly **delegates** JSON-LD emission to the `seo-schema` skill rather than reinventing it in-block (matches Spec 35's "no schema in blocks" rule) — architecturally correct separation of concerns, avoids duplicate/conflicting schema emission, a real risk when multiple blocks each try to emit `SiteNavigationElement`.
- Performance/CLS: sticky header reserves space, overlays don't reflow, perf budget stated, `wp-perf-gate` gate named in §8 — Core Web Vitals (CLS specifically) are a ranking factor and a common nav-specific failure (menu paint-in causing layout shift). Correctly addressed.
- §8's crawl assertion is concrete and testable (see §3 below): "every bar + dropdown + mega link in the server HTML pre-JS" — this is a real, automatable check (fetch HTML without executing JS, grep for `<a href>` count matching menu item count), not a vague "should be crawlable" statement.
- Parity/no-cloaking clause (FR-36-17) — bar and drawer may bind different menus, but every link in either must be crawlable, "nothing hidden from crawlers that users reach, or vice-versa" — this correctly heads off a cloaking risk that a naive mobile-vs-desktop-menu implementation could introduce (Google's guidance explicitly warns against showing different linksets to crawlers vs. users).

**Gaps found (SEO/crawlability).**

1. **[A] No `hreflang` implication despite multilingual menus being a named Phase-3 deferral.** Not a v1.2 defect (correctly deferred, and the spec even flags WPML/Polylang's `intl`-extension caveat under Non-goals) — but there is no forward-note in FR-36-17 that when multilingual ships, the nav's link markup must carry `hreflang` alternates or be `hreflang`-compatible by construction. Because nav is the primary crawl-discovery surface, retrofitting hreflang onto an already-built nav block later is higher-risk than designing for it now. Graded A (high fix-value: cheap to add a one-line forward-compat note now; expensive to retrofit after Phase 1/2 ship) even though it's currently out of scope — the absence of even a one-line "when multilingual lands, nav link markup must not need restructuring for hreflang" note is a real gap for a spec that otherwise names its forward dependencies carefully (c.f. FR-36-9's "Forward dependency" note).
2. **[B] Mega-menu content SEO is only addressed as "crawlable links," not as indexable/valuable content in its own right.** A mega panel with descriptions per FR-36-5 ("optional descriptions") is real on-page content, but the spec doesn't state whether mega-panel body text is indexed as part of the page it appears on (nav is typically excluded from `main`/article content extraction by AI summarizers and sometimes by Google's main-content algorithms if it's in a `<nav>` landmark) — worth a one-line note on whether rich mega descriptions should also appear as a semantic landmark distinct from pure link-list content, since over-stuffing `<nav>` with paragraph copy can dilute rather than help topical signals.
3. **[C] No explicit statement on `rel` attributes for external/CTA links in the nav (e.g. `rel="noopener"` for target=_blank, or a stance on `nofollow` for non-content utility links like "Show header"-adjacent controls).** Minor, standard WP core-nav behaviour likely covers this by inheritance but it's not stated as a contract line.
4. **[D] Anchor-text quality ("descriptive anchor text") is stated as a requirement but not testable/gated in §8** — see testability section below, this is really a cross-cutting gap between SEO and testability.

Score is 4 (ship-to-a-paying-client, strong differentiator correctly scoped) rather than 5: the core crawlability contract is excellent and honestly self-scoped, but the spec treats SEO as "don't break crawlability" rather than "actively engineer for AI-search extraction" beyond the one moat paragraph — e.g. no mention of structured heading hierarchy inside mega panels, no mention of `aria-label` text itself being crawl-signal-relevant, no llms.txt/AI-crawler-specific consideration beyond naming the bots. Good, not yet exemplary.

## 3. Testability of the a11y + SEO acceptance criteria — score 4/5

**Evidence.**

- §8 (FR-36-16) is unusually concrete for a spec's acceptance section: named breakpoint sweep (375/768/1440) **plus** an explicit non-default collapse-N sweep (catches the common bug of only testing the default breakpoint); named cache-clear-first step (a real, previously-bitten gotcha — cache staleness invalidating a "gate passed" result); **axe = 0 on both the open drawer AND an open desktop mega** (not just one surface — this closes a common gap where teams only axe-test the drawer and never test an open mega panel, which has its own focus/ARIA surface); `elementFromPoint` sweep with a **named numeric baseline** (10/10 Mama's, 18/18 Indus) rather than a vague "no dead click zones" — this is genuinely a strong, regression-proof, machine-checkable bar.
- ESC/focus-return/Tab-containment, scroll-lock frame sweep, drawer geometry with a **measurable tolerance** (`top` = header-bottom ±1px) — precise enough to automate via Playwright `getBoundingClientRect` comparison.
- **The crawl assertion is explicitly defined as "every bar + dropdown + mega link in the server HTML pre-JS"** — this is directly automatable (fetch raw HTML, count `<a href>` vs menu-item count) and ties straight back to FR-36-17. Good closed loop between the requirement and its gate.
- "the mega renders at real position N, not last" is a specific regression test tied to a named historical bug (FR-36-5's "fatal-bug fix") — testable via DOM order assertion.
- The integrity sweep (FR-36-9a, "no dangling refs on the built pages") is named as part of the gate, not left as an unenforced aspiration.
- `wp-perf-gate` named for the perf budget (FR-36-17) — ties to an existing tool rather than inventing a new one.
- Per Spec 31's R-31-13 doctrine (numbers + Bean's eye both required, neither alone closes) the gate correctly also requires "Bean's cropped screenshot pair" alongside the numeric/automated checks — consistent with the project's own closing methodology, not just a numbers-only gate that could pass while looking broken.

**Gaps found (testability).**

1. **[B] "Descriptive anchor text" (FR-36-17) and "unique labels on multiple `<nav>`s" (FR-36-11) are both real WCAG/SEO requirements but neither has a corresponding line in the §8 gate.** Axe-core partially catches missing/duplicate `aria-label` on landmark regions (`landmark-unique` rule) so this may be implicitly covered by "axe = 0," but the spec doesn't state that explicitly — worth a one-line cross-reference so the gate's axe scope is provably comprehensive rather than assumed.
2. **[B] `prefers-reduced-motion` and `forced-colors`/`prefers-contrast` (FR-36-11, FR-36-4, FR-36-6) are stated as requirements but §8's sweep list doesn't name a reduced-motion or forced-colors pass explicitly** (only "375/768/1440 sweeps," a "late-CSS A/B," and the axe pass are named as visual/behavioural sweeps). Axe-core does not test `prefers-reduced-motion` or Windows High Contrast Mode behaviour — these need an explicit manual or emulated-media-query pass, which the gate doesn't call out despite the FRs requiring them. This is the most concrete testability gap: two explicitly-required a11y behaviours (2.2-adjacent motion + forced-colors) have no named verification step.
3. **[C] The mega's "optional descriptions" / heading hierarchy inside a mega panel isn't given a specific automated check** (e.g. "no heading-level skip inside a mega panel") — axe-core's `heading-order` rule would catch this if enabled, but again not stated.
4. **[C] No named check for `aria-expanded` accuracy under rapid open/close (race-condition) states** — a common source of state-desync bugs in hover-intent + click-toggle hybrid disclosures (FR-36-4 explicitly supports both hover AND click on the same trigger, which is exactly the combination most prone to state races). The FR names the interaction mode but the gate doesn't test the interaction-mode combination itself.

Score 4: the gate is well above typical "click around and eyeball it" nav QC — it has named numeric baselines, a crawl assertion, and dual axe passes. It falls short of 5 because two explicitly-required behavioural FRs (reduced-motion, forced-colors) have no corresponding named verification step in the one section whose whole job is to enumerate exactly that.

---

## Overall score: 4.0 / 5

Weighted evenly across the three criteria (accessibility 4, SEO/crawlability 4, testability 4) — a genuinely consistent, no-outlier result rather than one strong axis masking a weak one. This is a **ship-to-a-paying-client** spec, not merely "solid AA with gaps": it correctly selects the disclosure-vs-dialog pattern (the single most consequential nav a11y decision and the one most specs get wrong), states its no-JS/crawlability scope honestly rather than overselling, and gates on machine-checkable numeric baselines rather than vibes. It is not yet a 5 (exemplary, beyond-AA + AI-search-optimised): the reduced-motion/forced-colors testability hole, the missing multilingual/hreflang forward-note, and the "crawlable ≠ AI-search-optimised beyond the moat paragraph" ceiling are the concrete distance from best-in-class.

## Upgrade opportunities (best-in-class)

1. **Close the reduced-motion/forced-colors gate hole (G1) and add it as a named §8 sweep step** — e.g. "emulate `prefers-reduced-motion: reduce` and `forced-colors: active` and re-run the drawer/mega open sweep; assert animation duration ≈0 and no shadow-only boundaries visible." This converts an already-correct FR into a provably-enforced one and is the single highest-leverage a11y+testability fix available (cheap: one extra emulated-media Playwright pass; high value: it's the exact class of bug that ships invisibly because it only appears under a non-default OS/browser setting).
2. **Add a one-paragraph "AI-search extraction" design note to FR-36-17 beyond the crawlability moat** — e.g. commit to heading hierarchy inside mega panels (H2/H3 group headings, already implied by FR-36-5's "group headings" starter-template guidance but not stated as an SEO-relevant contract), and state whether/how mega-panel descriptive text should be structured so an AI crawler summarising "what does this business offer" from the nav alone gets accurate group-level context (not just link text). This is genuinely differentiating — most competitor mega-menus are link-only with no structured group semantics, and the spec already does 90% of the work (group headings, one-item-per-group, explicit descriptions) without naming it as the AI-search lever it actually is.
