## Per-criterion (evidence → score → reasoning)

### 1. Completeness — 4.0
**Evidence:** Covers data contract (FR-36-1), block/CPT/plumbing roster (FR-36-2/3), desktop disclosure + mega (FR-36-4/5), drawer (FR-36-6), shared utility + honest no-JS scope (FR-36-7), responsive collapse + two modes (FR-36-8), decoupling + referential integrity/orphan lifecycle (FR-36-9/9a), a11y contract grounded in APG/WCAG (FR-36-10/11/12), no-inline + control-completeness (FR-36-13/14), MVP-first phasing with pre-registered gates (§7), a concrete live-QC gate absorbing Spec-34 baselines (§8), converter-emittability (FR-36-15), discoverability/schema/perf (FR-36-17), a resolved-decisions table (§12), and a cutover plan (FR-36-18). This is unusually complete for a pre-build spec.
**Reasoning:** Two behaviours are named but under-specified to buildable depth (priority+ overflow measurement + its no-JS behaviour; the mega-association spike *fallback*), and one dependency status is stated wrongly (labelCollapse). Complete enough to build Phase 1 with zero gaps; Phase 2 carries the residuals.

### 2. Clarity / testability — 4.0
**Evidence:** Plain-English §0 + per-FR precision + a §12 resolved-decisions table + a concrete, re-derivation-proof §8 gate (axe=0 on open drawer AND open mega; `elementFromPoint ≥ 10/10 Mama's, 18/18 Indus`; crawl assertion pre-JS; "mega renders at real position N not last"; integrity sweep; wp-perf-gate). A cold implementer is very unlikely to build the wrong *thing*.
**Reasoning:** Could a cold implementer build the wrong thing? Mostly no. Two clarity soft-spots: (a) FR-36-8's labelCollapse "UNBUILT" tag would send an implementer to rebuild something already live (see gap G1); (b) priority+ "items that don't fit fold into More" doesn't state the width-measurement mechanism or no-JS state, so two builders could diverge.

### 3. Internal consistency — 4.5
**Evidence:** The bar-vs-drawer dual rendering of `sgs/nav-menu` is reconciled in the roster row ("Inside a drawer: a vertical accordion/drill-down list") and FR-36-6. The native-association model (FR-36-5) is internally coherent with "no bespoke map/ID" (§12d) and "real-position render" (the walk reads the tree in order → old append-last bug structurally impossible). KIND=section/layout keeping `SGS_Container_Wrapper` (FR-36-5/13) is consistent with the D294 selector cited. Error-states (FR-36-9a) vs informational a11y (FR-36-12) are explicitly disambiguated.
**Reasoning:** No self-contradiction found. Half-point off only for the labelCollapse status line, which contradicts the spec's own "verified against live code" posture.

### 4. Buildability with zero rework — 3.5
**Evidence:** Phase 1 (MVP) is fully buildable from the text: `sgs/nav-menu` flat bar + burger → `sgs/nav-drawer` `<dialog showModal>` accordion + shared `store('sgs/nav')` utility + emit + crawl assertion. `SGS_Nav_Menu_Source` confirmed live (resolves `ref` → `parse_blocks` → tree; the primary-nav-*location* default in FR-36-1 is a small NEW addition, not yet in the class — low risk). Phase-2 rework risks are honestly flagged but real: (i) the **mega-association BUILD-SPIKE** (does the block-based Navigation editor surface a CPT as a linkable item + does the walk see `link.type==='sgs_mega_menu'`?) gates Bean's decision (d); the fallback ("classic-menu support / inline attach affordance") is *named, not specified*. (ii) `ResponsiveTriStateControl` is a genuine unbuilt P2 dependency. (iii) priority+ under-spec.
**Reasoning:** Not 4.0 because the central Phase-2 architecture choice rests on an unresolved spike whose fallback isn't buildable-detailed; if the spike fails, Phase 2 stalls. MVP-first phasing correctly quarantines this so it never blocks the demoable slice — which is why it's 3.5 not lower.

### 5. Consistency with locked companions — 4.0
**Evidence (verified):** FR-S9-4/5 (adaptive-nav + drawer a11y contract) exist and are the fold-source Spec 36 claims; DP2a "informational-only Notice, never a gate" (P2 §7 / line 188) matches FR-36-12; DP6 "schema-conformance not walker clone-in" (P2 §9 / lines 191, 623) matches FR-36-15; "converter is the pack factory" (P2 §306, flagged unbuilt) matches FR-36-3/15's "pack factory unbuilt" note; `templateLock:"contentOnly"` (P2 §133/835, Spec 35 Part G) matches FR-36-14; section/layout KIND keeps `SGS_Container_Wrapper` (D294) matches FR-36-13.
**Reasoning:** One real drift (G1): FR-36-8 + FR-36-14 tag **`labelCollapse` as "DESIGNED-but-UNBUILT"**, but Spec 17 FR-S9-8 records it **BUILT + LIVE (D331, 2026-07-14)** on `sgs/business-info` (+ `sgs/button` D332), and the universal per-tier **visibility** model (`device-visibility.php`, `sgsHideOn*`) is *also* live. ## Overall soundness score: **4.0 / 5** — would ship to a paying client after closing G1–G3; MVP slice is 4.5-grade, the Phase-2 residuals pull the whole down.

## Residual GAPS (value-graded: S/A/B/C = fixing changes the built outcome; D/F = cosmetic)

- **G1 — labelCollapse mis-stated as unbuilt (FR-36-8, FR-36-14). Grade B.**
  Spec 17 FR-S9-8 marks `labelCollapse` BUILT + LIVE (D331) and per-tier visibility (`device-visibility.php`) universal-live. FR-36-8 says "*both* are DESIGNED-but-UNBUILT — a hard build-order dependency the nav phase either waits for or builds." **Changes the built outcome:** an implementer would either wait on / rebuild a shipped control, or invent a parallel one (the R-31-9 anti-pattern the FR is trying to prevent). Fix: split the three — per-tier visibility = LIVE (use it); `labelCollapse` = LIVE on business-info/button, note "Nav iconOnly DESCOPED D331" so nav doesn't need it; only `ResponsiveTriStateControl` = unbuilt dep.

- **G2 — mega-association spike fallback under-specified (FR-36-5, §12d). Grade B.**
  The native-menu association is Bean's locked decision (d) but rests on an unproven spike; the fallback path is only named ("classic-menu support / an inline attach affordance"). **Changes the built outcome** if the spike fails: no buildable fallback = Phase-2 stall. Fix: spec the fallback to one buildable sentence each (which mechanism, where the item→post link is stored, how the walk detects it) so a fail-open path exists before build.

- **G3 — priority+ overflow mechanism + no-JS behaviour unspecified (FR-36-8, FR-36-7). Grade C.**
  "Items that don't fit fold into More ▾" omits the width-measurement mechanism (ResizeObserver? server-guess?) and the no-JS state (all items shown? static More?). FR-36-7's "crawlable without JS" honesty-scope doesn't cover priority+ specifically. **Changes the built outcome** for the priority+ path (Phase 2, operator-optional). Fix: one paragraph — measure-and-hide via the shared utility, no-JS renders the full inline list (priority attribute ignored), links stay crawlable.

- **G4 — FR-36-1 "primary nav menu location" default is new work presented as existing. Grade C.**
  `SGS_Nav_Menu_Source` fallback chain today is explicit-ref → header innerBlocks → **most-recent-published** (no `get_nav_menu_locations()` primary-location read). FR-36-1's default ("the menu assigned to the theme's primary nav menu location, else most-recent") adds a resolution tier the class lacks. Low blast (small addition) but flag it as build-work, not "already lives here."

- **G5 — `<details name>` exclusive-accordion no-JS fallback carries an unresolved ⚠VERIFY (FR-36-6). Grade C.**
  The no-JS drawer fallback depends on `<details name>` exclusive-accordion browser support (flagged ⚠VERIFY / "degrades to non-exclusive"). Acceptable degradation is stated, so it won't break the build — but the acceptance gate (§8) doesn't assert the no-JS drawer fallback actually renders navigable. Fix: add a no-JS drawer assertion to §8.

- **G6 — perf budget shared across header+nav is unallocated (FR-36-17, §10). Grade D.**
  `<100KB CSS / <50KB JS per page` is a *page* budget, but the interactivity store + mega panels + drawer share it with the header (Spec 17) and page content. No per-nav sub-allocation. wp-perf-gate catches a breach post-hoc; a pre-build sub-budget would prevent late rework. Cosmetic for MVP.

- **G7 — two `sgs/nav-menu` instances (bar + drawer) not called out as an operator/emit burden. Grade D.**
  The model correctly uses a separate nav-menu inside the drawer, but neither §8 nor FR-36-15 explicitly notes the converter must emit *two* nav-menu blocks + keep their `ref`s coherent. Implied, not stated. Low.

## Upgrade OPPORTUNITIES (what this spec could BECOME)

1. **Make the native-association model a stated competitive moat, not just a fragility-fix.** FR-36-5's "item carries the real WP post ID natively, no bespoke map" is architecturally *better* than Spectra/Kadence (which mint their own panel↔item bindings that break on export/import). The spec frames it defensively ("dissolves association+ID fragility"); it should also frame it as the **portability/clone moat** — a menu exported/re-imported/cloned keeps working because the association IS core WP data. This is the strongest "meets-and-exceeds" claim in the spec and it's buried. Pairs with FR-36-17's under-sold "server-rendered rich mega + AI-auto-generation" differentiator.

2. **Promote the shared `store('sgs/nav')` utility to a declared framework primitive with a versioned public contract.** FR-36-7 already designs it framework-reusable (future header-search / cart-flyout). One step further: define its public method surface (`open/close/toggle/trapFocus/inert/intentTimer`) as a *named, versioned* internal API in the spec, so the deferred WooCommerce mini-cart flyout (Phase 3) and header search (Spec 17) build against a stable contract instead of re-deriving disclosure/dialog logic. Turns a utility into the SGS overlay-behaviour standard.

3. **Add an AI-nav-generation emit contract now (thin), since the converter already writes the whole tree.** FR-36-15 already writes `wp_navigation` + mega CPT posts + the native link. Because association is native post-ID, an AI builder can *generate a whole rich nav from a sitemap* by emitting exactly these primitives — no bespoke glue. A 3-line "AI-generation target" sub-section would convert the spec from "clone an existing nav" to "synthesise a best-practice nav from site structure," which is the actual end-goal differentiator and costs almost nothing given the emit path already exists.

Verify at build before committing Phase 2 to the native-only path.
- **`<dialog showModal>` top-layer surviving a transformed sticky-header ancestor** (FR-36-6) is correct per spec, but iOS Safari body-scroll-lock + top-layer interaction has historically been quirky; §8's scroll-lock frame sweep + "incl. iOS fix" covers it, but treat the iOS drawer as the highest live-QC risk.
