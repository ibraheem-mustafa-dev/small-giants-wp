# Navigation Accessibility — Beyond a Basic WCAG Pass (P2.5 Phase 1)

**Date:** 2026-07-18
**Scope:** General web / WAI-ARIA navigation accessibility best practice BEYOND WordPress's baseline. WordPress core nav + WP-theme competitors were researched separately and are NOT re-covered here.
**Surfaces under design:** (1) desktop bar with simple **dropdowns**, (2) desktop **mega-menu**, (3) mobile off-canvas **drawer**.
**Method:** every claim below is grounded in a primary source (W3C WAI-ARIA APG, WCAG 2.1/2.2 Understanding docs, MDN, Adrian Roselli, Scott O'Hara/community), fetched and quoted 2026-07-18. Unverifiable claims are marked `[unverified]`.

---

## HEADLINE VERDICT — the disclosure/menubar/dialog split is CONFIRMED by primary sources

The already-established finding is **correct and reinforced by the APG itself**, not just secondary opinion:

- **Dropdowns and the mega-menu are DISCLOSURE widgets**, not `role="menu"/"menubar"`. Site navigation is a set of **links**, not an application command menu. The APG's own navigation example says so explicitly (quoted below).
- **The mobile drawer is a DIALOG** (`role="dialog"` + `aria-modal="true"`) with focus containment, mandatory Escape, and focus return.
- **These are incompatible contracts and must NOT be merged into one component.** A disclosure promises "a button that shows/hides content, Tab works normally, no trap." A dialog promises "focus is contained, Tab cannot leave, Escape is mandatory, background is `inert`." A screen reader announces them differently and the keyboard model is different. One component cannot honestly announce both.

This is the single most important architectural finding and it holds under primary-source scrutiny.

## A. THE CORE ARCHITECTURAL DECISION — disclosure vs menu/menubar vs dialog

### A1. Site nav dropdowns = the DISCLOSURE pattern, NOT `role="menu"/"menubar"`
- **What:** A dropdown/submenu toggle is a `<button>` with `aria-expanded` that shows/hides a section of links. It is NOT a `role="menu"` with `role="menuitem"` children.
- **Source:** APG Disclosure pattern — https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/ ; APG Disclosure **Navigation Menu** example — https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/examples/disclosure-navigation/
- **Direct quote (APG nav example):** *"Although this example uses the word 'menu' in the colloquial sense to refer to a set of navigation links, it does not use the WAI-ARIA menu role"* — because typical site navigation lacks the application-menu behaviours the menu role requires.
- **Why it matters:** `role="menu"` makes assistive tech promise a desktop-app keyboard model (arrow-only navigation, roving tabindex, "you are in a menu" mode). Site visitors get a list of links instead and can become trapped or confused. Wrong role = actively worse than no ARIA.
- **Tier:** **MUST-HAVE**

### A2. The APG menu/menubar pattern is for application menus, not link navigation
- **What:** `role="menubar"/"menu"/"menuitem"` models an OS-style command menu (File/Edit/View). It carries roving tabindex, arrow-key-only movement, Home/End, type-ahead, and Escape.
- **Source:** APG Menu & Menubar — https://www.w3.org/WAI/ARIA/apg/patterns/menubar/
- **Why it matters:** The pattern exists and even ships a "Navigation Menubar Example," but its behaviours are wrong for a marketing/site nav of links. Adopting it means implementing the *entire* roving-tabindex + arrow-key contract correctly, or shipping a broken half-menu (the common real-world failure).
- **Tier:** **MUST-HAVE** (as a "do not adopt for our surfaces" decision)

### A3. MDN explicitly warns against the menu role for site nav
- **What:** Confirmation from a second primary source that the menu role is reserved for composite widgets, not navigation.
- **Source:** MDN ARIA `menu` role — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles/menu_role
- **Direct quote:** *"When including a 'menu' for site navigation, do not use the `menu` role. Rather, for the main site navigation use the native HTML `<nav>` element or simply a list of links. The `menu` role should be reserved for composite widgets requiring focus management."*
- **Why it matters:** Independent corroboration of A1/A2 from the reference docs developers actually read.
- **Tier:** **MUST-HAVE**

### A4. Adrian Roselli — "Don't Use ARIA Menu Roles for Site Nav" (the canonical argument)
- **What:** The widely-cited essay establishing the application-menu vs link-navigation distinction.
- **Source:** https://adrianroselli.com/2017/10/dont-use-aria-menu-roles-for-site-nav.html (published 2017-10-23, updated 2025-08-11 — still current)
- **Direct quotes:** *"A properly-coded web page navigation menu needs no ARIA."* / The failure is *"a combination of patterns that do not use semantic HTML, a misunderstanding of ARIA, and not having practical experience with assistive technology."*
- **His four concrete harms of menu roles on site nav:** (1) AT announces complex keyboard commands users don't understand; (2) keyboard conflicts confuse users; (3) precise ARIA nesting is fragile and breaks easily; (4) semantic pollution when extra elements are announced.
- **Recommendation:** `<nav>` landmark + nested `<ul>/<li>` + plain `<a href>`; add disclosure buttons (`aria-expanded`) only for the show/hide toggles.
- **Tier:** **MUST-HAVE** (this is our design's governing rationale)

## B. CORRECT ARIA FOR EACH SURFACE

### B1. `aria-expanded` on the disclosure trigger
- **What:** The toggle button carries `aria-expanded="false"` when its panel is hidden, `"true"` when shown. This is the *only* state ARIA a dropdown/mega toggle needs.
- **Source:** APG Disclosure — https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/ ; APG nav example (above).
- **Why it matters:** It is the honest, minimal signal: "this button expands/collapses content." Screen readers announce "collapsed/expanded" and users know what to expect. No mode-change, no trap promise.
- **Tier:** **MUST-HAVE**

### B2. `aria-controls` linking trigger to panel
- **What:** The trigger's `aria-controls` references the `id` of the panel it shows/hides.
- **Source:** APG Disclosure + nav example (uses `aria-controls` by `id`).
- **Why it matters:** Programmatically associates button and content. Note: `aria-controls` support in screen readers is uneven `[partially unverified — support varies by AT]`, so treat it as a "should," not a load-bearing behaviour. The visible DOM order (panel immediately follows its trigger) is what actually carries most users.
- **Tier:** **SHOULD-HAVE**

### B3. `aria-current="page"` on the current-page link
- **What:** The nav link matching the current URL carries `aria-current="page"`. Other set-current values exist (`step`, `location`, `date`, `time`, `true`) but `page` is correct for site nav.
- **Source:** MDN aria-current — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Attributes/aria-current
- **Direct quote:** *"`page` — Represents the current page within a set of pages such as the link to the current document in a breadcrumb."*
- **Why it matters:** Communicates the visually-styled "you are here" state to screen-reader users, who otherwise get no equivalent to the highlight sighted users see. A basic WCAG pass usually ships the highlight but omits the ARIA.
- **Tier:** **MUST-HAVE**

### B4. `aria-haspopup` — when NOT to use it (and when it's acceptable)
- **What:** `aria-haspopup` announces "opens a popup." On a disclosure toggle for links it over-promises; on a genuine `role="menu"` it is required. For our disclosure-based nav we generally OMIT it and rely on `aria-expanded` alone.
- **Sources:** MDN aria-haspopup — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-haspopup ; community synthesis of Scott O'Hara's guidance (see G3).
- **Why it matters:** `aria-haspopup="true"` (or `"menu"`) tells AT to expect an application menu with arrow-key navigation. If the "popup" is just a list of links, the user is promised a menu-mode that doesn't exist and can get stuck. Pair `aria-haspopup` ONLY with an actual menu implementation. `aria-expanded` communicates the show/hide without the false menu promise.
- **Tier:** **MUST-HAVE** (specifically: the rule to *not* apply `aria-haspopup` to our link disclosures)

### B5. `<nav>` landmark + unique labels for multiple navs
- **What:** Wrap each navigation region in `<nav>` (implicit `role="navigation"`). When a page has more than one nav (primary bar, footer nav, mobile drawer, breadcrumb), each MUST be uniquely labelled.
- **Source:** APG Landmarks — navigation example — https://www.w3.org/WAI/ARIA/apg/patterns/landmarks/examples/navigation.html
- **Direct quote:** *"If a page includes more than one `navigation` landmark, each should have a unique label."* The APG demonstrates this with `aria-labelledby` pointing at a heading `id`; `aria-label` (e.g. `aria-label="Main"`, `aria-label="Footer"`) is the common alternative.
- **Why it matters:** Screen-reader users navigate by landmark ("navigation region"). Two unlabelled navs both announce as just "navigation" — indistinguishable. Our desktop bar and mobile drawer will BOTH be navs and MUST carry distinct labels (e.g. "Main" and, for the drawer dialog, a dialog label — see D-section).
- **Tier:** **MUST-HAVE**

### B6. Accessible name for icon-only toggles (hamburger, close)
- **What:** The hamburger button and drawer close button must have a text accessible name via `aria-label` (e.g. `aria-label="Open menu"` / `aria-label="Close menu"`) or visually-hidden text. Update the hamburger's expanded state via `aria-expanded`.
- **Source:** WCAG 4.1.2 Name, Role, Value (Level A) — https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html
- **Why it matters:** An icon-only button with no name announces as "button" with no purpose. This is one of the most common real-world nav failures.
- **Tier:** **MUST-HAVE**

## C. KEYBOARD INTERACTION CONTRACTS

### C1. Dropdown / mega — the disclosure keyboard contract (Tab-based, no roving tabindex)
- **What:** Focus reaches the toggle via **Tab** (normal document order). **Enter** and **Space** toggle it. Once open, **Tab** moves through the panel's links in DOM order and then out to the next top-level item — focus is NOT trapped. **Escape** closes the panel and returns focus to the toggle. Arrow keys / Home / End are OPTIONAL enhancements, not required.
- **Source:** APG Disclosure — https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/ ; APG nav example — https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/examples/disclosure-navigation/
- **Direct quote (APG nav example, core support):** *"Tab and Shift+Tab: Move keyboard focus among … buttons"* and links; *"Space or Enter: … activate"*; *"Escape: … closes open dropdowns"* and returns focus to the button. Arrow/Home/End are listed as **optional** enhancements.
- **Why it matters:** This is the WHOLE keyboard model for disclosures — deliberately simple, matching users' link-navigation expectations. Do NOT implement roving tabindex / arrow-only movement here (that's the menubar contract we rejected in A2).
- **Tier:** **MUST-HAVE** (Tab + Enter/Space + Escape-to-close). Arrow-key enhancement = **NICE-TO-HAVE**.

### C2. Escape closes the open dropdown/mega and returns focus to its trigger
- **What:** For disclosures, Escape is not *strictly* required by the base APG disclosure pattern, but the APG **navigation** example includes it and it satisfies SC 1.4.13 "Dismissible" for hover/focus content. So for us it is required.
- **Sources:** APG nav example (Escape in core support); WCAG 1.4.13 (C6 below).
- **Why it matters:** Gives keyboard and low-vision users a way to dismiss an unwanted panel without moving focus — and it is the mechanism that satisfies 1.4.13 Dismissible.
- **Tier:** **MUST-HAVE**

### C3. Drawer dialog — focus containment (the OPPOSITE of C1)
- **What:** When the mobile drawer opens: move focus INTO the dialog; **Tab** cycles only among the dialog's focusable elements (last → first, first ← last on Shift+Tab); focus cannot leave the dialog while open.
- **Source:** APG Dialog (Modal) — https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/
- **Direct quote:** *"Tab: Moves focus to the next tabbable element inside the dialog. If focus is on the last tabbable element inside the dialog, moves focus to the first tabbable element inside the dialog."*
- **Why it matters:** This is why the drawer cannot share a component with the dropdown — the dropdown must NOT trap (C1), the dialog MUST contain. Contradictory contracts.
- **Tier:** **MUST-HAVE**

### C4. Drawer dialog — Escape is MANDATORY
- **What:** Escape closes the drawer dialog. (For a dialog this is required, unlike the optional status it has in the base disclosure pattern.)
- **Source:** APG Dialog (Modal) — *"Escape: Closes the dialog."*
- **Why it matters:** A modal that can't be dismissed by keyboard is a keyboard trap (violates SC 2.1.2 No Keyboard Trap, Level A).
- **Tier:** **MUST-HAVE**

### C5. Drawer dialog — return focus to the trigger on close
- **What:** When the drawer closes, focus returns to the element that opened it (the hamburger button), unless that element no longer exists.
- **Source:** APG Dialog (Modal) — focus returns to the triggering element on close.
- **Why it matters:** Without focus return, keyboard focus lands at the top of the document (or nowhere), and the user loses their place. Extremely common real-world failure.
- **Tier:** **MUST-HAVE**

### C6. Hover-triggered dropdown/mega must satisfy SC 1.4.13 Content on Hover or Focus
- **What:** If a dropdown/mega opens on hover, it must be **Dismissible** (Escape without moving the pointer), **Hoverable** (pointer can move onto the panel without it vanishing — no gap that closes it), and **Persistent** (stays until hover/focus leaves, user dismisses, or it's invalid).
- **Source:** WCAG 2.1 SC 1.4.13 (Level AA) — https://www.w3.org/WAI/WCAG21/Understanding/content-on-hover-or-focus.html
- **Direct quotes:** Dismissible — *"A mechanism is available to dismiss the additional content without moving pointer hover or keyboard focus…"*; Hoverable — *"the pointer can be moved over the additional content without the additional content disappearing"*; Persistent — *"remains visible until the hover or focus trigger is removed, the user dismisses it, or its information is no longer valid."*
- **Why it matters:** The classic mega-menu bug: a diagonal pointer path from trigger to panel crosses dead space and the menu snaps shut before the user's cursor arrives. Low-vision/magnifier and motor-impaired users are worst hit. Must-fix for any hover-capable menu.
- **Tier:** **MUST-HAVE** (for any hover-triggered surface)

### C7. Never trap the keyboard in a disclosure; never leak focus from a dialog
- **What:** Two sides of SC 2.1.2 No Keyboard Trap (Level A): disclosures must let Tab pass through and out; dialogs must contain but ALWAYS provide Escape.
- **Source:** WCAG SC 2.1.2 — https://www.w3.org/WAI/WCAG22/Understanding/no-keyboard-trap.html
- **Tier:** **MUST-HAVE**

## D. FOCUS MANAGEMENT

### D1. `inert` on background content while the drawer is open
- **What:** When the drawer dialog is open, apply `inert` to everything else (the page behind it) so background links/buttons are removed from tab order AND the accessibility tree.
- **Source:** MDN `inert` — https://developer.mozilla.org/en-US/docs/Web/HTML/Global_attributes/inert
- **Direct quote:** *"An inert element, along with its descendants, gets removed from the tab order and accessibility tree"* and they *"are hidden from assistive technologies as they are excluded from the accessibility tree."* Baseline widely available since April 2023.
- **Why it matters:** `inert` is the modern, correct way to make a modal actually modal for screen-reader users — it prevents "escaping" the dialog by virtual cursor into background content, without the fragility of managing `aria-hidden` on siblings. Pair with `aria-modal="true"`.
- **Caveat (must design for):** MDN warns *"There is no visual way to tell whether or not an element or its subtree is inert"* — provide the visible scrim/overlay so sighted users see the background is disabled.
- **Tier:** **MUST-HAVE** (drawer). Do NOT use `inert` for dropdowns/mega — they are not modal.

### D2. `:focus-visible` for the focus ring (not bare `:focus`)
- **What:** Style focus indication with `:focus-visible` so the ring shows for keyboard/AT users but not on mouse click, and NEVER remove the outline without a replacement.
- **Source:** MDN `:focus-visible` — https://developer.mozilla.org/en-US/docs/Web/CSS/:focus-visible ; WCAG SC 2.4.7 Focus Visible (Level AA) — https://www.w3.org/WAI/WCAG22/Understanding/focus-visible.html
- **Why it matters:** `outline: none` with no replacement is the single most common keyboard-nav accessibility failure. `:focus-visible` gives a clean keyboard-only ring without the "ugly ring on click" objection designers use to justify removing it.
- **Tier:** **MUST-HAVE**

### D3. SC 2.4.11 Focus Not Obscured (Minimum) — sticky headers must not fully hide focus
- **What:** When a nav item receives focus, it must not be **entirely** hidden by author content (e.g. a sticky/fixed header covering the just-focused link as you Tab down).
- **Source:** WCAG 2.2 SC 2.4.11 (Level AA) — https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum.html
- **Direct quote:** *"When a user interface component receives keyboard focus, the component is not entirely hidden due to author-created content."*
- **Why it matters:** Our desktop bar is likely sticky. A sticky bar that overlaps focused content below it fails 2.4.11. Enhanced version (2.4.12, AAA) requires *no* obscuring at all. Focus Appearance (SC 2.4.13, AAA) sets a minimum ring size/contrast.
- **Tier:** **MUST-HAVE** (2.4.11 AA). 2.4.12 / 2.4.13 = **NICE-TO-HAVE** (AAA).

### D4. Skip link — "Skip to main content" bypassing the nav
- **What:** A visible-on-focus link at the very top of the tab order that jumps past the header/nav to `#main`.
- **Source:** WCAG SC 2.4.1 Bypass Blocks (Level A) — https://www.w3.org/WAI/WCAG22/Understanding/bypass-blocks.html
- **Why it matters:** Keyboard users otherwise Tab through every nav link on every page load. A mega-menu makes this dozens of tab stops. The skip link must become VISIBLE on focus (a hidden-but-focusable skip link that never appears is a frequent half-implementation).
- **Tier:** **MUST-HAVE**

### D5. Move focus into the drawer on open (not left on the page behind)
- **What:** On opening the drawer, programmatically move focus to the dialog (first focusable element, the close button, or the dialog container with `tabindex="-1"`).
- **Source:** APG Dialog (Modal) — *"When a dialog opens, focus moves to an element inside the dialog."*
- **Why it matters:** If focus stays on the (now-inert) hamburger or page behind, keyboard/AT users don't know the drawer opened and can't reach it.
- **Tier:** **MUST-HAVE**

## E. MOTION & SENSORY

### E1. `prefers-reduced-motion` — tame drawer slide / mega reveal animations
- **What:** Wrap slide-in / expand animations so that when the user requests reduced motion, large-movement transitions are removed or replaced with a simple opacity fade (or made instant).
- **Source:** MDN `prefers-reduced-motion` — https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion
- **Direct quote:** the setting signals *"the user prefers an interface that removes, reduces, or replaces motion-based animations"*; large scaling/panning *"can be vestibular motion triggers."*
- **Why it matters:** A drawer that slides across the whole viewport, or a mega-menu that scales/pans, can cause nausea/dizziness for users with vestibular disorders. Respecting the OS setting is a cheap, high-impact win. (Also relates to SC 2.3.3 Animation from Interactions, AAA.)
- **Tier:** **MUST-HAVE**

### E2. No motion-only or colour-only signalling of state
- **What:** "Open," "current page," and "has submenu" states must not be conveyed by animation or colour alone — pair with a shape/icon change (chevron rotates AND `aria-expanded` changes), text, or underline, plus the ARIA state.
- **Sources:** WCAG SC 1.4.1 Use of Color (Level A) — https://www.w3.org/WAI/WCAG22/Understanding/use-of-color.html ; SC 1.3.1 Info and Relationships (Level A).
- **Why it matters:** Colour-blind users and users who don't perceive animation still need to know which item is current and which items expand.
- **Tier:** **MUST-HAVE**

### E3. `forced-colors` / Windows High Contrast Mode support
- **What:** In forced-colors mode the browser overrides author `color`, `background-color`, `border-color`, and forces `box-shadow`/`text-shadow` to `none`. Nav dividers/borders/indicators drawn with box-shadow or background will VANISH. Add a `@media (forced-colors: active)` block using system colour keywords (`ButtonText`, `Canvas`, `Highlight`) and real `border`s where shadows were used.
- **Source:** MDN `forced-colors` — https://developer.mozilla.org/en-US/docs/Web/CSS/@media/forced-colors
- **Direct quote:** *"In forced colors mode, the values of the following properties are treated as if they have no author-level values specified"* (incl. border-color/background-color; box-shadow forced to `none`). Guidance: *"make small tweaks to improve usability or legibility,"* not a separate design.
- **Why it matters:** A mega-menu whose panel is distinguished only by a subtle shadow becomes invisible/indistinct in High Contrast Mode; a focus ring drawn with box-shadow disappears. Real Windows-user impact, routinely missed.
- **Tier:** **SHOULD-HAVE** (MUST-HAVE if any focus/boundary indicator relies on box-shadow)

### E4. `prefers-contrast` — honour increased-contrast requests
- **What:** When `prefers-contrast: more` is set, strengthen borders/focus rings/text contrast on nav UI.
- **Source:** MDN `prefers-contrast` — https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-contrast
- **Why it matters:** Complements forced-colors for users who want more contrast without full HCM. Low-cost enhancement.
- **Tier:** **NICE-TO-HAVE**

### E5. Non-text contrast for focus indicators and UI boundaries (SC 1.4.11)
- **What:** Focus indicators, submenu-affordance icons, and the boundary of interactive nav elements need ≥ 3:1 contrast against adjacent colours.
- **Source:** WCAG 2.1 SC 1.4.11 Non-text Contrast (Level AA) — https://www.w3.org/WAI/WCAG22/Understanding/non-text-contrast.html
- **Why it matters:** A pale-grey focus ring on white fails; a chevron icon that's too low-contrast can't be seen. Frequently missed because text contrast passes but the *indicator* contrast is never checked.
- **Tier:** **MUST-HAVE**

## F. TOUCH / TARGET SIZE

### F1. SC 2.5.8 Target Size (Minimum) — 24×24 CSS px (WCAG 2.2, AA)
- **What:** Every nav tap target (links, hamburger, close, submenu toggles) is at least 24×24 CSS px, OR has 24px of spacing clearance (a 24px-diameter circle centred on it doesn't intersect neighbours), OR meets an exception (inline/equivalent/user-agent/essential).
- **Source:** WCAG 2.2 SC 2.5.8 (Level AA) — https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html
- **Direct requirement:** minimum **24 by 24 CSS pixels**, with Spacing / Inline / Equivalent / User-Agent-Control / Essential exceptions.
- **Why it matters:** Tightly-packed mobile nav links and small icon buttons commonly fail. This is a NEW WCAG 2.2 AA criterion many "WCAG 2.1-compliant" navs miss entirely.
- **Tier:** **MUST-HAVE**

### F2. SC 2.5.5 Target Size (Enhanced) — 44×44 CSS px (WCAG 2.1, AAA)
- **What:** For important nav controls, aim for the stronger 44×44 CSS px target (matches Apple HIG and the existing SGS 44px baseline).
- **Source:** WCAG SC 2.5.5 Target Size (Enhanced), introduced WCAG 2.1, Level AAA — https://www.w3.org/WAI/WCAG22/Understanding/target-size-enhanced.html (carried into 2.2 Understanding docs)
- **Direct requirement:** *"The size of the target for pointer inputs is at least 44 by 44 CSS pixels except when: Equivalent, Inline, User Agent Control, [or] Essential."*
- **Why it matters:** SGS already commits to 44px (beats both SCs). Documented here so the framework's 44px default is understood as the AAA-level target, exceeding the 2.2 AA minimum.
- **Tier:** **SHOULD-HAVE** (SGS already targets this; keep it)

### F3. Adequate spacing between adjacent tap targets
- **What:** Even where individual targets meet size, provide gaps so users don't hit the wrong link — this is the "Spacing" route to passing 2.5.8.
- **Source:** SC 2.5.8 (Spacing exception) — same URL as F1.
- **Tier:** **SHOULD-HAVE**

## G. SCREEN-READER BEHAVIOUR (VoiceOver / NVDA / JAWS)

### G1. Disclosure announces honestly; menu role announces a mode
- **What:** A disclosure toggle is announced as e.g. "Products, button, collapsed" → on activation "expanded," and Tab continues normally through links. A `role="menu"` is announced as "menu" and puts JAWS/NVDA into "application/forms mode" expecting arrow keys — a different interaction the user must know.
- **Sources:** APG Disclosure & Menubar patterns (behavioural intent); Adrian Roselli (A4) on the AT-announcement mismatch.
- **Why it matters:** The disclosure announcement matches what site nav actually is (links). The menu announcement sets a false expectation that leads to the "stuck in the menu, arrow keys do nothing useful" real-world failure.
- **Tier:** **MUST-HAVE** (reinforces A1)

### G2. Common real-world nav failures to design against (checklist)
- **What / Source:** synthesised from the primary sources above:
  - Icon-only hamburger/close with no accessible name → announces "button" (fix: B6). [4.1.2]
  - `aria-expanded` never updated → screen reader says "collapsed" while panel is open (fix: B1).
  - Menu snaps shut before pointer reaches it → 1.4.13 Hoverable fail (fix: C6).
  - No focus return from drawer → user dumped at page top (fix: C5).
  - Background reachable behind "modal" drawer via virtual cursor → not truly modal (fix: D1 `inert` + `aria-modal`).
  - `outline:none` with no `:focus-visible` replacement → invisible keyboard focus (fix: D2).
  - Two unlabelled `<nav>`s → both announce "navigation" (fix: B5).
  - `role="menu"` applied but roving tabindex/arrow keys not implemented → half-broken menu (fix: don't use menu role, A1).
- **Tier:** **MUST-HAVE** (as a QA checklist)

### G3. `aria-haspopup` on a link creates a broken promise (Scott O'Hara)
- **What:** Putting `aria-haspopup` on a link/button that only reveals more links makes AT announce a menu that the markup doesn't deliver; users get instructions for a complex menu and find a plain link list, and can get "stuck."
- **Sources:** community synthesis of Scott O'Hara's guidance (see his accessible modal + ARIA writing) — https://github.com/scottaohara/accessible_modal_window ; corroborated by MDN aria-haspopup — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-haspopup ; Sarah Higley on 1.4.13-era popup content — https://sarahmhigley.com/writing/tooltips-in-wcag-21/
- **Note:** `[partially unverified]` — the specific Scott O'Hara blog post URL for this exact claim returned 404 on fetch; the principle is corroborated by MDN + APG + the disclosure-nav example, so treat the *principle* as verified and the specific-post attribution as unverified.
- **Tier:** **MUST-HAVE** (as the "don't add aria-haspopup to link disclosures" rule; see B4)

## H. DISCLOSURE vs MENUBAR vs DIALOG — DECISION TABLE (our three surfaces)

| Property | **Desktop dropdown** | **Desktop mega-menu** | **Mobile drawer** |
|---|---|---|---|
| **Correct pattern** | Disclosure | Disclosure | Dialog (modal) |
| **NOT this** | `role="menu/menubar"` | `role="menu/menubar"` | Disclosure (no containment) |
| **Container ARIA** | `<nav aria-label="Main">` + `<button aria-expanded>` | `<nav aria-label="Main">` + `<button aria-expanded>` | `role="dialog"` `aria-modal="true"` `aria-label`/`aria-labelledby` |
| **State attr** | `aria-expanded` on toggle | `aria-expanded` on toggle | (dialog open/closed via mount + `aria-modal`) |

**Confirmed: disclosure and dialog have incompatible contracts (no-trap vs contain, Escape-optional-base vs Escape-mandatory, no-inert vs inert). They MUST be separate components.** The mega-menu is a *bigger* disclosure, not a different pattern — it shares the dropdown's contract, so dropdown + mega can share a component; the drawer cannot join them.

## I. TOP a11y REQUIREMENTS A BASIC-WCAG-PASS NAV WOULD MISS

A "basic WCAG 2.1 AA pass" typically ships semantic links, colour contrast, and a keyboard-operable menu. Here is what it almost always MISSES — our differentiators:

1. **Right pattern, not just "some ARIA."** Basic navs sprinkle `role="menu"` (or nothing) and pass automated checks; they don't make the deliberate **disclosure-not-menu** choice. This is the headline gap. *(A1–A4)*
2. **1.4.13 Content on Hover or Focus** — hover mega-menus that fail Hoverable (snap shut on the diagonal path) or Dismissible (no Escape). Automated tools rarely catch this. *(C6)*
3. **`inert` for a truly-modal drawer** — most mobile menus leave the background reachable by virtual cursor; they look modal but aren't. *(D1)*
4. **Focus RETURN from the drawer to its trigger** — silently omitted in the majority of hand-built drawers. *(C5)*
5. **SC 2.4.11 Focus Not Obscured (2.2 AA)** — sticky header hiding the focused item; brand-new in 2.2 and widely un-implemented. *(D3)*
6. **SC 2.5.8 Target Size 24px (2.2 AA)** — a genuinely new 2.2 requirement a 2.1-era "compliant" nav never tested. *(F1)*
7. **`aria-current="page"`** — the highlight exists visually but the ARIA equivalent is missing, so screen-reader users get no "you are here." *(B3)*
8. **Unique labels on multiple `<nav>` landmarks** — primary + footer + drawer all announce as bare "navigation." *(B5)*
9. **`forced-colors` / High Contrast Mode** — shadow-only panel boundaries and box-shadow focus rings vanish; essentially never handled by a basic pass. *(E3)*
10. **`prefers-reduced-motion`** on the drawer slide / mega reveal — vestibular-safety, cheap, routinely skipped. *(E1)*
11. **`:focus-visible` (not removed `:focus`)** with ≥3:1 non-text contrast (1.4.11) on the ring itself — the ring passes "exists" but fails "visible enough." *(D2, E5)*
12. **Icon-only hamburger/close accessible names + live `aria-expanded`** — announces as unlabelled "button" in the common failure. *(B6)*
13. **Escape dismiss on desktop dropdowns** (not just the drawer) — usually only wired for the modal, missing on dropdowns where 1.4.13 needs it. *(C2)*
