# P2.5 Phase 1 — General web / UI-UX navigation best practice (the one input not yet gathered)

## 0. Why this exists (plain English)

The WP-competitor teardown (Stream B), the WP core-nav capability audit (Stream A), and the two client
menus (Stream C) were already gathered in the P2 design-gate §15.0. The ONE input missing was **what
"excellent navigation" looks like on the wider web** — the standards and patterns we would miss if we
only matched WordPress competitor themes. This doc closes that gap so Phase 2's inventory is complete.

## 1. Method + fact-check note (§15.7 discipline)

Four independent agents, each told to cite community-validated + authoritative sources and to treat every
recommendation as a hypothesis until attributed. **These are UX-standard claims, not WP-platform-capability
claims** — the platform-capability verification (what WP internals a bespoke block can/can't call) already
happened in the P2 council §15.2 and is NOT re-opened here. Two honesty flags carried forward from the
agents themselves:
- **Baymard's exact numbers are paywalled** — the "~28–36 links sweet spot / 50-link abandonment cliff /
  32% icon-discoverability drop" figures are secondary-source paraphrases. Treat as **directional**, validate
  against our own client sites before hard-coding any as a DB default.
- A few "transparent→solid on scroll" / "shrink-on-scroll" mechanics are **observed conventions** (Vercel/
  Linear/Apple), not NN/g-tested — flagged NICE, not MUST.

## 2. CONFIRMED rulings (convergent across ≥2 agents + the P2 council — treat as settled inputs)

| # | Ruling | Convergence |
|---|---|---|
| R1 | **Bar / dropdown / mega = the WAI-ARIA DISCLOSURE pattern** (`<button aria-expanded aria-controls>` + plain `<a>` links). NEVER `role="menu"`/`menubar`/`menuitem` — that's the application-menu contract (roving tabindex, arrow-key nav, type-ahead) that hijacks Tab and flips some screen readers into a different mode. | a11y + modern + mega agents + council (Roselli, Heydon, APG, USWDS) |
| R2 | **Mobile drawer = a DIALOG contract** (focus containment, ESC mandatory, background `inert`) — structurally INCOMPATIBLE with the disclosure contract. No single "nav panel + isModal flag" component; two distinct vanilla-JS behaviours sharing only the `aria-expanded`/`aria-controls` toggle mechanics. | a11y + council (this is the codebase's own view.js note, now externally corroborated) |
| R3 | **Click-to-open is the primary code path**, not hover. Hover has no touch equivalent, no keyboard equivalent, and unpredictable timing. Hover-with-safe-triangle is a **later desktop-polish enhancement layer** that must degrade to the click behaviour — GOV.UK chose click explicitly "because it works better for screen readers." | mega + a11y + modern + mobile agents (NN/g #13, Smashing, GOV.UK/DfE) |
| R4 | **`aria-current="page"` auto-wired on the active-trail link** (primary nav, breadcrumbs, scrollspy) — colour-only "you are here" is a WCAG 1.4.1 failure and invisible to screen readers. Must be wired by the block, never left to the operator. | all 4 agents (MDN, APG) |
| R5 | **One menu data source drives BOTH desktop render AND the mobile fold** — desktop mega-menu columns → drawer accordion sections; never a second hand-authored mobile menu. Directly serves the cloner's "reproducible" requirement and the R-31 "no silent drop" doctrine. | mega + mobile agents (Baymard) — matches FR-S9-4 one-source rule already locked |
| R6 | **The transform-ancestor trap is real and standard CSS**: a hide-on-scroll header uses `transform: translateY()`, which creates a containing block that silently converts a `position:fixed` drawer/overlay into ancestor-relative → clips/mispositions. Fix = **body-level portal** (runtime re-parent). This was §15.3's added requirement; now independently confirmed with primary CSS-spec sources. | mobile + modern agents + council (mtsknn.fi, aworkinprogress.dev) |
| R7 | **Never hide primary nav behind a hamburger on DESKTOP** — NN/g's 179-participant study: >20% lower discoverability, 39% slower task time. Hamburger is a mobile last-resort overflow container, not a default. | mobile + modern agents (NN/g) |
| R8 | **`prefers-reduced-motion` gates every drawer/panel transition from day one** (not bolted on) — collapse slide/fade to instant. Cheap, WCAG 2.3.3, already a Spec 35 Part-C rule. | a11y + modern + mobile agents |

## 3. The consolidated MISSING list (deduplicated across all 4 agents, ranked)

"MISSING" = a standard/feature a team copying only WP competitor themes would likely skip. Mapped to the
surface/client it serves. This is the Phase-2 NEED-side backbone.

### MUST-HAVE (definition-of-done for the nav rebuild)
| Item | Surface | Why (one line) |
|---|---|---|
| Disclosure pattern (R1) — `button[aria-expanded][aria-controls]` + real `<a>` links, no `role=menu` | bar/dropdown/mega | Baseline correctness; WP core nav's nested `<ul>` hover is a WCAG 1.4.13 + 2.1.1 failure at the starting line |
| Dialog contract for the drawer (R2) — focus containment via `inert`, ESC-to-close, focus-return-to-toggle | drawer | WP nav blocks toggle a class but don't manage focus → keyboard user tabs into content behind the open drawer |
| Click-primary interaction (R3) with full keyboard (Enter/Space/Esc) | all | Touch + keyboard parity by construction |
| `aria-current="page"` auto-wired (R4) | bar/breadcrumb/scrollspy | Silent screen-reader gap otherwise |
| Group headings mandatory inside every mega panel (never a bare list dressed as columns) | mega | NN/g's #1 named mega anti-pattern; directly measurable |
| One-source → dual-render fold (R5): mega columns → drawer accordion sections, no content dropped | mega↔drawer | Cloner reproducibility + R-31 no-silent-drop |
| Body-portal so drawer/overlay survives a transformed header ancestor (R6) | drawer/overlay | Invisible-until-hide-on-scroll-added CSS trap |
| Content-driven collapse breakpoint measured against the block's ACTUAL rendered nav width, not a hard-coded device px | bar→drawer | A cloner reproducing arbitrary drafts can't inherit a donor theme's 768px item-count assumption (NN/g) |
| CLS-safe sticky: `position:sticky` + reserved height + sentinel/IntersectionObserver triggers — never a JS `fixed`-swap that reflows | header | Single biggest perf-regression risk in "just add sticky.js" |
| WCAG 2.2 **2.4.11 Focus Not Obscured** — a sticky header must never fully hide a focused nav item | header+bar | Keyboard user loses focus under the sticky bar |
| WCAG 2.2 **2.5.8 Target Size** (keep our 44px bar) on every link/toggle/close | all | Mobile-first, non-technical clients |
| `aria-expanded`+`aria-controls` on toggle, `aria-hidden` on closed panel; accessible names on icon-only toggles | all | Competitor themes animate visually but never update the a11y tree |
| Cart badge accessible live count: `role="status" aria-live="polite" aria-atomic="true"` around the count node | header CTA slot (Mama's + shops) | WCAG 4.1.3 — the gap most WooCommerce themes leave open (visual badge, no announcement) |

### SHOULD-HAVE (strong meet-and-exceed; sequence after MUST correctness)
| Item | Surface | Why |
|---|---|---|
| Top-level item = a real clickable link with a genuine landing page, independent of whether it also toggles a panel | bar | Removes the link-vs-toggle ambiguity; graceful no-JS fallback (Roselli/APG hybrid) |
| Distinct, ≥44px chevron/caret affordance separating "expands" from "navigates" | bar/dropdown | Users can't tell clickability from hover state alone |
| Explicit tunable hover-delay constants IF hover-enhancement ships (open ~300–500ms settle, close ~500ms grace) — separate from "animation speed" | mega (hover layer) | 60% of hover menus get this wrong; competitor plugins conflate open-speed with intent-delay |
| Two-row header as first-class composable slots (utility/announcement row + nav row) with INDEPENDENT sticky/dismiss state machines | header (Indus 2-row) | Naive "stack two sticky elements" produces a CLS event the moment the announcement bar is dismissed |
| Breadcrumb-in-drawer / persistent parent-label for drill-down submenus | drawer | Prevents the "lost 3 levels deep" failure (Baymard/ASOS) |
| Priority+/overflow as an option so low-item-count menus stay visible instead of defaulting straight to hamburger | bar→collapse | NN/g's ≤4-items-stay-visible rule is rarely implemented in WP |
| Command palette (Cmd/Ctrl-K) — ARIA combobox, focus-trapped, Esc-closes, never the ONLY nav route | header (differentiator) | Increasingly expected on content-rich/app-like sites; WP themes essentially never ship one |
| Sticky sub-nav / in-page jump nav auto-derived from H2 heading blocks (with `scroll-margin-top`) | long pages | Near-zero maintenance if auto-generated; WP core has no equivalent |
| `:focus-visible` (≥3:1) not bare `:focus` | all | Keyboard rings without mouse-click rings |
| `aria-haspopup` used ONLY when its value matches the popup's actual role, else omitted (disclosure) | mega/dropdown | Mismatched value = misannouncement across NVDA/JAWS/VoiceOver (Matuzovic) |
| Backdrop scrim, full-viewport, tap-to-close, body-level stacking | drawer | Present in Foundation/UIkit; often skipped for "X only" in bespoke WP blocks |
| Hard link-count / column ceiling as an EDITOR-FACING informational notice (NOT a gate — see §4 decision D5) | mega editor | Baymard bounce cliff; a non-coder client keeps adding items until warned |
| Skip-to-content link as the first focusable element, NOT wrapped inside `<nav>` | header | GOV.UK build rule |

### NICE-TO-HAVE (real but narrow / data-dependent — later waves)
| Item | Why it's NICE not MUST |
|---|---|
| Hide-on-scroll header (hide-down/reveal-up) as a configurable option | Real NN/g win but adds the body-portal interaction complexity — later toggle |
| Transparent→solid / shrink-on-scroll header states | Observed convention (Apple/Vercel/Linear), not NN/g-tested |
| Amazon "safe-triangle" trajectory tracking | ~~NICE~~ **→ PROMOTED to v1 MUST (Bean 2026-07-19):** hover IS the default open behaviour (DP3), so diagonal-dismissal is a real defect and the safe-triangle is its fix, not optional polish |
| Scroll/reading-progress indicator | Long-form content only; cheap once the sticky scroll-listener exists |
| Back-to-top control | Situational; Nielsen himself sceptical of universal value |
| Imagery/icon slots in mega (product thumbnails, "skip if none" state) | Text-only mega is fully functional; guard against decorative-filler anti-pattern |
| One capped promo/featured-content slot per mega panel (SaaS pattern) | Real conversion value but easy scope-creep; cap at exactly one |
| Personalised mega content ("recently viewed") | Requires a behavioural-data pipeline the framework doesn't have — backlog, flag as a data dependency |

## 4. OPEN design decisions for Phase 3 (genuine forks — decide from the full inventory, not here)

These are the real architecture choices the research SURFACES but does not settle. Each carries the
evidence + the SGS-policy constraint that bears on it. **Do not pre-commit — Phase 3 decides from the
inventory + council + QC.**

- **D1 — Interaction default: click-only single path, or click-with-hover-enhancement-layer.** Research
  leans click-primary (R3). Hover done properly (Stripe/Amazon safe-triangle + timing) is strictly more
  code and still needs the click/keyboard path underneath. Fork: ship click-only v1 (simplest, WCAG-parity
  by construction) and add hover as a later opt-in layer, OR build the hover layer in v1.
- **D2 — Drawer: modal or non-modal dialog.** The a11y research prescribes `role="dialog" aria-modal="true"`.
  But **our current Spec 34 deliberately uses a NON-modal dialog** (no `aria-modal`) so the header top row
  stays live as the close control (Bean's GOV.UK-disclosure model). These conflict. The non-modal choice is
  a legitimate, Bean-locked design (header-visible drawer) — but it means "modality-minus-header" must be
  provided by the selective-freeze mechanism, not by `aria-modal`. **Reconcile explicitly in Phase 3/4** —
  this is a design decision, not a defect. (Note: Bean's guardrail — "zero influence from the current
  architecture" — applies to block COUNT/shape, but the header-visible-drawer INTERACTION MODEL is a
  Bean-locked UX decision from Spec 34, not architecture baggage; keep the model, re-derive the blocks.)
- **D3 — Mobile multi-level: accordion vs drill-down.** Literature is genuinely mixed; the deciding factor
  is IA depth, not taste. Research recommendation: **default accordion for ≤2 levels, auto-promote to
  drill-down at 3+ levels** (accordion = less state, easier to keep accessible; drill-down needs a
  breadcrumb-in-drawer). Decide whether we build both or ship accordion-only v1.
- **D4 — Body-portal cloner-emit question (R-31-7 design-gate flag).** Runtime body-portal means the
  drawer's SAVED markup position ≠ its RENDERED DOM position. Fork: does the converter emit the drawer
  markup at document-root (matching runtime) or nested-and-re-parented-by-JS (standard: keep authoring
  semantic, portal at runtime, tie via `aria-controls`)? Standard practice = the latter, but the
  save/render divergence needs a converter-emittability design-gate before build (Spec 31/33 Part 2).
- **D5 — Link-count / column ceiling: hard gate or informational notice.** Baymard evidence supports a
  ceiling. **BUT Bean locked (2026-07-18) that a11y/validation feedback is INFORMATIONAL-only, never a
  gate/auto-enforce.** → The consistent choice is a **soft editor-facing notice**, and the numeric
  thresholds stay directional (validate on our sites) rather than a hard DB default. Recorded so Phase 3
  doesn't accidentally build a gate against the locked policy.
- **D6 — Overlay vs push drawer.** Research strongly favours **overlay** (push transforms the whole content
  wrapper → a second-order transform-ancestor trap cascading to every fixed element inside content).
  Near-decided by evidence, but note it for the record.
- **D7 — Command palette & hide-on-scroll: v1 or later wave.** Both are meet-and-exceed differentiators
  (SHOULD/NICE). Decide sequencing against the "no gold-plating before structural correctness" instinct.
