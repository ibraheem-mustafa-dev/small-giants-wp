# P2.5 Phase 1 — Mobile Navigation, Off-Canvas Drawers & Emerging Nav Patterns

**Doc type:** research report (best-practice, externally sourced)
**Date:** 2026-07-18
**Author:** navigation-UX researcher (subagent)
**Scope:** GENERAL WEB / UI-UX navigation best practice BEYOND WordPress — mobile + off-canvas + emerging only.
**Explicitly OUT of scope (already covered by other inputs, not re-researched here):** WordPress core Navigation block baseline; WP-theme competitors (Kadence, Spectra, GenerateBlocks, Elementor, Bricks).

## How to read this
Each entry: **(1) pattern → (2) plain-English what → (3) source → (4) why it matters → (5) tier.**
Tiers: **MUST** (table stakes / ship-blocker), **SHOULD** (strong best practice, ship soon), **NICE** (polish / differentiator), **EMERGING** (bleeding-edge — evaluate maturity before building).

**Grounding targets** (the two menus we must reproduce, referenced throughout):
- **Restaurant** — flat 5-item bar + featured item + cart badge.
- **B2B trade** — 7-item bar, 3 dropdowns, 1 mega menu, two-row header; on mobile the whole lot collapses into a drawer (accordion or drill-down).

## 1. The hamburger debate — visibility of the toggle

### 1.1 Hidden navigation measurably hurts engagement
- **What:** Putting primary nav behind a hamburger (vs. showing links) reduces discovery, slows task completion, and lowers satisfaction.
- **Source:** NN/g, *Hamburger Menus and Hidden Navigation Hurt UX Metrics* — https://www.nngroup.com/articles/hamburger-menus/ (179-participant study, Dec 2015, run with WhatUsersDo across 6 sites, mobile + desktop).
- **Why it matters:** NN/g's finding is that hiding the menu roughly **halves discoverability** and increases perceived task difficulty. For a website-builder this means: never *default* a site's whole nav to a hamburger on viewports where the links would fit. Show links when you can; hide only what you must.
- **Tier:** MUST (as a default-behaviour principle).

### 1.2 "Obvious always wins" — visible nav lifts real product metrics
- **What:** Multiple apps that moved from a hamburger to a visible bottom tab bar saw large engagement gains.
- **Source:** Luke Wroblewski, *Obvious Always Wins* — https://www.lukew.com/ff/entry.asp?1945= (cites Redbooth: session time **+70%**, daily active users **+65%**; Facebook engagement rose moving hamburger → bottom tab bar).
- **Why it matters:** Independent corroboration of 1.1 from production A/B data, not lab tasks. Strengthens the case for a *visible* primary-nav mode in the block set, especially on mobile.
- **Tier:** MUST (informs default), SHOULD (bottom-tab option).

### 1.3 The hamburger icon IS recognised — but label it and don't box it
- **What:** Users now generally recognise the ☰ icon as "menu", but a text label helps novices and decorative borders/frames cause misidentification.
- **Source:** NN/g, *The Hamburger-Menu Icon Today: Is It Recognizable?* — https://www.nngroup.com/articles/hamburger-menu-icon-recognizability/ ("Label the icon *Menu*… A label is especially helpful for less-experienced users"; "Avoid extra styling or borders… can cause users to mistake the icon for something else").
- **Why it matters:** Cheap, universal wins for the toggle: expose an optional `MENU` text label attribute (default on for older/less-tech audiences), and do NOT wrap the glyph in a bordered box by default. Directly relevant to the B2B trade audience (older, less tech-comfortable buyers).
- **Tier:** MUST (label option + no default border).

### 1.4 Toggle count threshold — show links up to ~4–5, hide beyond
- **What:** If top-level links number ≤4 (NN/g) / ≤5 (fits a tab bar), display them; only beyond that is hiding justified.
- **Source:** NN/g, *Hamburger Menus…* (≤4 → show as visible links) — https://www.nngroup.com/articles/hamburger-menus/ ; NN/g, *Basic Patterns for Mobile Navigation* ("If your site has more than 5 options, it's hard to fit them in a tab or navigation bar") — https://www.nngroup.com/articles/mobile-navigation-patterns/
- **Why it matters:** Gives an automatable rule for the cloning pipeline: **restaurant flat-5** stays visible (tab/bottom bar); **B2B 7-item** exceeds the bar budget → collapse into a drawer on mobile. The block should expose a "collapse-below-N" breakpoint, not force one behaviour.
- **Tier:** MUST.

### 1.5 Alternatives to a bare hamburger
- **What:** Options that beat an opaque hamburger for discoverability: a bottom tab bar (1.2 / §4), the Priority+ / "More" overflow pattern (§6.1), a labelled "Menu" button, or a persistent search entry.
- **Source:** NN/g, *Basic Patterns for Mobile Navigation* — https://www.nngroup.com/articles/mobile-navigation-patterns/ ; Brad Frost, *Revisiting the Priority+ Pattern* — https://bradfrost.com/blog/post/revisiting-the-priority-pattern/
- **Why it matters:** Positions the hamburger as one tool among several rather than the only mobile answer — a genuine differentiator vs. WP-competitor block sets that ship "hamburger, full stop".
- **Tier:** SHOULD.

## 2. Off-canvas drawer mechanics

### 2.1 Slide-in geometry (edge, direction, width)
- **What:** Drawer slides in from an edge (left/right most common; top for "notification-style"; full-screen overlay for content-heavy sites). Partial-width leaves a visible backdrop; full-screen maximises tap targets.
- **Source:** dbushell, *The Final Off-Canvas Navigation?* — https://dbushell.com/2021/06/17/css-off-canvas-responsive-navigation/ ; Flowbite drawer docs — https://flowbite.com/docs/components/drawer/
- **Why it matters:** Geometry should be an *attribute* (side + width), not a hard-coded choice — restaurant may want a compact right drawer beside a cart; B2B may want a full-screen drawer to hold 7 items + nested mega content. Right-edge drawers also sit closer to the thumb for right-handed cradle grips (§4).
- **Tier:** MUST (side + width as attributes).

### 2.2 Backdrop / scrim
- **What:** A dimmed, click-to-dismiss overlay behind the drawer that separates it from page content and captures outside taps.
- **Source:** Flowbite drawer docs — https://flowbite.com/docs/components/drawer/ ; MDN, *Using the Popover API* (native `::backdrop` for top-layer elements) — https://developer.mozilla.org/en-US/docs/Web/API/Popover_API/Using
- **Why it matters:** The scrim is both a visual affordance and a dismissal target (users expect tapping outside to close). With the Popover/`<dialog>` route you get `::backdrop` for free (§5.1).
- **Tier:** MUST.

### 2.3 Focus containment (focus trap) + return focus
- **What:** While the drawer is open, keyboard focus must stay inside it (Tab wraps first↔last), and on close focus returns to the toggle that opened it. The rest of the page should be made inert.
- **Source:** Kahoot Tech, *Focus on accessibility — accessible menus and modals* — https://kahoot.com/tech-blog/focus-on-accessibility-accessible-menus-and-modals/ (describes wrap behaviour + the `inert` attribute on sibling `header/main/footer`); MDN `inert` — https://developer.mozilla.org/en-US/docs/Web/HTML/Global_attributes/inert
- **Why it matters:** WCAG 2.1 keyboard-operability + focus-order. A drawer that leaks focus to the page behind it is a hard a11y failure. Native `<dialog showModal>` / popover give trapping + inert-ish behaviour without hand-rolled JS (§5.1) — strongly preferred for a vanilla-JS-only framework.
- **Tier:** MUST.

### 2.4 Body-scroll-lock (and letting the drawer scroll)
- **What:** When the drawer is open the page behind must not scroll; the drawer's own content may scroll if it overflows. Some libraries expose an opt-in "keep body scrollable" flag.
- **Source:** Bootstrap 5 Offcanvas (`data-bs-scroll` toggles body scroll while open) — https://canvastemplate.com/blog/bootstrap-5-offcanvas-menu ; Mantine Drawer scroll-lock props — https://mantine.dev/core/drawer/
- **Why it matters:** "Scroll bleed" (background scrolls under an open drawer) is one of the most common mobile nav bugs and a jarring UX failure. Note the iOS Safari gotcha: `overflow:hidden` on `<body>` alone is unreliable; frameworks often also fix position. Must be handled in the block's view.js.
- **Tier:** MUST.

### 2.5 Explicit close affordance (never gesture-only)
- **What:** A visible close control (× button) plus `Esc` key support — in addition to any swipe-to-close gesture and backdrop tap.
- **Source:** TestParty, *Mobile Patterns that Break (and Make) Accessibility* — https://testparty.ai/blog/mobile-accessibility-patterns ("bottom sheets often dismiss on swipe-down gesture only… users who can't swipe need an explicit close button or Escape key support").
- **Why it matters:** Gesture-only dismissal excludes motor-impaired and screen-reader users (WCAG 2.5.1 Pointer Gestures / 2.1.1 Keyboard). The × + `Esc` are non-negotiable; the gesture is an *enhancement on top*.
- **Tier:** MUST.

### 2.6 Drag / swipe-to-close gesture
- **What:** Let users drag the drawer off-screen or swipe to dismiss (and optionally swipe from edge to open), tracking the pointer so the panel follows the finger.
- **Source:** LogRocket, *Designing swipe-to-delete and swipe-to-reveal interactions* — https://blog.logrocket.com/ux-design/accessible-swipe-contextual-action-triggers/ ; Material Design gestures — https://m2.material.io/design/interaction/gestures.html
- **Why it matters:** Feels native and is expected on mobile — but MUST be an enhancement over 2.5, never the only path. Precise swipes exclude users with tremor/limited dexterity. Provide a generous drag threshold and snap-back if the drag is incomplete.
- **Tier:** SHOULD (enhancement), gated on 2.5 being present.

### 2.7 Animation + `prefers-reduced-motion`
- **What:** Animate the slide/backdrop-fade, but disable or drastically reduce the transform-based motion when the user has requested reduced motion (swap slide for a simple fade, or none).
- **Source:** W3C WCAG technique C39, *Using the CSS `prefers-reduced-motion` query to prevent motion* — https://www.w3.org/WAI/WCAG21/Techniques/css/C39 ; relates to WCAG 2.3.3 (Animation from Interactions). Overview: Pope Tech, *Design accessible animation and movement* — https://blog.pope.tech/2025/12/08/design-accessible-animation-and-movement/
- **Why it matters:** Large sliding panels are exactly the "big movement triggered by interaction" that vestibular-sensitive users need suppressed. A `@media (prefers-reduced-motion: reduce)` block is trivial to ship and is a WCAG-aligned quality signal. Keep motion values in CSS/attributes, never inline (matches SGS no-inline rule).
- **Tier:** MUST.

### 2.8 ARIA wiring for the toggle ↔ drawer
- **What:** Toggle carries `aria-expanded` (true/false) + `aria-controls` pointing to the drawer's id; drawer carries `aria-labelledby` (or `aria-label`) naming it, and an appropriate role (`dialog` for modal drawers).
- **Source:** Flowbite drawer docs — https://flowbite.com/docs/components/drawer/ ("apply `aria-controls` to the trigger and `aria-labelledby` to the drawer… screen readers announce this label when the panel opens").
- **Why it matters:** Without this, screen-reader users get an unlabelled, stateless toggle. This is cheap, deterministic, and fully automatable by the cloning pipeline.
- **Tier:** MUST.

## 3. In-drawer submenu models — accordion vs drill-down

### 3.1 Subnav pattern by subcategory volume (the decision rule)
- **What:** NN/g's algorithm: **<6** subcategories across all primary categories → accordion / inline submenu; **6–15** for some categories → section menu; **>15** per category → category landing page.
- **Source:** NN/g, *Mobile Subnavigation* — https://www.nngroup.com/articles/mobile-subnavigation/
- **Why it matters:** A concrete, automatable branch for the pipeline. The **B2B trade** site (3 dropdowns + 1 mega) likely falls in the accordion→section-menu band; a genuinely large mega (>15 children) should degrade to a linked category page rather than an unwieldy in-drawer tree.
- **Tier:** MUST (as sizing logic).

### 3.2 Accordion (expand-in-place) — best for shallow hierarchies
- **What:** Parent taps expand children inline within the same panel; multiple levels stack vertically. Best for 2–3 levels.
- **Source:** NN/g, *Mobile Subnavigation* — https://www.nngroup.com/articles/mobile-subnavigation/ ; LogRocket, *Designing effective accordion UIs* — https://blog.logrocket.com/ux-design/accordion-ui-design/ ; Smashing, *Designing Navigation for Mobile* ("accordions are likely to perform better" than slide-in) — https://www.smashingmagazine.com/2022/11/navigation-design-mobile-ux/
- **Why it matters:** Keeps context visible (parent stays on screen), no page-load, easy back-out (just collapse). Smashing's testing found accordions outperform slide-in submenus. Default choice for most SGS client menus.
- **Tier:** MUST (default in-drawer model).

### 3.3 Drill-down / section menu (slide to child level) — for deep trees
- **What:** Tapping a parent slides the whole panel to a child level with a back control; only one level shown at a time.
- **Source:** NN/g, *Mobile Subnavigation* (section menus for 6–15 subcats; note NN/g's caution that *sequential/drill-down menus risk disorientation and confusion with the browser Back button*) — https://www.nngroup.com/articles/mobile-subnavigation/ ; Boundev multi-level menu guide — https://www.boundev.ai/blog/multilevel-menu-design-ux-guide
- **Why it matters:** Wins when the tree is deep or each level is long (an accordion would produce an exhausting scroll). But it hides context, so it needs a clear back affordance and a visible breadcrumb/parent label. Offer it as an attribute-selected mode, defaulting to accordion.
- **Tier:** SHOULD (deep-tree option).

### 3.4 Keep the navigation stack (breadcrumb of levels)
- **What:** In a drill-down, retain and show the path of levels the user traversed so they can jump back multiple steps, not just one.
- **Source:** Smashing, *Designing Navigation for Mobile* ("keep the navigation stack of the levels that users browsed through to allow for quick jumps") — https://www.smashingmagazine.com/2022/11/navigation-design-mobile-ux/
- **Why it matters:** Directly mitigates NN/g's disorientation warning (3.3). The difference between a good drill-down and a frustrating one.
- **Tier:** SHOULD (paired with 3.3).

### 3.5 Two-icon disclosure state (open vs closed)
- **What:** Use exactly two clear states for an expandable row — e.g. chevron-down (closed) / chevron-up (open), or +/−.
- **Source:** Smashing, *Designing Navigation for Mobile* ("you'll be able to achieve better results with just two icons, indicating whether an accordion is open or not") — https://www.smashingmagazine.com/2022/11/navigation-design-mobile-ux/
- **Why it matters:** Avoids ambiguous single-glyph states. Pair the visual state with `aria-expanded` (2.8) so it's conveyed non-visually too.
- **Tier:** SHOULD.

### 3.6 Separate the parent link from the expander
- **What:** A parent that is both a page AND has children needs two distinct targets: tapping the label navigates to the parent page; tapping a separate chevron area expands the children.
- **Source:** Smashing, *Designing Navigation for Mobile* — https://www.smashingmagazine.com/2022/11/navigation-design-mobile-ux/ ; corroborated by Baymard mobile-nav benchmark theme ("allow users to access broad parent category pages while also going deep into sub-subcategory levels") — https://baymard.com/mcommerce-usability/benchmark/mobile-page-types/navigation-menu
- **Why it matters:** A classic mobile-menu failure is a parent whose only tap target expands the submenu, leaving the parent's own page unreachable (or vice-versa). Both the restaurant's featured item and any B2B category-with-landing-page need this split-target treatment.
- **Tier:** MUST.

## 4. Bottom navigation, tab bars & thumb reachability

### 4.1 The thumb zone (ergonomic foundation)
- **What:** One-handed users reach the bottom-centre of the screen comfortably (green zone); mid-screen sides need a stretch (yellow); top corners are awkward/two-handed (red). Primary actions belong in the green zone.
- **Source:** Smashing, *The Thumb Zone: Designing For Mobile Users* — https://www.smashingmagazine.com/2016/09/the-thumb-zone-designing-for-mobile-users/ ; original observational data: Steven Hoober via UXmatters, *How Do Users Really Hold Mobile Devices?* — https://www.uxmatters.com/mt/archives/2013/02/how-do-users-really-hold-mobile-devices.php (**1,333 observations**; **49%** one-handed, **36%** cradling, **15%** two-handed).
- **Why it matters:** The single strongest argument for a *bottom* nav option on mobile. Top-anchored hamburgers sit in the red zone — reachable but not comfortable one-handed. As phones grow, this gap widens.
- **Tier:** MUST (as a design principle informing placement options).

### 4.2 Bottom tab bar — visible primary nav in the thumb zone
- **What:** A fixed row of 3–5 top-level destinations pinned to the bottom edge, each with icon + short label; the active tab is visually marked.
- **Source:** NN/g, *Basic Patterns for Mobile Navigation* (tab bars suit sites with ≤5 options) — https://www.nngroup.com/articles/mobile-navigation-patterns/ ; Luke Wroblewski, *Obvious Always Wins* (bottom-tab migration lifted engagement) — https://www.lukew.com/ff/entry.asp?1945=
- **Why it matters:** Ideal for the **restaurant flat-5** menu: 5 items fit a bar exactly, sit in the thumb zone, and the cart can be one of the tabs (with a badge). This is a pattern most WP-competitor block sets *don't* offer as a first-class option — a clear differentiator.
- **Tier:** SHOULD (offer as a mobile primary-nav mode).

### 4.3 Item-count ceiling (3–5) + labels, not icon-only
- **What:** Cap bottom-bar destinations at 3–5; always pair icons with text labels (icon-only bars fail recognisability).
- **Source:** NN/g mobile patterns (≤5) — https://www.nngroup.com/articles/mobile-navigation-patterns/ ; icon-label pairing echoes the hamburger label finding (1.3) — https://www.nngroup.com/articles/hamburger-menu-icon-recognizability/  *(NB: the widely-cited "Spotify +30% discovery / Instagram 5-tab" figures circulating in blog posts are* **[unverified precise figures]** *— treat as anecdote, not evidence.)*
- **Why it matters:** Over-stuffed bars shrink touch targets below the 44px minimum and become icon soup. The B2B **7-item** menu exceeds this ceiling → it should NOT force into a bottom bar; use drawer + optional bottom bar of the 4–5 most-used items only.
- **Tier:** MUST (count + labels).

### 4.4 Touch-target size (≥44px) inside nav
- **What:** Every tappable nav item — bar tabs, drawer rows, expanders, close × — needs a minimum interactive size around 44×44px.
- **Source:** SGS baseline already mandates 44px (beats WCAG 2.2's 24px SC 2.5.8). Ergonomic basis: Smashing thumb-zone article — https://www.smashingmagazine.com/2016/09/the-thumb-zone-designing-for-mobile-users/
- **Why it matters:** Applies to *every* nav surface, especially split parent/expander rows (3.6) where two targets share one row — both must stay ≥44px. Automatable as a CSS floor.
- **Tier:** MUST.

## 5. Sticky / shrink / hide-on-scroll headers + the transform-ancestor gotcha

### 5.1 Native top-layer overlay (`<dialog>` / popover) — the drawer's correct home
- **What:** Render the drawer in the browser **top layer** via `<dialog showModal>` or the Popover API. Top-layer elements escape ancestor `overflow`, `z-index`, AND `transform` containment, and get `::backdrop` + focus handling largely for free.
- **Source:** MDN, *Using the Popover API* — https://developer.mozilla.org/en-US/docs/Web/API/Popover_API/Using ; Chrome, *Introducing the popover API* — https://developer.chrome.com/blog/introducing-popover-api ; Open UI popover explainer ("should not be required to re-parent a top-layer element to `document.body` to escape ancestor containment and transforms") — https://open-ui.org/components/popover.research.explainer/
- **Why it matters:** **This is the clean fix for the SGS hide-on-scroll constraint.** Because the hide-on-scroll header uses CSS `transform`, any `position:fixed` drawer that is a DOM descendant of that header would be trapped and mispositioned (see 5.4). A top-layer `<dialog>`/popover drawer is *immune* — it renders above the transformed ancestor by construction, no JS portal needed. Popover API is **Baseline** (widely available across current Chrome/Edge/Firefox/Safari) per MDN. **This should be the default drawer implementation.**
- **Tier:** SHOULD (strongly recommended default; MUST if hide-on-scroll header ships).

### 5.2 Hide-on-scroll header — when it helps vs. hurts
- **What:** Header hides on scroll-down (maximises content), reveals on scroll-up (nav always one flick away). Use a movement threshold (~150px) so tiny scrolls near the top don't flicker it.
- **Source:** Envato Tuts+, *How to Hide/Reveal a Sticky Header on Scroll* (150px threshold; store last scroll position, compare) — https://webdesign.tutsplus.com/how-to-hide-reveal-a-sticky-header-on-scroll-with-javascript--cms-33756t ; Motion.dev scroll-hide-header tutorial — https://motion.dev/tutorials/react-scroll-hide-header
- **Why it matters:** Helps on long content/feed pages (Medium, Airbnb pattern); hurts when the header is tall/content-rich or the viewport is short (it eats scarce space and distracts). Should be an *opt-in attribute*, off by default, with the reveal-on-up behaviour and a threshold. Keep the transform value in CSS, not inline.
- **Tier:** SHOULD (opt-in), with the caveat below.

### 5.3 Shrink-on-scroll (condense, don't hide)
- **What:** Header stays pinned but shrinks (smaller logo/padding) after a scroll threshold — keeps nav permanently visible while reclaiming some space.
- **Source:** General pattern documented alongside hide-on-scroll: Envato Tuts+ — https://webdesign.tutsplus.com/how-to-hide-reveal-a-sticky-header-on-scroll-with-javascript--cms-33756t ; DonDivi, *Hide/Reveal your Sticky Header on Scroll* — https://dondivi.com/hide-reveal-your-sticky-header-on-scroll/
- **Why it matters:** A safer middle-ground than full hide for sites where nav must always be visible (e.g. the B2B two-row header can collapse its secondary row on scroll while keeping the primary row). Lower motion, lower risk.
- **Tier:** NICE.

### 5.4 The transform-ancestor / `position:fixed` gotcha (the core constraint)
- **What:** Any ancestor with a non-`none` `transform` (also `filter`/`perspective`/`will-change`/`contain`) becomes the containing block for its `position:fixed` descendants — so a "fixed" drawer/modal positions relative to that ancestor, not the viewport, and can be clipped or mis-placed.
- **Source:** MDN, *Stacking context* / containing-block behaviour — https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Positioned_layout/Stacking_context ; Eric Meyer, *Un-fixing Fixed Elements with CSS Transforms* — http://meyerweb.com/eric/thoughts/2011/09/12/un-fixing-fixed-elements-with-css-transforms/ ; salilnaik, *The uncanny relationship between position:fixed and transform* — https://dev.to/salilnaik/the-uncanny-relationship-between-position-fixed-and-transform-property-32f6
- **Why it matters:** This is EXACTLY the SGS hide-on-scroll situation. If the drawer markup lives inside the transformed header, it breaks. Two robust fixes: **(a)** render the drawer in the top layer (5.1 — preferred, no JS portal); **(b)** portal the drawer's DOM node out to `<body>` so it has no transformed ancestor. Do NOT rely on the drawer being a sibling of the header unless it's outside every transformed wrapper.
- **Tier:** MUST (know it; pick fix (a) or (b)).

### 5.5 Scroll-direction detection without brittle JS
- **What:** Newer CSS can express hide-on-scroll declaratively — scroll-driven animations + scroll-state / style queries — reducing or removing the scroll-listener JS.
- **Source:** Bram.us, *Solved by CSS Scroll-Driven Animations: hide a header…* — https://www.bram.us/2024/09/29/solved-by-css-scroll-driven-animations-hide-a-header-when-scrolling-up-show-it-again-when-scrolling-down/ (Oct-2025 update notes newer `scroll-state()` container queries make the earlier hack unnecessary); MDN scroll-driven animations — https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Scroll-driven_animations
- **Why it matters:** A future-facing implementation path, but **Chromium-only today** (see §7). For now, ship the vanilla-JS scroll-direction detector (5.2) with a threshold; treat the CSS-only route as progressive enhancement.
- **Tier:** EMERGING (see §7.5).

## 6. Emerging & bleeding-edge nav patterns (2024–2026)

### 6.1 Priority+ / "More" overflow menu
- **What:** Show as many top-level links as fit the available width; overflow the rest into a "More ▾" dropdown. As the viewport widens, more links surface automatically.
- **Source:** Brad Frost, *Revisiting the Priority+ Pattern* — https://bradfrost.com/blog/post/revisiting-the-priority-pattern/ ; CSS-Tricks, *The Priority+ Navigation Pattern* — https://css-tricks.com/the-priority-navigation-pattern/
- **Why it matters:** The best-of-both answer for the **B2B 7-item** bar: on wide screens all 7 show; as space shrinks, low-priority items fold into "More" *before* the whole thing collapses to a drawer. Far more discoverable than an all-or-nothing hamburger. Needs a priority order (attribute per item) + a `ResizeObserver` in vanilla JS. Frost's own caveat: the org's priority ranking may not match users' — expose it as editable.
- **Tier:** SHOULD (high-value differentiator; NICE if scope-constrained).

### 6.2 Native CSS-only disclosure via `<details>`/`<summary>` + exclusive accordion
- **What:** `<details>`/`<summary>` give open/close toggling with zero JS; giving grouped `<details>` the same `name` attribute makes them an *exclusive accordion* (opening one closes the siblings). `::details-content` + `transition-behavior: allow-discrete` animate it.
- **Source:** MDN Blog, *Exclusive accordions using the HTML details element* — https://developer.mozilla.org/en-US/blog/html-details-exclusive-accordions/ ; Chrome, *Exclusive Accordion* — https://developer.chrome.com/docs/css-ui/exclusive-accordion
- **Why it matters:** A robust, accessible, JS-free fallback for in-drawer accordions (§3.2) — works even if view.js fails to load, aligning with SGS's progressive-enhancement + no-jQuery stance. Support for `details name` is broad across current Chrome/Firefox/Safari (reported enabled by default Firefox 130+ / broadly available through 2025) — **[verify exact versions at build]**. The animation bits (`::details-content`, discrete transitions) are newer — degrade to instant open/close.
- **Tier:** SHOULD (as the no-JS baseline for accordions); animation is NICE.

### 6.3 Popover API + CSS anchor positioning — JS-free dropdowns/mega
- **What:** `popovertarget` opens a popover with light-dismiss + top-layer for free; CSS anchor positioning (`anchor-name` / `position-anchor` / `position-area`) tethers it to its trigger and auto-flips when it would overflow — no JS positioning library.
- **Source:** css-irl, *Anchor Positioning and the Popover API for a JS-Free Site Menu* — https://css-irl.info/anchor-positioning-and-the-popover-api/ ; Chrome, *Introducing the CSS anchor positioning API* — https://developer.chrome.com/blog/anchor-positioning-api ; MDN Popover API — https://developer.mozilla.org/en-US/docs/Web/API/Popover_API
- **Why it matters:** Could power the B2B dropdowns/mega with almost no JS and inherent overflow-safety — and (via top layer) sidesteps the transform gotcha (5.4). BUT: Popover API is Baseline; **anchor positioning is NOT yet Baseline** (Chromium 125+, Safari 26, Firefox only from late-2025) — **[verify]**. Ship popover now; treat anchor positioning as progressive enhancement over a JS/fallback position.
- **Tier:** EMERGING (popover=SHOULD-ready; anchor positioning=EMERGING).

### 6.4 Command palette / search-first nav (⌘K)
- **What:** A keyboard-summoned overlay (typically `Cmd/Ctrl-K`) mixing search + navigation + actions + recent items in one filterable list.
- **Source:** UX Patterns for Developers, *Command Palette* — https://uxpatterns.dev/patterns/advanced/command-palette ; Rob Dodson, *Command palettes for the web* — https://robdodson.me/posts/command-palettes/
- **Why it matters:** Power-user accelerator seen in Linear/GitHub/Notion/Vercel. For SGS, more relevant to *large* content/commerce sites or the *editor* experience than a 5-item restaurant site. A differentiator to offer, not a default; must be an *enhancement* on top of visible nav (keyboard-only summons exclude touch-only users if it's the sole path).
- **Tier:** NICE (site-dependent); EMERGING as a mainstream *site* nav.

### 6.5 View Transitions API — animated page/section transitions
- **What:** Smoothly animate between states/pages (e.g. drawer level changes, or whole-page nav) with the browser crossfading/morphing matched elements.
- **Source:** MDN, *View Transition API* — https://developer.mozilla.org/en-US/docs/Web/API/View_Transition_API ; Chrome, *What's new in view transitions (2025 update)* — https://developer.chrome.com/blog/view-transitions-in-2025
- **Why it matters:** Can make drill-down (3.3) level changes and MPA page navs feel app-like. **Same-document (SPA) transitions reached Baseline "newly available" ~Oct 2025** (Chrome/Edge 111+, Safari 18+, Firefox 144+). **Cross-document (MPA) is Chromium 126+ / Safari 18.2+, not Firefox yet** — **[verify]**. Must be wrapped in `prefers-reduced-motion` (2.7) and degrade to instant.
- **Tier:** EMERGING.

### 6.6 Scroll-driven animations — declarative hide-on-scroll/scroll progress
- **What:** Drive animations from scroll position/element visibility with pure CSS (`animation-timeline: scroll()`/`view()`), replacing scroll-listener JS.
- **Source:** MDN, *CSS scroll-driven animations* — https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Scroll-driven_animations ; Chrome, *Animate elements on scroll with Scroll-driven animations* — https://developer.chrome.com/docs/css-ui/scroll-driven-animations ; WebKit, *A guide to Scroll-driven Animations with just CSS* — https://webkit.org/blog/17101/a-guide-to-scroll-driven-animations-with-just-css/
- **Why it matters:** Future path for hide-on-scroll (5.5) and scroll-progress indicators. **Full support Chrome/Edge; Safari partial/landing; Firefox in development** — **[verify]**. Non-supporting browsers just don't play the animation (content stays visible) — safe to layer on. Not a baseline you can rely on yet.
- **Tier:** EMERGING.

## 7. TOP GAPS — what a WP-competitor-only feature set would miss

These are the highest-value items a nav block set benchmarked only against Kadence/Spectra/GenerateBlocks/Elementor/Bricks would likely under-deliver. Ordered by value.

1. **Top-layer drawer (`<dialog>`/popover) that survives a transformed ancestor.** (5.1/5.4) The direct, no-JS-portal fix for the SGS hide-on-scroll `transform` constraint. Most theme drawers are hand-rolled `position:fixed` inside the header and *will* break under a transformed header. **Highest value + directly de-risks a known SGS constraint.**
2. **Bottom tab bar as a first-class mobile primary-nav mode.** (4.2) Thumb-zone placement with production evidence of large engagement lift; rarely offered as a native option by WP competitors. Perfect for the restaurant flat-5 + cart.
3. **Priority+ / "More" overflow before full collapse.** (6.1) Keeps the B2B 7-item nav discoverable across the awkward mid-widths instead of jumping straight to a hamburger.
4. **NN/g subcategory-volume decision rule baked into the pipeline.** (3.1) Automatable branch (accordion vs section-menu/drill-down vs landing page) — turns a design judgement into a deterministic, universal rule the cloner can apply.
5. **Split parent-link vs. expander targets.** (3.6) The most common broken-mobile-menu bug (parent page unreachable). Cheap to get right, painful to get wrong; matters for both grounding menus.
6. **Full a11y contract on the toggle+drawer by default** — `aria-expanded`/`aria-controls`/`aria-labelledby`, focus trap + return focus, `inert` background, `Esc` + visible × (2.3/2.5/2.8). Shipped by construction, not left to the site-builder.
7. **`prefers-reduced-motion` handling on every nav animation.** (2.7) WCAG-aligned, trivial, frequently omitted by competitor blocks.
8. **Reliable body-scroll-lock (incl. the iOS Safari fix).** (2.4) "Background scrolls under the open drawer" is a top-3 mobile nav bug; competitors often ship it broken.
9. **Native `<details>` exclusive-accordion as the no-JS accordion baseline.** (6.2) Progressive-enhancement backbone that keeps menus working if view.js fails — aligns with SGS's vanilla-JS/no-jQuery ethos.
10. **Labelled, unboxed hamburger + "show links up to N, then collapse" breakpoint logic.** (1.3/1.4) Small, evidence-backed defaults that quietly beat the generic competitor hamburger.

## 8. EMERGING — watch, don't necessarily build (honest maturity notes)

Build behind capability detection / as progressive enhancement only. Re-verify all version numbers against caniuse + MDN Baseline before shipping.
