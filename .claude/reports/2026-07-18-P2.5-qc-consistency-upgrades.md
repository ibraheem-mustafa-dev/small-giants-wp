upgrade-hunting now the whole header/footer/nav system is laid out.

## Counts

- **CONFIRMED:** 11 findings — 3 HIGH, 5 MED/MED-HIGH, 3 LOW/LOW-MED.
- **PLAUSIBLE:** 2 findings.
- Split by type: 4 INCONSISTENCY, 9 UPGRADE.
- Bottom line: Spec 36 is well-structured and mostly faithful to P2/Spec 17, BUT it **silently reverses Spec 34's Bean-locked drawer model** (the single most important finding) and **under-absorbs three things Spec 34 got right** (crawlable-links contract, the live-QC drawer gate, the migration discipline). None of the findings is fatal; F1 needs Bean's adjudication before build.

## Top 5 (read these first)

1. **F1 (HIGH, INCONSISTENCY) — the drawer goes MODAL, silently reversing Spec 34's Bean-locked "header stays live" model**, while *claiming* to "absorb" Spec 34's geometry and FR-S9-5. Needs Bean's explicit decision, not a buried default.
2. **F2 (HIGH, UPGRADE) — the crawlable-server-render / NO-AJAX contract (D334, AI-search differentiator) is absent** from Spec 36's explicit requirements, especially for mega panels.
3. **F3 (HIGH, UPGRADE) — Spec 34's concrete live-QC drawer gate (FR-34-7) is not absorbed;** Spec 36 §8 is a one-line acceptance where Spec 34 had elementFromPoint sweeps + baselines + axe-on-open + cache-clear discipline.
4. **F4 (MED-HIGH, UPGRADE/GAP) — no cutover plan for the 2 live `sgs/adaptive-nav` production instances** the rebuild replaces (mirror P2 §6c + Spec 34's zero-instance measurement).
5. **F5 (MED-HIGH, INCONSISTENCY/GAP) — "drawer default = the header's menu (one-source)" has no mechanism** across the decoupled header↔drawer boundary; the parent-child context Spec 34 used no longer holds.

## CONFIRMED — HIGH

### F1 — Drawer modality: Spec 36 reverses Spec 34's non-modal "header-stays-live" model and misrepresents it as an "absorb"
- **Type:** INCONSISTENCY · **Target:** spec36 ↔ spec34 / spec17 (FR-S9-5) · **Severity:** HIGH
- **Refs:** Spec 36 FR-36-6, FR-36-9, FR-36-10, §1 ("FR-S9-5 fold here") ↔ Spec 34 FR-34-1 + D1/D2 ↔ Spec 17 FR-S9-5 (amended text, lines 730–737).
- **Spec 36 side (modal):**
  - FR-36-6: *"the drawer renders in the browser top layer (`<dialog showModal>` / Popover API)… **Dialog a11y (absorbs FR-S9-5):** `role="dialog"` + `aria-modal="true"` + label; focus moves INTO on open; Tab is CONTAINED (cycles)… background `inert`."*
  - FR-36-10: *"The drawer = the DIALOG (modal) pattern: contained focus, mandatory Escape, `inert` background."*
  - FR-36-9: *"when the header is visible/sticky, the open drawer's top edge **attaches to the bottom of the header** (the header-visible drawer — absorbs Spec 34's geometry)."*
- **Spec 34 / Spec 17 side (non-modal, header live):**
  - Spec 34 FR-34-1: *"The drawer stops using `showModal()`/top-layer. It stays a `<dialog>` opened with `.show()`… role `dialog` **WITHOUT `aria-modal`** — a non-modal dialog is the honest AT announcement… `aria-modal` is **NEVER set**, statically or dynamically."*
  - Spec 34 D1/D2 (Bean, "do not re-litigate"): *"Header top row stays visible **AND interactive** while the drawer is open"* / *"Burger ↔ X toggle in place; the header toggle IS the close control."*
  - Spec 17 FR-S9-5 (live amended text): *"`showModal()` was retired by the disclosure redesign (the header must stay interactive, **which a modal dialog forbids by construction**)."*
- **Why it's a real contradiction (not just a supersession):** `aria-modal="true"` + a full `inert` background is *definitionally incompatible* with a live, reachable header row. Spec 34 deliberately keeps the header reachable (burger↔X in place is the close control); Spec 36's modal makes the in-header burger unreachable and needs a separate × inside the drawer (FR-36-6's "visible × close" + "focus returns to the burger on close" confirms the burger is now outside the modal). So Spec 36 has taken the **opposite side of a Bean decision made on lived experience (D340)** — and it does so while (a) asserting it "absorbs FR-S9-5" (§1) and "absorbs Spec 34's geometry" (FR-36-9), when Spec 34's geometry was inseparable from the non-modal model, and (b) never flagging the reversal. Worse on a11y: a visually-present header that `aria-modal` hides from AT is *more* confusing than Spec 34's honest non-modal announcement — arguably a regression on the exact axis Spec 34 optimised.
- **Fix (surface it as a decision, per ADHD Rule 9 + "requirement-used-to-justify vs design-constraint"):** Spec 36 must present this as an explicit fork for Bean at the Phase-5 gate, not a silent default:
  - **Option A (recommended) — keep Spec 34's non-modal selective-freeze model:** it is already BUILT, QC'd to 10/10 Mama's + 18/18 Indus, and is the stated differentiator ("commercial builders under-document this"). Header stays live, burger↔X in place, `aria-modal` never set, selective `inert` on everything except {header row + drawer + scrim + `#wpadminbar`}. Rewrite FR-36-6/FR-36-10 to this contract.
  - **Option B — go modal (APG "Dialog (Modal)")** only if Bean deliberately drops the header-stays-live behaviour; then FR-36-9's "attaches to the bottom of the header" is purely cosmetic and the in-header X close path is gone. State that trade-off.
  - Either way: **stop claiming to "absorb FR-S9-5/Spec 34 geometry"** while reversing the model. (Note the P2.5 Phase-1 a11y research cited for FR-36-10 (Adrian Roselli, APG) almost certainly defaulted to the generic modal without reconciling Bean's D340 decision — this is exactly the reconciliation the gate exists for.)

### F2 — The crawlable-server-render / NO-AJAX contract (D334) is dropped from Spec 36's explicit requirements
- **Type:** UPGRADE (absorb a locked requirement) · **Target:** spec36 (absorbs spec17/spec34) · **Value:** HIGH
- **Refs:** Spec 17 FR-S9-4 + D334 ↔ Spec 36 FR-36-1, FR-36-4, FR-36-5, §9.
- **Ground truth (Bean-locked):** Spec 17 FR-S9-4: *"ALL links are server-rendered as real `<a href>` — **NO AJAX lazy-load** — so every link stays visible to crawlers and legitimate AI search (Perplexity/ChatGPT/Claude/Gemini)… AJAX lazy-load would actively harm AI-indexability"* (D334, Bean-decided). This is a commercial + SEO/GEO differentiator, reinforced across CLAUDE.md's AI-visibility rules.
- **Spec 36 side:** FR-36-1 says the block *"walks the tree in render.php to emit its OWN scoped SGS markup"* (server-side, good) and FR-36-4 says the mega is *"height-animated to real content height"* (implies in-DOM), but **nowhere states the no-AJAX / all-links-crawlable constraint** — most dangerously for the mega CPT panels, whose rich content is the obvious candidate for hover-time lazy loading. An implementer optimising panel weight could lazy-load mega content and silently break the differentiator with zero test catching it.
- **Fix:** Add an explicit constraint to FR-36-4/FR-36-5 and §9: *"Every menu item AND every mega-panel link renders as a server-side `<a href>` in the initial HTML; NO AJAX/lazy-load of nav or mega content (D334 — crawler + AI-search visibility)."* Add a crawl assertion to §8 acceptance (all bar + all mega links present in server HTML before any JS).

### F3 — Spec 34's concrete live-QC drawer gate (FR-34-7) is not absorbed; Spec 36 §8 is far weaker
- **Type:** UPGRADE (capability Spec 34 got right, Spec 36 under-specifies) · **Target:** spec36 (absorbs spec34) · **Value:** HIGH
- **Refs:** Spec 34 FR-34-7 + FR-34-1 test strategy ↔ Spec 36 §8 FR-36-16.
- **Spec 34 side (rigorous):** FR-34-7 mandates, in one live pass on the canary: 375/768/1440 sweeps; **axe = 0 on the OPEN drawer**; the **`elementFromPoint` sweep ≥ baseline (10/10 Mama's, 18/18 Indus)**; ESC/focus-return/Tab-containment; scroll-lock + scrollbar-pin frame sweep (anchor constant); geometry `getBoundingClientRect().top === header bottom ±1px`; the **late-CSS A/B** (force-disable then restore the block's CSS); a **cache-clear-first sequence** (`hosting_clearWebsiteCacheV1` + `wp litespeed-purge all` before any measurement); the **"measure on a real desktop scrollbar, device emulation is vacuous"** note; and Bean's cropped screenshot pair (R-31-13). Each clause names its tool.
- **Spec 36 side (thin):** FR-36-16 closes the whole acceptance with *"Both reproduced on the real page (live DOM + computed style) + Bean's visual sign-off."* This re-derives a weaker gate for the *same drawer* and drops every hard-won regression probe (elementFromPoint, axe-on-open, geometry ±1px, late-CSS A/B, cache-clear-first).
- **Fix:** Have Spec 36 §8 absorb FR-34-7's clauses verbatim for the drawer surface (with the measured baselines), plus add the F2 crawl assertion and a mega "renders at real bar position N, not last" assertion (the fatal-bug regression guard FR-36-5 names). This prevents re-introducing the exact bug classes Spec 34 fixed.

### F4 — No cutover plan for the two LIVE `sgs/adaptive-nav` production instances the rebuild deletes
- **Type:** UPGRADE / GAP · **Target:** spec36 (+ spec17/P2 discipline) · **Value:** MED-HIGH
- **Refs:** Spec 17 FR-S9-4 BUILT note (*"live on BOTH sandybrown (Mama's) and palestine-lives (Indus, verified D334)"*), Spec 17 FR-S9-8 BUILT, P2 §6c (migration discipline), Spec 34 §6 (zero-instance measurement) ↔ Spec 36 §1 + §8.
- **Specifics:** Spec 36 declares the whole trio *"REFERENCE-ONLY and replaced"* and folds/deletes Spec 34, but the current `sgs/adaptive-nav` markup is live on two production pages. There is **no stated migration/cutover**: no "re-clone/re-author onto the new blocks, canary-first (sandybrown → palestine-lives), with a before/after live check," and no "measure zero stored instances of the old blocks' attrs before deletion" (the exact discipline P2 §6c and Spec 34 §6 both applied for the header flags and the 6 removed drawer attrs). Replacing live blocks with no cutover is how a deploy silently breaks a production nav.
- **Fix:** Add an FR (or fold into §8) mirroring P2 §6c: re-author both live headers onto `sgs/nav-menu`/`sgs/nav-drawer`/`sgs_mega_menu` via the CPT editor (no WP-CLI `post_content` edits), canary-first, with a live before/after computed check that both menus still render + collapse; measure zero live instances of the retired blocks' attrs before any deletion.

### F5 — "Drawer default = the header's menu (one-source)" has no mechanism across the decoupled boundary
- **Type:** INCONSISTENCY / GAP · **Target:** spec36 (+ vs spec34) · **Value:** MED-HIGH
- **Refs:** Spec 36 FR-36-1, FR-36-6, FR-36-8 ↔ Spec 34 D6 + FR-34-4.
- **Spec 34 side (mechanism specified):** the drawer's `sgs/nav-menu` was a *descendant* of `sgs/adaptive-nav`, so one-source was delivered by **block context** — FR-34-4/D6: *"`sgs/adaptive-nav` declares `providesContext: { "sgs/navRef": "ref" }`; `sgs/nav-menu` `usesContext`… the FR-S9-4 one-source rule holds by default."*
- **Spec 36 side (mechanism missing):** FR-36-8 decouples them — *"the burger lives on `sgs/nav-menu`… opens the associated `sgs/nav-drawer`. The drawer stays a separate authorable container."* So the drawer's inner `sgs/nav-menu` is **no longer a descendant** of the header's `sgs/nav-menu`; they're cousins in different subtrees. Yet FR-36-6 still asserts *"Default content = the header's menu (one-source, FR-36-1)."* **How the drawer's menu inherits the header menu's `ref` by default is unspecified** — the parent→child context that made it "by default" in Spec 34 no longer holds.
- **Fix:** Specify the cross-boundary one-source mechanism: e.g. the `sgs/nav-drawer` (or its inner `sgs/nav-menu`) reads the associated header `sgs/nav-menu`'s `ref` via the nav→drawer association (see F9), or both resolve a single site-level "primary menu" as the default source, with the per-instance `ref` picker as the deliberate override (FR-36-1). Without this, "one-source by default" is a claim with no delivery path.

### F6 — Nav↔header runtime coupling needs a *published* header-state hook the header spec exposes
- **Type:** UPGRADE (the nav design exposes a header-side hook that isn't formally exposed) · **Target:** spec17/P2 (+ spec36 consumes) · **Value:** MED-HIGH
- **Refs:** Spec 36 FR-36-9 ↔ Spec 17 FR-S9-9 + Spec 34 FR-34-1 (`--sgs-header-height`).
- **Specifics:** FR-36-9 makes real-time claims — *"when the header is visible/sticky, the open drawer's top edge attaches to the bottom of the header… when the header hides on scroll, the drawer covers full height"* — but names no contract for reading the header's live state. The header already publishes `--sgs-header-height` (FR-S9-9 ResizeObserver, used by Spec 34) and body classes (`sgs-header-behaviour-*`, `is-header-scrolled`, `is-header-shrunk`), but there is **no published "header currently hidden/off-screen" signal** — and the hide-on-scroll header FR-36-9 references *isn't even built yet* (it's a future feature flagged in P2 §15.6 / Spec 34 as `transform: translateY()`). Left as an ad-hoc DOM read, this becomes brittle coupling the "nav → header only" rule (FR-36-9) is meant to prevent.
- **Fix:** Define a one-directional contract now: Spec 17 FR-S9-9 exposes a documented header-state surface — `--sgs-header-height` (exists) + a `data-sgs-header-state="visible|hidden|sticky|shrunk"` attribute (or the existing body classes) — as *the* hook FR-36-9 consumes. Spec 36 cites that surface instead of reading header internals. Add the "header-hidden" signal when the hide-on-scroll feature lands.

### F7 — The header side the nav rebuild exposes: typed palette + unassigned Mama's/Indus items
- **Type:** UPGRADE (header/footer side the nav design now exposes) · **Target:** spec17/P2 · **Value:** MED
- **Refs:** Spec 17 FR-S9-2 (typed palette) + P2 §15.6 ("currently unassigned") ↔ Spec 36 FR-36-8, FR-36-16.
- **Specifics:** (a) `sgs/site-header`'s typed palette is *"logo, **adaptive-nav**, search, cart, account, button/CTA, contact, social, HTML, widget-area"* (FR-S9-2). When adaptive-nav is replaced, the palette + the header block's allowed-elements must **swap `adaptive-nav` → `sgs/nav-menu`** and account for the burger/drawer association — Spec 36 doesn't mention this header-side change. (b) P2 §15.6 explicitly left *"Cart badge + 2-row utility header… (currently unassigned)"*; Spec 36 FR-36-16 mentions Mama's "featured" item + "cart badge" and Indus's two-row header but doesn't assign them to blocks/attrs.
- **Fix:** Spec 36 (or the Spec 17 pointer it leaves) should state: update FR-S9-2's typed palette to `sgs/nav-menu`; the **"featured" item = a per-item style flag in the `sgs/nav-menu` item map** (same map that keys item→mega, FR-36-5); the **cart badge = the existing `sgs/cart` header element** (a header concern, not nav); the Indus two-row header = the header builder's two rows, one carrying `sgs/nav-menu` (Spec 36 already says this — make the palette/element wiring explicit).

### F8 — Reuse P2's `ResponsiveTriStateControl` + FR-S9-8's per-tier visibility for the nav, don't invent a nav-specific one
- **Type:** UPGRADE (shared component both should use) · **Target:** spec36 (reuse P2/Spec 17) · **Value:** MED
- **Refs:** P2 §4.1 (`ResponsiveTriStateControl`), Spec 17 FR-S9-8 (`labelCollapse` enum + per-tier visibility) ↔ Spec 36 FR-36-14.
- **Specifics:** FR-36-14 cites only *"`ResponsiveControl` for 768/1024 tiers"* (values). But the nav has natural **per-device on/off** needs — show a menu item on desktop only, show Mama's "featured" item on mobile only, priority+/"More" mode on/off per tier — which is exactly what P2 built `ResponsiveTriStateControl` for. And FR-S9-8 already shipped a `labelCollapse` (`none|mobile|tablet|all`) + per-tier visibility model the nav's items should inherit rather than reinvent.
- **Fix:** FR-36-14 should name `ResponsiveTriStateControl` for any per-device on/off nav control and reuse FR-S9-8's `labelCollapse`/visibility model for nav items/CTAs — keeping the editor consistent and avoiding a parallel mechanism (R-31-9).

### F9 — The `sgs/nav-menu` → `sgs/nav-drawer` association is under-specified (contrast with the well-specified item→mega map)
- **Type:** UPGRADE (close an internal gap) · **Target:** spec36 · **Value:** MED
- **Refs:** Spec 36 FR-36-8 ("opens the associated `sgs/nav-drawer`… the menu targets it") ↔ FR-36-5 (item→mega map, fully specified).
- **Specifics:** FR-36-5 specifies the item→mega reference precisely (a map on `sgs/nav-menu` keyed by `wp_navigation` item ID/position → `sgs_mega_menu` post ID, with an explicit rationale). But FR-36-8's nav-menu→drawer "association" is asserted without a storage/resolution mechanism (a `drawerRef` attr? a shared uid? auto-find-sibling?). This is the same class of reference and deserves the same rigour — and it's the natural carrier for the F5 one-source resolution.
- **Fix:** Specify a `drawerRef` (or equivalent) attribute on `sgs/nav-menu` that targets a `sgs/nav-drawer` instance, with a default resolution rule (single drawer on the page → auto-target; multiple → explicit pick). Let the drawer's default menu source follow this association (resolves F5).

### F10 — FR-36-3 over-claims "reuses its machinery" (the `Sgs_Header_Rules` active-binding engine doesn't apply to a mega)
- **Type:** INCONSISTENCY (imprecision) · **Target:** spec36 ↔ P2 · **Value:** LOW-MED
- **Refs:** Spec 36 FR-36-3 ↔ P2 §2.2.
- **Specifics:** FR-36-3: *"`sgs_mega_menu` mirrors how the header/footer builder (P2) already works, **and reuses its machinery**."* P2 §2.2's machinery includes the `Sgs_Header_Rules` display-conditions engine + `sgs_active_header_cpt_id` **active-binding** — which is header-specific (a mega is referenced by ID from a menu item and rendered at a bar position, never "activated" into a template slot). So the reuse is *partial*: editing-home + starter-picker + converter-pack-factory yes; rules-engine/active-binding **no**.
- **Fix:** Reword FR-36-3 to name exactly what's reused (CPT editing home, starter-picker per P2 §2.5, converter pack-factory per P2 §9) and note the mega does **not** use the `Sgs_Header_Rules` active-binding path — so a builder doesn't wire a rules engine a mega doesn't need.
