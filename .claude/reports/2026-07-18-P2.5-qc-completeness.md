## Verdict

**The spec is fundamentally sound.** Every factual claim it makes about other specs, WP/browser APIs, the a11y contract, the research findings, and the two client menus **checks out** — I found **zero fabrications** and **zero invented APIs/files/numbers**. The disclosure-vs-dialog architecture, the CPT model, the 300–500ms/0.5s timing, the top-layer drawer, the 5 mega layouts, the 768/1024 tiers, and all the cross-spec FR references (S9-4/5/9, FR-34-3, Spec 33 Part 2, Spec 35 D2/Part L, P2 DP2a/DP6/§2.5/§9) are all faithful to source.

The findings are **all completeness gaps** — a handful of NEED-side items (one MUST-tier, several SHOULD/MUST) that are neither represented in an FR nor explicitly deferred — plus **one internal-consistency wrinkle in the acceptance gate** that it inherited from the inventory.

**Counts:** 1 HIGH · 6 MED · 5 LOW. By type: MISSING 9, UNSUPPORTED/inconsistency 3 (2 inherited from source), FABRICATION 0.

## Top 5 findings (ranked)

1. **[HIGH · MISSING]** Drawer geometry (which edge it opens from — L/R/top/full — and width) is not a named attribute anywhere. This is a MUST-tier need with concrete client mapping (Nike-style right drawer; a compact right drawer beside Mama's cart).
2. **[MED · MISSING]** Progressive-enhancement / no-JS functional baseline (research finding S7 / F2 MUST — `<details name>` accordion, semantic-first) is absent as a requirement — `<details name>` appears only in the §11 Sources list.
3. **[MED · MISSING]** Mega-panel content IA guidance (column grouping, one-item-per-group, vertical scan order, "view all", link descriptions — B1/B3/B4/B8/B11) has no representation, not even as starter-template guidance or an explicit "operator-owned" non-goal.
4. **[MED · UNSUPPORTED]** FR-36-16 Indus acceptance is internally inconsistent: "1 mega 'Brands' at position 4" vs "5 rich mega panels (the 5 layouts as `sgs_mega_menu` posts)". As the pass/fail gate this must be reconciled against the real Indus draft.
5. **[MED · MISSING]** Cart/utility-count badge (C6 SHOULD; a named Mama's requirement) is required by the acceptance gate (FR-36-16) but no FR defines the capability — which block owns it, what attribute drives it.

## CONFIRMED findings (verified against a source)

### F1 — [HIGH · MISSING] Drawer geometry (edge + width) is unspecified
- **Spec ref / text:** FR-36-6. The drawer is defined only as "one InnerBlocks container … default template `[ container, nav-menu, container ]`" and "a top-layer dialog (`<dialog showModal>` / Popover API)". Nowhere are the drawer's **opening edge** or **width** named as attributes. Grep confirms the only occurrence of "edge" in the spec is the drawer's *top* edge attaching to the header (FR-36-9), not the slide-in side.
- **Source (need):** Inventory §2D (mobile row): *"Drawer geometry as attributes (edge L/R/top/full, width); backdrop/scrim click-to-dismiss | **MUST**"* (from mobile-emerging 2.1–2.2). Visual teardown corroborates the variation is real and load-bearing: Nike ships a right-edge drawer at ~85% width; Stripe/Apple/Shopify ship full-screen — *"Geometry should be an attribute (side + width), not a hard-coded choice."*
- **Why HIGH:** A build "by construction from the spec" would hardcode one geometry (likely full-screen). That blocks reproducing a Nike-style right drawer and Mama's compact-right-drawer-beside-cart, and undercuts the "meet-and-exceed competitors" goal (Kadence/Spectra expose drawer side + width). FR-36-14's Part-L control-completeness only backstops properties the spec has *established exist* — it won't invent this one.
- **Fix:** Add to FR-36-6 (or FR-36-8) that `sgs/nav-drawer` exposes **edge (`left`/`right`/`top`/`full-screen`)** and **width** as block attributes (default full-screen, per the a11y-simpler default), and that backdrop/scrim is **click-to-dismiss**. Then FR-36-14 covers exposing them as controls.

### F2 — [MED · MISSING] No progressive-enhancement / no-JS functional baseline
- **Spec ref / text:** No FR requires the nav to *function* without JS. FR-36-1 gives a server-rendered semantic base ("walks the tree in render.php to emit its OWN scoped SGS markup"), which covers crawlability, but dropdown/drawer *operability* without JS is never stated. `<details name>` appears **only** in §11 Sources ("`<details name>`, view transitions…"), never as a requirement.
- **Source (need):** This is convergent finding **S7** — one of only 8 in the inventory §1 synthesis: *"Progressive enhancement / no-JS baseline is achievable now. Semantic `<nav>`+`<ul>`+`<a>` that works without JS; `<details name>` exclusive-accordion as the no-JS drawer-accordion… Robustness + crawlability + clone-fidelity all improve."* Backed by mega-dropdown F2 (**MUST**), mobile-emerging 6.2 (SHOULD, the no-JS accordion baseline) + §7 top-gap #9, and accessibility A4. Aligns explicitly with the SGS vanilla-JS/no-jQuery ethos.
- **Fix:** Add an FR (or a clause to FR-36-7) requiring a functional no-JS baseline: server-rendered semantic markup that navigates without JS, `<details name>` exclusive-accordion as the in-drawer no-JS accordion fallback (⚠VERIFY support), and JS layered as enhancement. If the framework consciously declines the `<details>` route, record that as a reasoned non-goal rather than leaving it silent.

### F3 — [MED · MISSING] Mega-panel content IA best-practice has no representation
- **Spec ref / text:** FR-36-5 makes the mega panel a free-form block canvas ("the operator drags any SGS blocks in … No fixed template cage — the 5 layouts are starter templates"). It captures the mega-vs-dropdown **threshold** (B10, in FR-36-4 "chosen per item by a documented rule") but nothing about *how a good panel is organised*.
- **Source (need):** mega-dropdown report — **B1** column grouping (MUST), **B3** show each choice once (MUST), **B4** vertical scan order (MUST), **B8** "view all"/parent-category link (SHOULD), **B11** descriptions under links (NICE); plus C3 active-trail *inside the panel*. The report's TOP-GAPS #? frames these as part of "reproduce the best navs."
- **Why it matters:** With a blank canvas and no IA scaffolding, operators (and the converter's starter packs) get no guidance toward the grouping/headings/"view all"/vertical-scan that separate a good mega from a link dump. At minimum B8 (view-all) and B4 (vertical scan) are MUST.
- **Fix:** Either bake the IA guidance into the **5 starter-template designs** (a natural home — they already exist as the layout presets) and reference it in FR-36-5, or add an explicit note that panel IA is operator-owned (informed by starter-template defaults), not framework-enforced. Do not leave it silent.

### F4 — [MED · MISSING] Converter per-client panel naming/scoping convention absent
- **Spec ref / text:** FR-36-15 covers "Write one `sgs_mega_menu` CPT post per rich panel … + reference it by ID". It does not mention a naming/collision-avoidance convention.
- **Source (need):** Inventory §2G (council-verified converter needs): *"A converter-owned template/panel **naming convention** so clients never collide (e.g. per-client scoping)."*
- **Why it matters:** CPT posts have unique numeric IDs, so ID-collision is a non-issue — but slug/title collisions and (per FR-36-3) shared starter-pack naming across clients are real. The council flagged it explicitly.
- **Fix:** Add a clause to FR-36-15 naming the per-client scoping/naming convention for emitted `sgs_mega_menu` posts + starter packs (or state why the numeric-ID model makes it moot).

### F5 — [MED · MISSING] Cart/utility badge has an acceptance requirement but no capability FR
- **Spec ref / text:** FR-36-16 Mama's acceptance requires "a **cart badge**". No functional FR (FR-36-2/4/5/8/14) describes which block provides a count badge or what attribute drives it.
- **Source (need):** mega-dropdown **C6** "Cart/utility badges on the bar" (SHOULD); inventory §2A Mama's row: *"cart-count badge affordance"* — a named forces-it feature.
- **Why it matters:** The build team is told to *reproduce* it but given no spec for *how* (which block owns it, dynamic count source, a11y of a live count). It's "represented in the gate, unspecified as a capability."
- **Fix:** Add a short FR or a clause to FR-36-4/FR-36-2 describing the utility/cart-count badge affordance (owning block, count source, accessible-name/live-region treatment).

### F6 — [MED · UNSUPPORTED] Indus acceptance is internally inconsistent (1 mega vs 5 panels)
- **Spec ref / text:** FR-36-16 Indus: *"7-item bar; 3 simple dropdowns; **1 mega 'Brands' at position 4** … **5 rich mega panels** (the 5 layouts as `sgs_mega_menu` posts)."* If only one bar item ("Brands") opens a mega, "5 rich mega panels" as a live Indus reproduction target does not follow.
- **Source:** The inventory §2A carries the same ambiguity verbatim (*"1 rich MEGA 'Brands' at bar position 4; 5 distinct rich mega-panel layouts"*) and research S5 (*"Indus alone needs five"*). **The sources themselves don't resolve whether Indus has 1 live mega panel or 5** — so the claim is UNSUPPORTED as a precise pass/fail criterion, though it is inherited, not invented.
- **Why it matters:** This is *the* pass/fail gate. An ambiguous gate ("reproduce 1 mega" or "reproduce 5 mega panels"?) can't be cleanly closed.
- **Fix:** Reconcile against the actual Indus draft/mockup before build: state whether the "5 layouts" are (a) five live Indus panels, or (b) five framework starter-templates demonstrated by the build with Indus using one for "Brands". Pick one and write it into FR-36-16.

### F7 — [LOW · MISSING] Nav search (D7) is neither represented nor deferred
- **Spec ref / text:** No mention of site search in the nav. Grep confirms "internal search" (FR-36-14) refers to `LinkControl`'s page-search, not a header search field.
- **Source (need):** mega-dropdown **D7** "Search in the nav (persistent field or expand-on-click icon)" (SHOULD); inventory §2D lists it under "Header behaviours."
- **Note:** Ownership is genuinely ambiguous — as a "header behaviour" it may be Spec 17's, not the nav's. But Spec 36 neither claims it nor assigns it away.
- **Fix:** One line — assign nav search to Spec 17 (header) explicitly, or add it as a nav SHOULD. Resolve the ownership rather than leaving it silent.

### F8 — [LOW · MISSING] Per-device visibility of nav items/elements not addressed
- **Spec ref / text:** FR-36-8 covers the responsive *collapse* (bar → drawer) and the 768/1024 tiers, but not per-item/per-element show-hide-by-device.
- **Source (need):** Inventory §2C (competitor table-stakes, Stream B): *"per-device visibility."* Kadence/Spectra ship it.
- **Note:** May be covered indirectly (drawer InnerBlocks lets the operator put different content in the drawer vs bar) and/or be Spec 17's "move-to-drawer" (FR-S9-8). Low, but unaddressed.
- **Fix:** A one-line pointer to how per-device visibility is met (drawer content divergence + Spec 17 FR-S9-8) or an FR if it's a nav concern.

### F9 — [LOW · MISSING] `imageControls: true` discipline not stated for image-rendering mega layouts
- **Spec ref / text:** FR-36-14 mentions "`MediaGalleryPicker`/icon controls where relevant (logo-grid mega)" but not the `block.json` `supports.sgs.imageControls: true` flag the project's "Image controls discipline" mandates for any block rendering `<img>` (photo-grid / logo-grid mega).
- **Source (need):** Project CLAUDE.md "Image controls discipline" (framework rule). Low because FR-36-14 already gestures at media controls.
- **Fix:** Add `imageControls: true` to the mega/media blocks' declared supports in FR-36-14 (or the block-roster note).

## PLAUSIBLE findings (suspected, not fully resolvable from the sources)

### P1 — [LOW · UNSUPPORTED] "hide-on-scroll header" as a Spec 17 state the nav adapts to
- **Spec ref / text:** FR-36-9: "when the header **hides on scroll**, the drawer covers full height" — and FR-36-9 attributes header state to "Spec 17 FR-S9-9."
- **Check:** Spec 17 FR-S9-9 is titled *"Sticky / transparent-at-rest-to-solid-on-scroll"* and its bullets cover `headerSticky`/`headerTransparent`/`headerShrink` — I did **not** find an explicit hide-on-scroll (header fully hides) behaviour in FR-S9-9. Research treats hide-on-scroll as an *opt-in, not-yet-built* header pattern (mobile-emerging 5.2). So the nav's "header hides on scroll → drawer full-height" adaptation may depend on a header behaviour Spec 17 doesn't yet implement.
- **Fix:** Confirm whether hide-on-scroll is an actual Spec 17 capability; if not, mark this adaptation as conditional on that header behaviour landing (it's already a first-class tested requirement in FR-36-9, so the dependency should be explicit). Low — it's a forward dependency, not an error.

### P2 — [LOW] "every top exemplar collapses the same at 768 as at 375" (FR-36-8) slightly glosses Apple
- **Spec ref / text:** FR-36-8: *"every top exemplar collapses the same at 768 as at 375."*
- **Check:** The visual teardown notes *"only Apple's full desktop bar survives past 768px at all (down to ~900–1024px)"* — i.e. Apple's collapse point is ~768–900, not cleanly 768. But the inventory's S4 finding frames it exactly as the spec does, so the spec is **source-faithful**; the imprecision originates upstream. Not worth changing unless S4 is tightened.
