# P2.5 Phase-5 gate — nav architecture council register

## Convergent headline
**Unanimous: the block DECOMPOSITION is sound** — every one of the 7 critics credited the 5-block split, the disclosure-vs-dialog separation, the ID-referenced saved-panel mega (fixes the always-last bug by construction), and one-source dual-render. The WP-Platform Realist verified against live source that the load-bearing mechanisms (block-context-by-ref, dynamic-block body-portal + inert, Store-API cart live-region, CPT-as-converter-target) **already exist in shipped code** — they are not speculative. So this is NOT a NO-GO and NOT a re-decision of the shape.

**But it is not build-ready.** The critics converge on a set of *under-specified load-bearing mechanisms* + a few genuine correctness fixes + one undocumented supersession. All are fixable on paper in one revision pass.

## MUST-FIX REGISTER (convergence-ordered; convergent + fatal first)

### MF1 — Drawer modality: document the supersession + reconcile the mechanism + guarantee an exit. ⭐ (4+ voices)
*Voices: A11y (M1/M2/M3), completeness-a (#1), gap-analysis (#1), WP-Platform (S1).*
- DP4 "full modal" **supersedes** Spec 34's Bean-locked NON-modal drawer (FR-34-1/FR-34-2) + P2 §15.3's "Spec-34 disclosure model" line. This WAS Bean's deliberate 2026-07-19 decision — but the doc never says it supersedes them (violates the project's D101 "carry-forward-or-justify, never silently subtract" rule). **Fix:** add a §9 reconciliation stating DP4 supersedes those FRs, dated + Bean-attributed.
- **Mechanism conflict (prove-the-cause):** a native modal `<dialog>.showModal()` gets top-layer promotion for free, which defeats the transformed-ancestor bug **without** a body-portal. Carrying BOTH `showModal()` modality AND the R6 body-portal = two overlapping fixes for one problem (unfalsifiable). **Fix:** pick ONE — native showModal (free modality/inert/top-layer, no portal) OR non-modal + manual-inert + portal — and justify against the shipped code.
- **No-exit trap (fatal a11y):** if the drawer is a true modal AND the per-row checkboxes can hide every header row at the drawer top, a touch user (no ESC key) can have ZERO visible close control. **Fix:** mandate ONE always-visible, non-optional close control inside the drawer.
- **Live-region coherence:** the cart badge duplicated into the drawer vs the frozen original — define a single canonical announcing node while the drawer is open (drawer copy announces; original suppressed), no double-announce.

### MF2 — The mega ID-reference data model: carrier + portability + dangling-ref. ⭐ (4 voices, code-grounded)
*Voices: Spec-Lawyer (M5), Cynic (M3), WP-Platform (MISSING + M4), Converter (M4).*
- **Where the ref lives:** you can't add an attribute to WP's core menu-item block by editing it — but (WP-Platform, verified) you CAN **extend `core/navigation-submenu` via an `editor.BlockEdit` filter** to add a `megaPanelRef` attribute + an inspector control (unknown attrs round-trip through parse/serialize cleanly). So it is an ATTRIBUTE + FILTER, **not necessarily a 6th block** (this refines the Cynic's "6th block" claim — the code-grounded read wins).
- **Cross-environment portability (correctness):** a raw numeric post-ID `ref` is NOT portable dev→prod (IDs differ between the clone DB and the client's DB). Referencing the panel by raw ID reintroduces WP's notorious nav import/export breakage. **Fix:** reference by a **portable key** (slug / clone-postmeta) resolved to a post at render time; the converter re-resolves to the new ID at import.
- **Dangling ref:** define render behaviour when the panel is deleted — render the plain dropdown/link + an editor-only warning badge.

### MF3 — The converter needs a genuinely-new "create-then-reference" multi-post stage. ⭐ (2 code-grounded voices)
*Voices: Converter (M-a/b/c + MISSING), WP-Platform (M4).*
- Today the converter is single-page-PATCH only (0 `wp_insert_post`/create calls). Cloning a nav that needs a saved `wp_navigation` post + N saved `sgs_mega_panel` posts is a NEW orchestration stage. **Fix:** model it on the existing `media-sideload.py` pattern (GET-before-POST lookup → create-or-**update-in-place** → capture id → rewrite downstream attr). Specify: idempotency key = a `_sgs_clone_source_id` **postmeta** (NOT bare `post_name`, which WP auto-suffixes → duplicates on every re-clone); **update-in-place on match, not skip**; an **orphan-sweep** for panels removed from a later re-clone; cross-page **stale-ref** handling (a 2nd `wp_navigation` post silently becomes the fallback for other pages).
- Register the `sgs_mega_panel` CPT (`show_in_rest`, REST base, capability) — it does not exist yet.
- A **sub-tree/fragment converter entry point** to route a scraped mega panel's CSS+content into a *different* target post (the walker is one-page-one-target today).
- **CLOSE the §7.5 "body-portal cloner-emit" open question:** all these blocks are DYNAMIC (save→null), so WP's block validator never diffs saved-vs-rendered markup — the portal cannot trip it. Not a risk; delete it from the open list (both Converter + WP-Platform confirm).

### MF4 — Context flow + the sibling-drawer hole + wrong context claim. ⭐ (3 voices, code-grounded)
*Voices: WP-Platform (M1/M2/M3), Spec-Lawyer (M1/M2/M3), Cynic (M4).*
- **Context claim is wrong as written:** block context can carry only the menu's `ref` (id) — NOT a "resolved menu tree" (computed at render) nor a "breakpoint" (measured in JS post-render). **Fix:** reword B1 to "provides the menu `ref` id as context; each consumer resolves via `SGS_Nav_Menu_Source`." The content-driven collapse breakpoint is a JS-runtime value (CSS custom property / `data-*` via the Interactivity store), not context.
- **Sibling-drawer context hole:** the §4 diagram wires `sgs/nav → sgs/nav-drawer` as PEERS via `aria-controls` (an ARIA string, not a context channel). Context flows only down the InnerBlocks tree — so the drawer's menu would get NO `navRef` and silently fall back to page-list. **Fix:** nest `sgs/nav-drawer` INSIDE `sgs/nav` in the SAVED tree (the runtime body-portal happens after PHP render, so it doesn't affect context). Add an editor guard if placed outside a nav ancestor.
- **Stale resolver constant:** `SGS_Nav_Menu_Source::NAV_BLOCK_NAMES` hardcodes `['sgs/adaptive-nav','core/navigation']`. Retiring adaptive-nav without adding `'sgs/nav'` breaks fallback resolution for the drawer + footer menus. **Fix:** §6 gains "update NAV_BLOCK_NAMES → add sgs/nav" as a hard step.
- Spec the B1↔B2 placement mechanism (locked InnerBlocks template vs render_block call); rename the `inline` token collision (B1 `layout` variant vs B2 `mode` — both use "inline" for different things); justify B2's KIND (a horizontal bar IS layout, not content-block — either argue why its flex-row isn't "grid/section machinery" or reclassify).

### MF5 — The non-coder mega-BUILDER UX is undesigned + gated last (stall trap + deal-loser). ⭐ (5 voices)
*Voices: Ship-PM (MF1/MF2), Competitor (MF1), Cynic (S2), gap-analysis (#3), WP-Platform (MISSING).*
- Building a panel in a separate CPT editor + wiring it by ID = developer-grade friction across 3 screens; a non-coder client can't see the panel under its real menu item. Rivals edit the mega inline, live, in position. **Fix:** design an **inline authoring affordance** — selecting a menu item in the `sgs/nav` editor edits its referenced panel in place (via `ServerSideRender`, already proven on brand-strip); "create new panel" spawns the CPT record transparently and back-references it; **the client never sees the ID or the CPT post-type**. The CPT stays the storage layer only. Add `templateLock:"contentOnly"` to the panel pattern (Spec 35 Part G, near-zero cost).
- **No first demoable slice = ADHD stall risk.** **Fix:** name **Mama's-first** (nav + nav-menu + cart-badge + drawer; zero mega/CPT/safe-triangle) as the first end-to-end demo, with ONE Indus mega panel demoed EARLY (right after the bar), before drawer/scroll polish.

### MF6 — Coverage gaps: house the unhoused requirements. (multiple voices)
*Voices: completeness-a (#3/#4/#5), gap-analysis (#2), Competitor (SF1), A11y (M7).*
Add a home for each (mostly B1/B3/plumbing/site-header, cheap):
- **Desktop overflow "more" menu / priority+** (FR-S9-4, Phase-1 #19) → B1 — currently absent; without it a low-item bar jumps straight to a hamburger, which the research forbids on desktop.
- **Link-count / column-ceiling INFORMATIONAL notice** (was literally flagged "Phase-3 D5", never settled) → B1 + B3 inspector, informational-only per the locked a11y-feedback policy.
- **Mandatory mega group-headings** (NN/g #1 anti-pattern) → B3 content contract / editor notice.
- **Intrinsic shrink-to-fit (A7, Bean-locked block standard)** → standing constraint on all 5 blocks — currently absent even as carried-forward.
- **WCAG 2.2 2.4.11 Focus-Not-Obscured** (sticky header hiding a focused item) → DP5.
- **Skip-to-content link, first focusable, not in `<nav>`** → `sgs/site-header`.
- **Distinct ≥44px chevron affordance** + **tunable hover-delay constants** (promoted with hover-default) → plumbing.
- **FR-S9-8 per-ELEMENT move-to-drawer** — DP4 narrowed it to per-ROW; note the deliberate narrowing or add per-element.
- **Keyboard: parent link vs disclosure toggle must be SEPARATE ≥44px focusable elements** (A11y M4 — else Enter navigates away, WCAG 2.1.1 fail).
- **Disclosure ESC/1.4.13 contract** (dismissible + hoverable + persistent, no auto-close timer) fully specified in the plumbing (A11y M5).
- **aria-current wiring = client-side** (compare `location.pathname` at mount), NOT server-baked (LiteSpeed caches the wrong page's answer — A11y M6).

### MF7 — Tool wins that REDUCE scope / de-risk (fold in before build). (tools-QC)
*Voices: completeness-b (#1–4).*

DP5's real remaining scope = wire one attr + one resolver flag + one toggle, NOT a new state machine. (Sizes down the Ship-PM gold-plate worry.)

## SHOULD-FIX / decisions to surface to Bean (not blocking, but real)
- **Drill-down vs WooCommerce 3-level menus (Competitor MF2):** DP6 defers drill-down, but Category→Subcategory→Type is 3 levels on the very shops we sell to; Phase-1 D3 said auto-promote to drill-down at 3+ levels. **Bean choice:** keep accordion-only v1 (accept shallow-IA positioning) OR pull drill-down into v1 for ≥3-level menus. Surface, don't silently defer.
- **Performance/caching for N mega resolvers (Competitor MF3):** batch the panel fetch (one `get_posts` by referenced ids), transient/object-cache the rendered panel HTML keyed by panel-modified, and add a per-page JS/CSS budget gate (<50KB JS / <100KB CSS) as a build acceptance criterion.
- **Dynamic mega content (Competitor MISSING #1):** auto-populated product/taxonomy grids inside a mega are now table-stakes for shops; ours is static-composed. Consider an extra-phase item (not v1).
- **Retirement lockstep (Cynic S1):** retiring adaptive-nav/mega-menu dangles `mega-menu/block.json`'s `parent:[...]`, `block_composition` rows, `variant_slots`, pattern refs — migrate atomically + add a retire-time grep gate (stale `sgs/mobile-nav` rows surviving "for weeks" proves this surface isn't reliably cleaned).
- **Mega-to-accordion transform in the drawer (Cynic M2):** the CPT panel must carry an explicit link-list structure the drawer reads directly — NOT the current fragile "re-render + DOM-scrape every `<a>`" approach.
- **do_blocks recursion guard** for CPT-into-panel render (WP-Platform MISSING) — a panel containing a nav referencing the same menu can recurse.
