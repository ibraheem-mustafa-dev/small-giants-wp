# sgs/hero — split content-band control fix + headline font-hardcode revert (LANDED)

**Verdict: PASS.** Two changes, both verified LANDED live on palestine-lives.org (page 13,
Indus clone) at 375 / 768 / 1440 via Playwright live-DOM measurement + eyeball vs the source
(lightsalmon-tarsier-683012.hostingersite.com):

1. **Content-band control now works on the split variant.** Split heroes suppressed the
   wrapper's inner-div content cap (it would collapse the 2-col grid), so `contentWidth`
   (Normal/Wide/Custom) did nothing. Fix (render.php): when split + contentWidth set, emit
   the resolved band as centred `padding-inline: max(root-pad, calc((100% - band)/2))` on the
   section grid itself — the full-bleed background still paints edge-to-edge behind it while
   the two columns confine to a centred band. Mirrors the wrapper's own token resolver
   (normal→content-size, wide→wide-size, full/empty→no cap, else literal). No client value;
   split heroes had NO cap before, so zero regression risk.
2. **Headline font reverted to the per-client token.** `.sgs-hero__headline` was reading a
   hardcoded Indus-measured clamp baked into this shared block (which had replaced the
   `--wp--preset--font-size--hero` token). Reverted to `var(--wp--preset--font-size--hero)`;
   the exact per-client curve (flat 40px ≤1024 → 50px @1440) now lives in the client's
   theme-snapshot `hero` font-size preset `size`. Shared block stays generic.

## 2026-07-17 (later) — content-band normal/wide de-hardcode → theme.json globals (LANDED)

**Verdict: PASS.** render.php content-band resolver no longer hardcodes px fallbacks for the
`normal`/`wide` tokens — it ties purely to the theme.json globals, so the per-site snapshot
value is authoritative.

**Change** (`plugins/sgs-blocks/src/blocks/hero/render.php`):
- `normal` → `var(--wp--style--global--content-size)` (was `var(--wp--style--global--content-size,1200px)`)
- `wide`   → `var(--wp--style--global--wide-size)`    (was `var(--wp--style--global--wide-size,1400px)`)

**Why:** the `1200px`/`1400px` fallbacks masked the theme value — an Indus hero set to
`contentWidth=normal` would have used the hardcoded 1200px instead of Indus's real 1140px
if the var ever failed to resolve. Framework default (1200/1400) lives in
`theme/sgs-theme/theme.json`; per-site override lives in the snapshot
(`sites/indus-foods/theme-snapshot.json` → `settings.layout` = 1140/1400). The original
Indus site's constrained content containers measure 1140px (Playwright, 2 instances at 1440),
confirming 1140 is correct and 1200 was wrong.

**Verification (live, palestine-lives.org, CDN cleared):**
- `getComputedStyle(:root).--wp--style--global--content-size` = **1140px** (was 1200 hardcode path).
- Hero `padding-inline` = 142.5px = `(clientWidth 1425 − 1140) / 2` → content band is 1140, not 1200.
- pageOverflow = 0 at 1440 and 375. Subheadline weight 700 (bold) unchanged.
