# Visual-diff report — sgs/heading `textColour` default removed (D343), 2026-07-16

verdict: PASS
first_paint_capture_passed: true

## Problem (Bean-directed fix)
`sgs/heading` forced its own text colour onto every heading, silently disabling
the client's heading colour. Three sites all did the same job — removing any one
alone is inert:

| site | was | now |
|---|---|---|
| `block.json` `textColour` | `default: "text"` | `default: ""` (= inherit) |
| `render.php:125` | `$attributes['textColour'] ?? 'text'` | `?? ''` |
| `style.css:15` | `.wp-block-sgs-heading:not([class*="has-text-color"]) { color: var(--wp--preset--color--text) }` | removed |

The emitted scoped rule `.{uid}.wp-block-sgs-heading{color:…}` is (0,2,0) and
beat the theme's own `h1..h6 { color: … }` (0,0,1). `theme.json` already owns
heading colour (`styles.elements.heading.color.text = var:preset|color|primary`).
Identical class to D338's `fontSize: 28 → null` fix on this same block.

## Live proof (canary, OPcache + CDN cleared)
Same page, same markup, before vs after the fix:

| | matched colour rules | computed |
|---|---|---|
| before | `.wp-block-sgs-heading:not([class*=has-text-color]){color:--text}` + `h1..h6{color:--primary}` + `.sgs-hdg-6f5117fe…{color:--text}` | `rgb(58,46,38)` brown ❌ |
| after | `h1..h6{color:var(--wp--preset--color--primary)}` **only** | `rgb(230,138,149)` pink ✅ |

Both overriding rules are gone from the matched set; the theme's rule is the
only one left. The instance uid changed (`6f5117fe` → `f8661a0c`), which proves
the attribute default genuinely changed rather than a cached render being served
— the uid is `md5(wp_json_encode($attributes))`.

## Known + accepted
Heading letter-spacing (`-0.36px` → `normal`) still differs from `core/heading`,
because `core-blocks-critical.css:246 .wp-block-heading{letter-spacing:-0.01em}`
is keyed to core's own class. **Bean accepted this 2026-07-16** ("letter-spacing
is never usually touched so I don't mind it being lost"). Not fixed here.

## Blast radius
Every `sgs/heading` on every client now inherits heading colour from theme.json
instead of being forced to `--text`. On any client whose theme sets a heading
colour, headings will now render that colour — which is the intended behaviour
and the reason the default was removed. An explicitly-set `textColour` still
wins (unchanged).
