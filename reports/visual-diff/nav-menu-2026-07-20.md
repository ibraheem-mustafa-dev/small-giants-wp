## Problem

The Mama's draft styles the featured nav item as a **filled pill**
(`sites/mamas-munches/mockups/homepage/index.html:231-235`):

```css
.sgs-header__nav-featured {
  background: var(--primary) !important;   /* #E68A95 */
  color: var(--text) !important;           /* #3A2E26 */
  font-weight: 600 !important;
}
```

plus the base link's `border-radius: 8px`, `padding: 10px 14px`, `min-height: 44px`.

`sgs/nav-menu` had **no `featuredBg` attribute at all** — only `featuredColour`. The
converter had nowhere to put the draft's fill and silently dropped it, falling back to the
`accent` default. The clone rendered gold-on-cream at **1.35:1** (WCAG AA needs 4.5:1).

**The contrast failure is a symptom of the missing capability, not a policy gap.**

## Fix

`featuredBg` attribute (default `''` → label form unchanged, no existing site changes shape).
When set, `render.php` 4d emits the pill; the foreground runs through
`sgs_wcag_preferred_text_colour_for_bg()` — the same helper the hover pill already uses — so
the operator's colour wins when it clears AA and falls back safely otherwise. No palette can
regress below AA. Inspector control added (client-experience rule).
`parts/header.html` bar → `featuredColour:text` + `featuredBg:primary`.

Contrast arithmetic (independently computed, WCAG relative-luminance formula):

| Pairing | Ratio | Verdict |
|---|---|---|
| Draft: #3a2e26 on #e68a95 | **5.28:1** | PASS |
| Clone before: #f5d050 on #fbf3dc | **1.35:1** | FAIL |
| Normal link: #3a2e26 on #fbf3dc | 11.86:1 | PASS |

Adopting the draft's own pairing therefore fixes fidelity and accessibility with one change.

## Known divergence from the draft — HOVER (Bean-deferred 2026-07-20)

Measured on the restored deploy. **Resting state matches the draft. Hover does not:**

| State | Draft | Live clone |
|---|---|---|
| rest | pink pill, dark text, weight 600, radius 8px | ✅ same |
| **hover** | background stays `--primary`, `box-shadow: inset 0 -2px 0 var(--accent)`, **no underline** | ❌ pill retained but `text-decoration: underline`, `box-shadow: none` |

**Cause (read from the emitted CSS, not inferred):** section 4c emits the generic hover rule for
`.sgs-nav-menu__link:hover` at specificity (0,3,0); with no `itemHoverColour` set it takes the
`else` branch and emits `text-decoration:underline`. The featured rule is (0,3,0) too and comes
later, so its colour/background win — but `text-decoration` doesn't conflict, so the underline
also applies. The draft's accent underline is an **inset box-shadow**, which the block has no
attribute for.

**Deferred at Bean's direction** — hover is being worked on at the block level separately, so
fixing it here would collide. Parked as `P-NAV-FEATURED-HOVER-DRAFT-PARITY`. **The featured
item is NOT fully draft-faithful until this closes** — do not treat this report as full parity.

Also unchanged and by design: the **drawer's** featured item stays the accent label (gold) —
the draft authors no drawer, and the drawer measured axe 0.
