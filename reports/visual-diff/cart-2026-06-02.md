# sgs/cart — Verification (2026-06-02)

## Verified live (canary, WooCommerce active)
- **SSR 0 → hydrate:** badge renders `0` server-side, then view.js fetches
  `GET /wc/store/v1/cart` (confirmed in Resource Timing: `storeApiCartFetch: true`)
  and reflects the real count. Empty cart → badge `0`.
- **Link + a11y:** links to `/cart/`; `aria-label="View your cart (0 items in cart)"`
  (WCAG name carries the count).
- **No legacy cart-fragments:** `cartFragmentsScript: false` (dequeued) +
  `cartFragmentsAjax: false` (no `?wc-ajax=get_refreshed_fragments`).
- **jQuery:** present on the page but loaded by WooCommerce core, NOT imported by the
  cart (per design: tolerate a 3rd-party jQuery, never import). Cart view.js is vanilla.
- Zero console errors.

## OPEN Q resolutions
- Q1 (iAPI store namespace): NOT used — the documented stable Store API REST endpoint is
  used instead (version-independent, no WC-bundle coupling).
- Q2 (hydrate with no other WC block): resolved — the Store API is a plain REST endpoint,
  no `wc-blocks-interactivity` dependency needed; the fetch worked on a page with no other
  WC block.

The canary has **0 WC products** (it's a clone-pipeline canary, not a shop), so the badge
INCREMENT on a real add-to-cart wasn't exercised end-to-end. The increment uses the SAME
