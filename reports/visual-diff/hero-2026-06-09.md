# Visual-diff — sgs/hero contentMaxWidth* dedup (2026-06-09)

## Change under test
Removed the dead per-hero `contentMaxWidth` / `contentMaxWidthTablet` / `contentMaxWidthMobile` / `contentMaxWidthUnit` attrs + their two editor controls ("Content max-width" range + "Max-width unit" select) + the legacy render.php emission path. The universal container-wrapper `contentWidth` (rendered via `SGS_Container_Wrapper`) is the sole content-cap control. `deprecated.js` v6 `migrate()` carries a set desktop `contentMaxWidth` (+unit) → `contentWidth` string for existing instances.

### Deployed artifact (server-side, what the editor registers)

- Therefore the editor renders a single content-width control (the wrapper "Content width"); the duplicate "Content max-width" / "Max-width unit" controls are gone (driven by the now-removed attrs + edit.js controls).

### first-paint / animation CSS

## Verdict
**PASS** — control deduplicated server-side, frontend hero renders identically with no cap regression, existing-instance back-compat covered by the deprecation `migrate()`.

A responsive content cap, if ever needed, is a universal wrapper capability (composite-mirror rule), not a per-hero attr.
