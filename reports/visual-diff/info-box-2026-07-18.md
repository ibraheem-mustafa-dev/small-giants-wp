# Visual-diff — `sgs/info-box` inline-zero (D345, GOTCHA-F fix) — 2026-07-18

Wave-1 exemplar of the framework-wide inline-zero rollout (Spec 32 FR-32-4 as
amended 2026-07-18 / D345; plan `.claude/plans/2026-07-17-phase-inline-zero-rollout.md`).
Removes the last inline `style` attribute from `sgs/info-box` and fixes the
latent GOTCHA-F hover break the removal would otherwise have caused.

## What changed (all inside `src/blocks/info-box/`)

1. **render.php** — the remaining per-instance custom-property VALUES (transition
   timing + hover scale/shadow) now emit as a scoped `.{uid}.sgs-info-box{ … }`
   rule inside the block's own `<style>` (consolidated to the collected
   stylesheet by the SGS CSS registry, FR-32-11) instead of via the
   `get_block_wrapper_attributes()` `'style'` key — which is **dropped**. The
   block already had a `$scoped_css`/`<style>` mechanism for its skip-serialised
   native supports; the vars now join it.
2. **render.php** — the three hover COLOURS (`background-color` / `color` /
   `border-color`) emit as a per-instance scoped `.{uid}.sgs-info-box:hover{ … }`
   rule (only when set). Its specificity (0,3,0) beats the variant base (0,2,0),
   so it is **variant-safe** — no resting-value fallback is needed even though
   the resting background is per-variant (`--subtle` transparent / `--card`
   surface / gradient cards).
3. **style.css** — deleted the three `[style*="--sgs-hover-*"]:hover`
   presence-selectors (footprint GOTCHA F). That inline-attribute-presence gate
   silently STOPPED matching the instant the var moved off the inline `style`
   attribute into the scoped rule — it would have killed the hover colour with
   no error. Replaced by the render.php scoped `:hover` rule above.

No block version bump, no `deprecated.js` (D270). The scale/shadow vars keep
their existing modifier-class-gated consumers (no presence-selector, no GOTCHA F).
