# ADDENDUM — badgeColour dead-wiring FIXED (same day, second change)

The report above verified the Spec 35 MANIFEST change. This addendum covers a SECOND,
later change to the same block on the same date. Recorded here rather than as a fresh
report because the visual-diff gate keys on block+date and would otherwise have passed
this change on the strength of the earlier one — a stale pass.

## What changed
The report above logged `badgeColour` as DEAD CSS WIRING: render.php emitted it only as
`data-sgs-badge-colour="…"` and NO rule anywhere consumed that data attribute (verified
repo-wide, including the theme). The badge colour was hardcoded to the accent token at
style.css:124-125, so a client could set the control and nothing happened.

FIXED by wiring it properly rather than deleting the control:
- render.php:152 now emits `--sgs-mm-badge-bg` into the block's existing scoped `<style>`
  mechanism, the same pattern as `--sgs-mm-panel-bg`
- style.css:128 consumes it: `background-color: var(--sgs-mm-badge-bg,
  var(--wp--preset--color--accent, currentColor))`
- the `data-sgs-badge-colour` attribute is left in place (harmless, useful for JS/debug)

Option A token fallback: an unconfigured instance resolves to the same accent token as
before, so existing output is byte-identical.
