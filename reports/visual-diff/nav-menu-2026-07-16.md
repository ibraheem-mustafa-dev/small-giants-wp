## The focus-colour bug (found live, fixed)

`linkHoverColour` defaults to `""` and the block's `:hover`/`:focus-visible` rule set only
`text-decoration` — so on hover/focus the colour fell through to **theme.json's global
`styles.elements.link[':hover'|':focus']` = `primary-dark`**. Specificity: WP emits that as
`a:hover` = **(0,1,1)**, which BEATS the block's resting `.sgs-nav-menu__link` = **(0,1,0)**.

Measured on the pink Mama's drawer: focused link `#c56a7a` on `#e68a95` = **1.48:1** — a
serious WCAG failure landing on the FOCUSED link, i.e. precisely what a keyboard user hits
first. This is the D339 class again: a token NAME (`primary-dark`) assumed to imply a
luminance.

**Fix:** re-assert `color: inherit` on the state selectors (0,2,0), so the link keeps the
drawer's COMPUTED, contrast-checked foreground on every client and the underline carries
the affordance. An operator's explicit `linkHoverColour` still wins (render.php emits it
at higher specificity).

**Caught only because the eye flagged it first** — the screenshot showed the focused link
looking washed out; the first probe read a mid-transition value (200ms `transition: color`),
so the settled value was re-measured after a 500ms wait rather than trusted mid-flight.

## First-paint (M1)

`animation-fill-mode:both` = 0, non-zero `animation-delay` = 0, `animation:` shorthand = 0
in `nav-menu/style.css` (it uses `transition`, not `animation`). Links are server-rendered
`<a href>` and painted opaque on first paint; the accordion's `hidden` panels are the
disclosure contract, not an opacity animation. Cheat-gate: its one `transition-duration:
!important` is the `@media (prefers-reduced-motion: reduce)` kill-switch — baselined
alongside 9 sibling blocks carrying the identical idiom, with the reason recorded.
