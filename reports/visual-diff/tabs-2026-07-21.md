# tabs resting-state fix — visual verification

## What changed
sgs/tabs let a client style the SELECTED tab and the HOVER background, but not the
resting state those sit between: `background: transparent` was hardcoded at style.css:68
with no custom property, and the underline indicator had no resting base. Added
`tabBgColour` and `tabIndicatorColour` as the missing BASE properties.

NOTE for future sessions: `tabTextColour` already existed (consumed at style.css:84).
The resting TEXT colour was never the defect. An earlier diagnosis in this session got
this wrong by reading style.css:63-72 and missing line 84.

## Ownership
Both attrs map to custom properties consumed on `.sgs-tabs__tab` — the tab BUTTON, which
sgs/tabs emits itself (render.php:86,89). They never touch panel content, which is nested
blocks tabs does not own. De-hardcoding the button is in bounds.

## Method
Two published pages, identical block content, differing only in the new attrs:
- untouched  → /sv-tabs-untouched/  (page 1558)

## Results — computed styles, per tab

| Tab | aria-selected | untouched bg | client-set bg | client-set indicator |
|---|---|---|---|---|
| One | true  | `rgba(0,0,0,0)` | `rgba(0,0,0,0)` | none |
| Two | false | `rgba(0,0,0,0)` | `rgb(0,255,0)` | `rgb(255,0,255)` inset |

Tab One is CORRECT at transparent: the selected tab is styled by the higher-specificity
`[aria-selected="true"]` rule (`--sgs-tab-active-bg`, unset here), which is exactly what
the resting base should defer to. A first measurement of tab One alone read as a failure;
it was the one element where the resting rule SHOULD lose. Verified on the resting tab.

`--sgs-tab-bg` resolved to `#00ff00` on the tab element on the client-set page and was
absent on the control.
