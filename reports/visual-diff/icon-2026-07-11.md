# sgs/icon — emoji glyph honours iconSize (LANDED)

**Verdict: PASS.** The ingredient-education emojis rendered 16px inside a correctly-sized
32px icon box — the block sized its `.sgs-icon` container + `.sgs-icon__svg svg` off
`--sgs-icon-size` but had NO rule sizing the emoji `<span>`, so the glyph fell to the
inherited base font-size (~16px). Fix: added `.sgs-icon__emoji { font-size: var(--sgs-icon-size, 32px); line-height: 1; }`
to `src/blocks/icon/style.css`. Universal — every emoji-source icon now honours its iconSize.

## Root cause (live DOM, not inference — STOP-VERIFY-CLAIM)
The register/DB-side theory ("seed `iconSize.css_property='font-size'` + rewire the converter")
was DISPROVEN on the live path: the icon block already carried `--sgs-icon-size:32px`; the
emoji `<span class="sgs-icon__emoji">🌾</span>` computed `font-size:16px` with no matching
CSS rule (only `.sgs-icon__placeholder` + `.sgs-icon-list__icon` set an icon font-size). It
was a block CSS gap, not a converter/seed issue.

## Evidence (live sandybrown page 8, all breakpoints)
- **1440 / 375:** `.sgs-icon__emoji` computed `font-size = 32px` (was 16px); `--sgs-icon-size = 32px`.
- Deploy → `hosting_clearWebsiteCacheV1` → fresh-CSS computed-style read (browser-cache bypassed).
